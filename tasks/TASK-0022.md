# TASK-0022 ‚Äî Refine query-biased summaries (bound their size; snippet polish)

- **Status:** Draft
- **Created:** 2026-07-08
- **Owner:** (unassigned)
- **Summary:** The opt-in `/search` summaries (TASK-0019 ¬ß4) are correct but **unbounded in size** ‚Äî a
  document with few sentence boundaries yields one enormous passage. Add a **size cap** (and any related
  snippet polish) so summaries are predictable and LLM-token-friendly, without changing ranking.

## Motivation / observation

Summaries are produced by `UnifiedHighlighter` with a sentence `BreakIterator`, joined by `" ... "`
(`LucindriSearchService.summarize()`). Passage boundaries are whatever the sentence `BreakIterator`
finds ‚Äî so when a document has a "sentence" that runs for a full paragraph (dense regulatory / legal
text), a single passage can be enormous.

Observed live on the t45mCR index (2026-07-08), query `#combine(#1("black bear") "attacks")`, result
**`FR940902-1-00053`** (a Federal Register hunting regulation): the summary was a single ~2,000-character
"sentence" of sub-clauses `(B) ‚Ä¶ (viii) ‚Ä¶ (ix) ‚Ä¶`. Not a highlighter bug ‚Äî the source genuinely has no
sentence break ‚Äî but a bad snippet.

**Why it matters:** the ISJ/Cottontail consumer feeds these summaries to an LLM judge. Unbounded snippets
waste tokens and can dominate the prompt; a bounded, predictable summary size is wanted. This is UX/cost
polish, not a correctness issue.

## Options (pick one; a decision is needed)

1. **`LengthGoalBreakIterator` (recommended).** Lucene ships
   `org.apache.lucene.search.uhighlight.LengthGoalBreakIterator`, which wraps another `BreakIterator` and
   targets a passage length, splitting an over-long "sentence" near a character goal while still preferring
   real sentence boundaries. Wrap the sentence iterator:
   `LengthGoalBreakIterator.createClosestToLength(BreakIterator.getSentenceInstance(...), targetChars)`.
   Keeps readable, sentence-ish passages *and* bounds their size ‚Äî the proper Lucene mechanism.
2. **Hard cap on the joined summary** ‚Äî truncate the final string to N characters + `"‚Ä¶"`. Dead simple,
   guarantees a max, but can cut mid-word / mid-sentence. Good as a **backstop** even if we adopt (1).
3. **Tune `maxPassages` / `setMaxLength`** ‚Äî note `setMaxLength` caps how much of the *field* is analyzed,
   not passage length, so it does **not** fix a single huge sentence; and `maxPassages` is a count, not a
   size. Neither alone solves this.

**Recommendation:** (1) `LengthGoalBreakIterator` with a configurable target, plus (2) a generous final
hard cap as a backstop.

## Sketch of work

- `LucindriSearchService.summarize()` ‚Äî wrap the break iterator per option (1); optionally apply the (2)
  backstop to the returned string.
- **Config:** expose the target/cap as startup config, consistent with TASK-0019 ¬ß5 CLI flags (e.g.
  `--summaryTargetChars` / `--maxSummaryChars`, with a sensible default). `maxPassages` already exists.
- **Test (server or service):** index a doc containing a pathologically long single "sentence"; assert the
  summary is bounded (‚â§ the configured max) and still non-empty, and that normal docs are unaffected.
- **Docs:** update TASK-0019 ¬ß4 / `docs/query-biased-summaries.md` note and the README server section
  (the "knob" mentioned there).

## Non-goals

- No change to ranking/scoring or to the search contract.
- Not proximity-faithful highlighting (term-based snippets stay; the "ephemeral index" alternative in
  `docs/query-biased-summaries.md` is a separate, larger effort).
