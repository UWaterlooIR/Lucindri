# TASK-0022 — Refine query-biased summaries (bound their size)

- **Status:** Ready
- **Created:** 2026-07-08
- **Owner:** (unassigned)
- **Summary:** The opt-in `/search` summaries (TASK-0019 §4) are correct but **unbounded in size** — a
  document with few sentence boundaries yields one enormous passage. **Decision:** keep the current
  sentence-passage `UnifiedHighlighter` approach and add a **hard word cap** on the result (default 75
  whitespace-separated words), plus make both the passage count and the cap **server-configurable**. No
  ranking change.

## Motivation / observation

Summaries are produced by `UnifiedHighlighter` with a sentence `BreakIterator`, joined by `" ... "`
(`LucindriSearchService.summarize()`). Passage boundaries are whatever the sentence `BreakIterator`
finds — so when a document has a "sentence" that runs for a full paragraph (dense regulatory / legal
text, or an SEO keyword wall), a single passage can be enormous.

Observed live on the t45mCR index (2026-07-08), query `#combine(#1("black bear") "attacks")`, result
**`FR940902-1-00053`** (a Federal Register hunting regulation): the summary was a single ~2,000-character
"sentence" of sub-clauses `(B) … (viii) … (ix) …`. Also seen on climbmix: SEO product-keyword pages whose
entire body is one punctuation-free run-on. Not a highlighter bug — the source genuinely has no sentence
break — but a bad snippet.

**Why it matters:** the ISJ/Cottontail consumer feeds these summaries to an LLM judge. Unbounded snippets
waste tokens and can dominate the prompt; a bounded, predictable summary size is wanted. This is UX/cost
polish, not a correctness issue.

## Options considered (design discussion 2026-07-08) — and why A

- **A (chosen) — sentence passages + hard word cap.** Keep the sentence `BreakIterator` and
  UnifiedHighlighter's query-biased passage selection; after it returns the joined passages, truncate to a
  **word cap** (word boundary + marker). Simplest; reuses UnifiedHighlighter's stemmed-term location and
  BM25-ish passage scoring; clean on well-punctuated docs.
- **B — word/line base iterator + length goal.** Robust to run-ons (can select a window anywhere), but
  breaks *clean* text mid-sentence and adds no value over A's cap for the judge. Rejected.
- **C — custom "bounded-sentence" `BreakIterator`.** Best quality (sentence boundaries for clean text, forced
  word-boundary breaks inside over-long sentences), but ~50–80 lines of custom iterator. Overkill for v1.
- **D — roll our own sliding/overlapping window summarizer.** Best query-centering (half-overlap or
  match-centered windows, word-accurate length), but the most code and re-implements the stemmed-term
  offset location UnifiedHighlighter already does. Revisit only if A's snippets prove inadequate.

**Findings that pinned the decision:**
- `LengthGoalBreakIterator` only *chooses among the base iterator's boundaries* — it never splits within a
  segment, so wrapping the sentence iterator does **not** cap a punctuation-free run-on (my original draft
  was wrong on this). Capping the messy pages therefore has to be a post-step (A) or a different base
  iterator (B/C) or a custom window (D).
- UnifiedHighlighter passages are **contiguous and non-overlapping**, so it can't re-center a window on a
  match near a boundary. Better centering is only possible with D. Accepted for v1.
- Summarization cost is a **per-document re-analysis run only on the top-K results being summarized (≈5)**,
  not the result set or corpus — so it's cheap regardless of A/B/C, and cost is not a deciding factor.
- **Accepted limitation:** on a run-on, A's cap keeps the document's *first* 75 words, which may miss a
  query match deep in the run-on. Acceptable — 75 words of an SEO/keyword wall is low-value to the judge no
  matter how it's sliced.

## Implementation plan (agreed)

**Behavior — `LucindriSearchService.summarize()`:**
- Keep the sentence `BreakIterator`, the `" ... "` passage joiner, and the leading-sentence no-match
  fallback exactly as they are.
- After `UnifiedHighlighter` returns the joined passages, enforce the cap: split the summary on whitespace;
  if it has ≤ `maxSummaryWords` words, return it unchanged (no marker); otherwise keep the first
  `maxSummaryWords` words (preserving the original text and any `" ... "` joiners up to that point) and
  append `" ..."` (a space then three literal periods — **not** the Unicode ellipsis).
- The `" ..."` marker does **not** count toward the cap (a truncated summary is exactly
  `maxSummaryWords` words + the marker). The cap applies to both highlighted results and the fallback.

**Config — server CLI flags, plumbed through the service constructor:**
- `--maxPassages` — already exists; **change the default 2 → 4**.
- `--maxSummaryWords` — **new**, default **75**.
- Add `maxSummaryWords` to `LucindriSearchService` (new constructor param; keep a delegating overload with
  the default so `IndriSearch`/tests are unchanged — batch doesn't summarize). Log both values at startup
  with the rest of the resolved config.

**Tests (service level):**
- A doc whose body is one very long, punctuation-free "sentence" (well over 75 words) → summary is ≤ 75
  words and ends with `" ..."`, and is non-empty.
- A short doc (< 75 words) → summary returned unchanged, no `" ..."` marker.
- A boundary case (exactly at the cap) → not truncated.

**Docs:**
- README server section — document `--maxPassages` (default 4) and `--maxSummaryWords` (default 75).
- `docs/query-biased-summaries.md` and the TASK-0019 §4 note — record the word cap + the accepted run-on
  limitation.

## Non-goals

- No change to ranking/scoring or to the search contract.
- Not proximity-faithful highlighting, overlapping/match-centered windows, or a custom summarizer (options
  C/D) — deferred; revisit only if A's bounded snippets prove inadequate for the judge.
- Not a per-request knob — cap and passage count are startup config for v1 (consistent with TASK-0019 §5).
