# TASK-0003 — Fix collection-statistics double-count in IndriIndexSearcher (correctness + startup cost)

- **Status:** Done
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** `IndriIndexSearcher.collectionStatistics(field)` computes the collection length
  (`sumTotalTermFreq`) **twice** — once correctly from `Terms.getSumTotalTermFreq()`, then again
  by scanning every document's norm — and the second pass is itself wrong (it sums the raw
  `SmallFloat`-**encoded** norm bytes, not decoded lengths). This (a) corrupts `p(w|C)` in the
  Dirichlet/JM smoothing (wrong scores) and (b) is an O(all-docs) scan that costs ~27 s of fixed
  per-JVM startup on the full 553M-doc index. Remove the redundant scan; add tests that prove the
  statistics are correct.

> Read `../CLAUDE.md` first. Searcher-module change. Related: [[climbmix-full-index]] (where the
> ~27 s startup was measured), [[TASK-0002]] (also adds searcher tests — coordinate the JUnit
> wiring).

---

## 1. The bug (verified in source)

`LucindriSearcher/.../searcher/IndriIndexSearcher.java`, `collectionStatistics(String field)`:

```java
docCount          += terms.getDocCount();
sumTotalTermFreq  += terms.getSumTotalTermFreq();   // line ~43: ALREADY the collection token count
sumDocFreq        += terms.getSumDocFreq();

NumericDocValues numericDocValues = leaf.reader().getNormValues(field);   // lines ~48–63:
int nextDoc = numericDocValues.nextDoc();
...
while (nextDoc != NO_MORE_DOCS) {
    ...
    long normValue = numericDocValues.longValue();  // RAW encoded byte (0–255), not a length
    tempFreq += normValue;
    sumTotalTermFreq += normValue;                   // line ~61: adds a SECOND count on top
    nextDoc = numericDocValues.nextDoc();
}
```

Two independent defects in the accumulation of `sumTotalTermFreq`:

1. **Double count.** `Terms.getSumTotalTermFreq()` is, by definition, the sum of term
   frequencies over the field = the exact collection length in tokens. Summed across leaves it is
   already complete and correct. The norm loop then adds a *second* per-document quantity to the
   same accumulator.
2. **Wrong encoding.** The index writes norms via Lucene's `LMDirichletSimilarity.computeNorm`,
   which stores `SmallFloat.intToByte4(fieldLength)` — a lossy 1-byte encoding.
   `numericDocValues.longValue()` returns that **raw byte (0–255)**, not the decoded length. (The
   correct decoder, `SmallFloat.byte4ToInt`, exists in `IndriSimilarity.getLengthValue()` but is
   used only by the scorer, not here.) So the extra addend is a sum of encoded bytes — not the
   lengths, and not anything meaningful.

Origin: leftover diagnostic code — `tempFreq`, `noNorm`, `normCount` are computed but referenced
only by two commented-out `System.out.println`s that compared "tokens (lucene)" vs "tokens
calculated"; the `sumTotalTermFreq += normValue;` line was left active.

### Consequences

- **Correctness.** `sumTotalTermFreq` ends up inflated (~+50–60% for ~300-token passages).
  `IndriSimilarity.fillBasicStats` feeds it into `setNumberOfFieldTokens(...)`, and the collection
  model computes `p(w|C) = totalTermFreq / numberOfFieldTokens`. Inflating the denominator shrinks
  `p(w|C)` uniformly for every term → the Dirichlet smoothing mass `μ·p(w|C)` is too small →
  effectively a **smaller μ than requested**. Rankings deviate from true Indri (modest but real;
  changes the balance between docs that do and don't contain a term, and multi-term combination).
  Affects both the 10% and full indexes identically.
- **Performance.** The per-doc loop is an O(#docs) scan — 553M on the full index — run once per
  field per JVM. At full scale (1.4 TB > RAM) the norms don't stay cached, so it is disk-bound and
  costs a fixed **~27 s per JVM invocation** (measured: 27.2 s cold ≈ 26.8 s warm). It is memoized
  per JVM, so amortized in a long-running server but re-paid per one-shot process.

## 2. The fix

Delete the norm-scan loop (and its dead `tempFreq`/`noNorm`/`normCount`/`getNormValues` lines);
keep only the metadata accumulation:

```java
for (LeafReaderContext leaf : getIndexReader().leaves()) {
    final Terms terms = leaf.reader().terms(field);
    if (terms == null) continue;
    docCount         += terms.getDocCount();
    sumTotalTermFreq += terms.getSumTotalTermFreq();
    sumDocFreq       += terms.getSumDocFreq();
}
```

This yields the correct collection length and reduces startup to segment-metadata reads
(milliseconds). Also **evaluate whether the whole override is now redundant** with Lucene's stock
`IndexSearcher.collectionStatistics` (which computes the same values from metadata and aggregates
across a composite reader). Prefer keeping the explicit leaf loop only if there is a concrete
reason (e.g. avoiding Lucene's `Math.max(1, …)` clamping or an assertion); otherwise deleting the
override entirely is the simplest fix. Decide based on behavior parity confirmed by the tests.

## 3. Tests — prove the statistics are correct (the point of this task)

**Depends on [[TASK-0004]]** (shared test framework). Do **not** add JUnit/surefire wiring here;
use the `searcher/testutil` `TestIndex`/query helpers TASK-0004 provides (its `TestIndex` already
builds indexes with `LMDirichletSimilarity` + a known-token-count analyzer and can wrap N sub-
indexes in a `MultiReader` — exactly what the tests below need).

Build a small, controlled Lucene index in a temp dir (or RAM) using the **same index-time
similarity Lucindri uses** (`LMDirichletSimilarity`) and the project analyzer, then:

### 3a. Collection length is correct (primary gate)
- **Oracle test.** Assert
  `new IndriIndexSearcher(reader).collectionStatistics(f).sumTotalTermFreq()`
  **equals** Lucene's stock `new IndexSearcher(reader).collectionStatistics(f).sumTotalTermFreq()`
  for the same reader. Lucene's stock value (from `getSumTotalTermFreq`) is the ground truth. This
  fails before the fix (Indri value is inflated) and passes after.
- **Known-corpus test.** Index a handful of docs whose exact token counts you control, and assert
  `sumTotalTermFreq() == Σ(known doc lengths)` (also equals `Σ tf` over all terms). Make the doc
  lengths small (e.g. ≤ 8 tokens) so any norm round-trip is exact and the truth is unambiguous.
- **Regression.** Assert `docCount()` and `sumDocFreq()` are unchanged by the fix (they were never
  part of the double count) and match Lucene's stock values.

### 3b. Multi-segment / MultiReader aggregation
- Build ≥2 segments (or 2 sub-indexes wrapped in a `MultiReader`, mirroring the 8-part
  production layout) and assert the aggregated `sumTotalTermFreq`/`docCount`/`sumDocFreq` equal the
  sums of the per-segment truths. Guards the fix under the real query-time reader shape.

### 3c. Scoring uses the corrected statistic (end-to-end)
- On a controlled index with docs short enough that `SmallFloat` length encoding is exact, run a
  single-term query and assert the returned Dirichlet score equals the closed form
  `log((tf + μ·p(w|C)) / (docLen + μ))` with `p(w|C) = ttf / sumTotalTermFreq` computed from the
  **correct** collection length and the default μ (=2000, or set via `<rule>`). Demonstrates the
  corrected statistic flows into `p(w|C)` and that the score matches Indri's formula (and differs
  from the pre-fix value).

## 4. Acceptance criteria

1. All tests in §3 pass; `mvn -q test` in `LucindriSearcher` is green.
2. `IndriIndexSearcher.collectionStatistics(f).sumTotalTermFreq()` matches Lucene's stock value
   and the known-corpus truth, single- and multi-segment.
3. On the full index (`/ssd-8TB/indexes/climbmix_full`), the O(#docs) norm scan is gone: startup
   drops materially, with the largest benefit **cold** (the scan's ~553M norm reads were
   disk-bound — the earlier ~27 s case). Note: opening 8 readers + terms-index FSTs across 1.4 TB
   is an inherent floor the fix does NOT remove, so "sub-second" is not achievable for a fresh JVM.
4. Scores change in the expected direction (larger effective smoothing → the corrected `p(w|C)` is
   larger than the buggy one). **Note in the progress log that any previously saved TREC runs
   against these indexes are now invalidated** — the fix changes scores; this is intended.
5. Rebuild `LucindriSearcher-1.5-jar-with-dependencies.jar`.
