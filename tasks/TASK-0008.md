# TASK-0008 — End-to-end Lucindri vs Indri comparison on TREC t45minusCR + topics 401–450

- **Status:** Ready
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** Index a real TREC collection (`t45minusCR`) in **both** C++ Indri and Lucindri, run
  the 50 Indri-language topics 401–450 (Metzler) plus generated queries covering the operators the
  50 don't exercise, and **compare the two systems' rankings** — a scaled stress test of all the
  Lucindri fixes (TASK-0002/0003/0006/0007) that also confirms Lucindri matches Indri reasonably
  well on real data. Investigation/measurement task; findings → the conformance catalog.

> Read `../CLAUDE.md` first. Reference: [[indri-cpp-reference]] (`/ssd-8TB/installs/indri-5.21/`).
> Builds on the small-corpus harness `scripts/diff_indri.sh` (TASK-0005).

---

## 1. Inputs

- **Corpus:** `/ssd-8TB/corpora/t45minusCR/` — 7 **gzipped** trectext files (`ft91–94`, `fr94`,
  `latimes`, `fbis5`); TREC disks 4 & 5 minus the Congressional Record (~500k docs). Tags observed:
  `<DOC>`, `<DOCNO>`, `<TEXT>`, `<HEADLINE>`, `<DATE>`, `<PAGE>`, `<PROFILE>`, `<PUB>`.
- **Queries:** `/ssd-8TB/trec-topics/401-450.metzler.xml` — 50 topics in **Indri query language**
  (TREC-8 ad-hoc). Every topic is `#weight( 0.8 #combine(terms) 0.1 #combine(#1 phrases) 0.1
  #combine(#uwN windows) )` (single-term topics omit the phrase/window parts).

## 2. Operator coverage (verified) and what to generate

The 50 topics use only: **`#weight` (all 50), `#combine`, `#1`, `#uwN` (`#uw8`,`#uw12`)**, plus
plain and numeric terms. So this collection stress-tests those four operators heavily but leaves
most of Lucindri's claimed language untested.

**Generate additional queries** (same collection) to cover the rest and stress the recent fixes:
- `#or`, `#not` (e.g. `#combine(topic #not(unwanted))` — TASK-0006), `#max`, `#wsum`, `#wand`.
- `#scoreif` / `#scoreifnot` (TASK-0006) — `#scoreif(requiredterm #combine(topic))`,
  `#scoreifnot(excludeterm #combine(topic))`.
- `#band`, `#syn`, and **compound operands** — `#band(#syn(a b) #syn(c d))`, `#uwN(#syn(a b) c)`,
  `#1(#syn(a b) c)` (the exact shapes fixed in TASK-0002; include a phrase at document position 0).
- **Field restriction** `term.field` — requires a fielded index (e.g. index `headline`/`title` as
  named fields via `fieldNames`; TASK-0007); then `term.headline`.
- **Jelinek-Mercer** smoothing — run a subset with Lucindri `<rule>jm:0.4</rule>` vs Indri
  `method:linear,collectionLambda:0.4`.

## 3. Prerequisites / wrinkles (resolve before comparing)

1. **Lucindri trectext reads plain files only** (`FileInputStream`), but the corpus is `.gz`. Indri
   `IndriBuildIndex` reads gzip directly. Options: (a) **add gzip support to
   `TrecTextDocumentParser`** — mirror the climbmix parser's `GZIPInputStream` (small, generally
   useful; recommended, possibly its own tiny task); or (b) decompress the corpus for Lucindri (a
   few GB). Do (a).
2. **Content tags:** configure Lucindri `contentTags=text,headline` (TASK-0007) to match Indri's
   include tags (`text`, `headline`, `title`, …) so both index the same body. Confirm Indri's
   trectext include-tag set and align.
3. **Analysis alignment:** Indri default **Krovetz** ≈ Lucindri **KStem**; give Indri the same
   stopword list Lucindri uses (Lucene English stop set) or disable stopping in both. Residual
   KStem-vs-Krovetz differences are expected (TASK-0005 Category B) — catalog them, don't treat
   them as bugs.

## 4. The central hypothesis to test: `#weight`

Every topic is wrapped in `#weight`. In Lucindri the parser recognizes `weight` and parses the
per-clause weights, but `createBooleanClause` has **no `#weight` branch** → it builds an
`IndriAndQuery` with the weights applied as `BoostQuery` boosts. So Lucindri's `#weight` ≈ a
**boosted `#combine`**, not necessarily Indri's `#weight` (weighted combination of beliefs).
**Determine whether Lucindri's `#weight` matches Indri's** — if not, all 50 topics diverge, and
that's a finding (`#weight` is what real Indri query generators emit, so it matters) → likely a
follow-up fix task.

## 5. Methodology

1. **Index both** (once): Indri via `IndriBuildIndex` (gz direct, Krovetz, stoplist); Lucindri via
   `BuildIndex` (`documentFormat=trectext`, `contentTags=text,headline`, `kstem/true/true`, gz
   support from §3). Confirm comparable document counts.
2. **Run** all 50 topics + the generated queries in both, TREC format, `count=1000`
   (`IndriRunQuery` vs `IndriSearch`); align μ (dirichlet:2000) and the JM subset.
3. **Compare rankings** (per query + aggregate) — this is a *system agreement* study:
   - **Overlap@k** (fraction of shared docs) at k = 10, 100, 1000.
   - **Rank-biased overlap (RBO)** (top-weighted) and/or **Kendall's τ** on the common docs.
   - Summarize: mean overlap@10/@100, and the queries with the largest disagreement.
4. **Attribute divergences** to: (a) `#weight` semantics (§4), (b) KStem vs Krovetz / stopwords,
   (c) tie-breaking among equal scores (benign), or (d) a real bug (→ new task).
5. **Optional effectiveness check:** with the TREC-8 ad-hoc qrels for 401–450, compute MAP / P@10
   for **each** engine and compare — do they achieve *similar effectiveness*? (Agreement is the
   primary goal; this is a sanity check that both are "reasonable.")

## 6. Harness

Extend `scripts/diff_indri.sh` (or add `scripts/compare_trec.sh`): build both indexes from the
gz corpus, run a topics file through both engines, and emit per-query overlap@k + an aggregate
table. Reuse the normalize-and-diff approach; add the overlap/RBO computation.

## 7. Acceptance criteria

1. Both engines index `t45minusCR` (comparable doc counts); Lucindri reads the gz trectext (§3).
2. All 50 topics + the generated queries run in both engines and produce TREC output.
3. An **agreement report**: per-query and mean overlap@{10,100,1000} (+ RBO/τ), with the
   `#weight` parity question answered explicitly.
4. Divergences triaged (§4); any real bug filed as a new task; expected differences (stemming,
   tie-breaking) noted.
5. Findings recorded (extend `docs/lucindri-vs-indri.md`, or a new `docs/` note) — this is the
   large-scale complement to the 6-doc catalog.

## 8. Out of scope

- Improving retrieval effectiveness (this is a *behavioral agreement* study, not a bake-off).
- Operators Lucindri does not claim (wildcards, `#prior`, passage/extent retrieval, `#wsyn`, …).
- Fixing `#weight` (if it diverges) — file a follow-up; here we only measure and diagnose.

## 9. Validates (regression at scale)

TASK-0002 (`#band`/`#uwN`/`#1`/`#syn`, compound operands, position-0), TASK-0003 (Dirichlet
scoring parity), TASK-0006 (`#scoreif`/`#scoreifnot`/`#not`), TASK-0007 (trectext + `contentTags`)
— all exercised on a real TREC collection with real Indri-language queries.
