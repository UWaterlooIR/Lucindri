# TASK-0023 — Intra-query parallelism: search one query across leaf slices with an executor

**Status:** Draft
**Created:** 2026-07-07
**Owner:** Mark D. Smucker
**Summary:** Make a *single* Lucindri query use multiple threads by giving `IndriIndexSearcher`
a Lucene `ExecutorService`, so the per-document scoring scan is split across leaf slices (the 8
climbmix parts and their segments) instead of running on one core. Rankings and scores must be
byte-for-byte identical to today's sequential run.

---

## 1. Motivation

Today a single query runs on **one thread**. The 8-part MultiReader speeds up *concurrent* queries
(each request gets a thread in the server pool) but does nothing for the *latency of one query*.
Measured on the 553M-doc `climbmix_full_keepstop_v2` index:

| query | latency |
|---|---|
| `#combine("black bear attacks")` (3 terms) | ~2.9 s |
| `#uw10("dog cat")` | ~12.9 s |
| nested `#uw50( #syn(#1("black bear") "bear") #syn(...) )` | ~122 s |
| 25-term `#weight(...)` Korean-War query | ~256 s |

Belief operators score the **union** of their terms' postings; that scan is embarrassingly
parallel across index leaves. Lucene's `IndexSearcher` already supports it — hand it an
`ExecutorService` and it splits the reader's leaves into **slices** and runs one
`CollectorManager` per slice concurrently, merging the per-slice top-*k*. We just don't use that
constructor, and one custom `Weight` has a data race that would corrupt concurrent scoring.

**Goal:** near-linear speedup of the scoring scan (bounded by cores / slice count). The 256 s query
should drop to tens of seconds on a many-core host, with **identical** results.

## 2. How Lucene concurrency works (background for the implementer)

- `new IndexSearcher(reader, executor)` enables concurrent search. With no executor (today) search
  is sequential over `reader.leaves()`.
- `IndexSearcher.slices(leaves)` groups leaves into `LeafSlice[]` (default heuristic:
  `MAX_DOCS_PER_SLICE=250_000`, `MAX_SEGMENTS_PER_SLICE=5`). Each slice is submitted to the executor.
- In Lucene **8.10**, the top-*N* path (`search(Query, int n)` → `searchAfter` → a
  `CollectorManager<TopScoreDocCollector, TopDocs>` → `search(Query, CollectorManager)`) **already
  routes through the executor** when one is present and there is more than one slice. So passing an
  executor should parallelize `search(query, count)` with no call-site change — **verify this during
  implementation** (step 4D); if some path bypasses it, drive `search(Query, CollectorManager)`
  explicitly.
- The per-slice results are combined with `TopDocs.merge(...)` (score, then shardIndex, then docid).

## 3. Scope / non-goals

**In scope:** parallelize the scoring scan of one query; make the custom `Weight`s safe for
concurrent per-leaf `scorer()`; own/size/shut-down the executor; prove identical results and measure
the speedup.

**Explicitly NOT in this task:**
- **Dynamic pruning (WAND / MaxScore / early termination).** Lucindri scorers report
  `getMaxScore()==0` and don't do competitive-score skipping, so this task keeps scoring *every*
  candidate — just on more threads. Pruning is a separate, harder task (it interacts with Indri's
  normalized-geometric-mean + background-smoothing scoring).
- **The proximity/synonym collection-stats pass.** `IndriTermOpWeight.ensureCollectionStats` is an
  O(collection) memoized pass that is already `synchronized`; it runs **once, serially**, and is the
  dominant cost of the 122 s nested-proximity query. This task will *not* parallelize it (a
  complementary future task). Parallelism here helps the per-doc scoring, not that one-time pass.
- **Any change to scores or ranking.** This is a pure performance change; correctness is measured by
  identity with the sequential result (§5).

## 4. Work items

### 4A. Executor-aware `IndriIndexSearcher` constructor
`IndriIndexSearcher` (`LucindriSearcher/.../searcher/IndriIndexSearcher.java`) currently has only
`IndriIndexSearcher(IndexReader r) { super(r); }`. Add
`IndriIndexSearcher(IndexReader r, java.util.concurrent.Executor executor) { super(r, executor); }`.
Keep the single-arg constructor (sequential) for tests/back-compat. Nothing else in the class needs
to change **except** the stats-cache hardening in 4C.

### 4B. Own and size the executor in `LucindriSearchService`
`LucindriSearchService` (line ~95) builds `new IndriIndexSearcher(reader)`. Change it to create a
shared `ExecutorService` and pass it in:
- One executor per service instance (shared by all queries), stored in a field.
- **Daemon threads** with a named `ThreadFactory` (mirror the server pattern), so a stray executor
  never blocks JVM exit.
- **Shut it down in `close()`** (`shutdownNow()` after `reader.close()` — order it so no query is in
  flight).
- **Size:** default to `Runtime.getRuntime().availableProcessors()` (or that capped, e.g.
  `min(cores, 8..16)`). Expose it as a constructor arg / config so `0` or `1` means *no executor*
  (pass `null` → sequential, preserving today's behavior as the escape hatch).
- Add a CLI/config knob: `LucindriServer` gets `--searchThreads N` (default = cores), threaded
  through to the service constructor; `IndriSearch` (batch) can default to sequential or cores —
  **decide** (batch already parallelizes at the query level by running queries back-to-back; intra-
  query threads may or may not help a batch run — note the choice in the progress log).
- **Interaction with the server request pool (call out, don't over-engineer):** the server already
  has an N-thread pool for *concurrent requests*. Intra-query parallelism multiplies load:
  `requestThreads × searchThreads` can oversubscribe cores under concurrent traffic. Document the
  interaction; a sensible default is a **single shared search executor** across all requests (which
  is what 4B gives) so total search parallelism is bounded by one pool, not per-request pools.

### 4C. Thread-safety audit + fixes (the real work)
Per-leaf `Weight.scorer(leafContext)` calls will now run **concurrently on different threads**.
`Scorer`s are inherently per-leaf (safe), but a shared `Weight` with mutable per-`scorer()` state is
a data race. Findings from the current code:

1. **`IndriTermOpWeight.simScorer` — confirmed race, must fix.** It is an *instance field* (line 43)
   assigned inside `getScorer(context)` (line 183: `this.simScorer = similarity.scorer(...)`) and read
   one line later (line 185) to build the leaf length source. Under concurrent leaf scoring, two
   threads overwrite each other's `simScorer`. **Fix:** make it a **local variable** in `getScorer`
   (it is used nowhere else — confirm with a grep before deleting the field).
2. **`IndriIndexSearcher.collectionStatistics` cache — harden.** It is a plain (non-volatile) field,
   lazily computed and does *not* cache the `null` (no-terms) case. In practice it is populated during
   `createWeight` (single-threaded, before slices run), and the executor submission provides a
   happens-before, so it is *probably* fine — but make it robust: either compute it **eagerly in the
   constructor**, or make the field `volatile` / guard the compute with `synchronized`. Prefer eager
   or `synchronized` so a future call from a worker thread can't double-compute or see a torn value.
3. **Audit every `IndriWeight` subclass for per-`scorer()` mutable writes** (same pattern as #1):
   `IndriTermQuery`/`IndriTermWeight`, `IndriWeight`, `IndriOrWeight`, `IndriWeightedSum`,
   `IndriMaxWeight`, `IndriNotWeight`, `IndriSynonymWeight`, `IndriBandWeight`, `IndriNearWeight`,
   `IndriWindowWeight`, `IndriFilterRequireWeight`, `IndriFilterRejectWeight`, `IndriTermOpWeight`.
   For each, confirm all instance fields are either `final`/set-once in the constructor
   (`createWeight` time, single-threaded → safe) or not written in `scorer()`/`getScorer()`. Fix any
   that write shared state per leaf (make it local, like #1). Record the audit result in the progress
   log so it's auditable.
4. **`ensureCollectionStats` is already `synchronized`** (line 143) and memoizes via `statsComputed` —
   safe under concurrency (first leaf thread does the O(collection) pass; others block then read the
   memoized result). No change; just noted so the implementer doesn't "optimize" the lock away.

### 4D. Confirm the search path actually uses the executor
Add an assertion/log or step-debug to confirm `search(query, count)` runs on multiple threads (e.g.
log the thread name in a scorer, or check `leafSlices.length > 1` and CPU usage). If Lucene 8.10's
top-*N* path does not dispatch to the executor for our query shape, switch the service to call
`searcher.search(query, collectorManager)` with an explicit
`TopScoreDocCollectorManager`-equivalent. (Expected: no change needed — verify.)

## 5. Correctness requirement — identical to sequential

Because Lucindri does **no competitive-score pruning** (§3), the parallel run scores the *same*
candidate set as the sequential run, so **scores must be identical** and the ranking must match.
Two things to verify explicitly:

- **Score identity.** Same query, same index, executor vs no-executor → identical `(docno, score)`
  pairs for the full result list.
- **Tie-break equivalence (subtle — call out).** For documents with *exactly equal* scores, global
  order is broken by docid. Sequential collection and per-slice-then-`TopDocs.merge` can, in
  principle, order equal-score docs differently (merge tie-breaks by score→shardIndex→docid). Indri
  scores are floats so exact ties are uncommon but possible (e.g. identical tf/length on a one-term
  query). **Verify** the tie order is preserved; if it differs, judge whether that is acceptable (it
  doesn't move MAP but changes exact docno order for ties) and document the decision. If it must be
  identical, ensure per-slice collectors use global docids and consistent shard indices so `merge`
  reproduces sequential order.

## 6. Testing plan (required — see CLAUDE.md testing policy)

1. **Parallel == sequential unit test (new).** Build a **multi-segment / multi-part** `TestIndex`
   (the fixture already supports multiple sub-indexes in a `MultiReader`), run a handful of queries
   (a plain `#combine`, a `#weight`, a `#uwN`, a query with a `#not`, and a query with tie-prone
   equal scores) through **both** an executor-backed and a sequential `IndriIndexSearcher`, and assert
   the returned `TopDocs` (docids + scores + order) are identical. This is the core regression.
2. **Concurrency stress.** Run the same query many times on the executor searcher (and/or from
   several threads against one service) and assert results are stable run-to-run — catches the
   `simScorer`-style races (a flaky/nondeterministic score = an unfixed race).
3. **401–450 end-to-end (existing kit `scripts/eval-401-450/`).** Re-run the TREC-8 eval with the
   executor enabled and confirm **MAP 0.2499 / P@10 0.4340** (the recorded baseline) are unchanged,
   and server==batch still 50/50.
4. **Benchmark.** Record wall-clock for the four queries in §1 sequential vs N threads on the full
   index (background the heavy runs — do not foreground multi-minute queries). Capture the speedup in
   the progress log; this is the payoff evidence.
5. `mvn test` green in `LucindriSearcher` and `LucindriServer` (the modules touched).

## 7. Risks / open questions

- **Slice count vs pool size.** With 553M docs the default slicer yields *many* slices; the executor
  pool bounds real concurrency. May want to override `IndexSearcher.slices()` to tune (fewer, larger
  slices) — treat as a knob, measure before tuning.
- **Memory / IO.** More concurrent leaf scans = more concurrent postings IO and heap churn. Watch
  `Xmx` under load; the server's `-Xmx16G` default may need review with `searchThreads > 1`.
- **The proximity pass stays serial** (§3) — the 122 s nested `#uw50` won't improve much from this
  task alone; set expectations. Its win comes from a separate collection-stats task.
- **Batch vs server default** for `searchThreads` (§4B) — decide and document.
