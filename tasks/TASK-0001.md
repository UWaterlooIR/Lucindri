# TASK-0001 — Add a JSONL document parser for the climbmix corpus

- **Status:** Ready
- **Created:** 2026-07-03
- **Owner:** Mark D. Smucker
- **Summary:** Add a new Lucindri indexer document parser, selected by
  `documentFormat=jsonl`, that reads gzipped JSON-Lines shards from the climbmix corpus and
  indexes each line as one document, mapping `docid → externalId` and `contents → fulltext`.

> Read `../CLAUDE.md` (repo root) before starting. It explains the build, the parser
> framework, and the field-name rules referenced below.

---

## 1. Motivation / context

We want to index the climbmix corpus with Lucindri and then run Indri-style queries against
it. The corpus is stored as gzipped JSON-Lines. Lucindri's existing `json` parser is a poor
fit for three reasons (all verified in source):

1. It cannot read gzip — it opens files with a plain `FileInputStream`. Our shards are `.gz`,
   and the corpus is ~400B tokens, so decompressing on disk is not acceptable.
2. It maps different JSON keys than ours (`text`, `docno`, `fields`) and would index nothing
   from our records (it would actually NPE / null-out on our data).
3. It stores the external id under field name `"id"`, which the **searcher never reads** — the
   searcher prints the TREC DOCNO from a field literally named `"externalId"`. So even if it
   ran, TREC output would have a blank id column.

Rather than patch the generic parser (and risk breaking other users), we add a small,
dedicated parser for our format. This is the intended extension mechanism.

## 2. The data

- Location: `/ssd-8TB/corpora/climbmix-400b-corpus-jsonl/`
- 79 files named `shard_00001.jsonl.gz` … `shard_00079.jsonl.gz` (gzipped JSON Lines: one JSON
  object per line).
- Each line looks like (keys observed by sampling `shard_00001.jsonl.gz`):
  ```json
  {"id":"shard_00001_0",
   "contents":"* Visions seen by the Dying of Persons Known by them to be Dead ...",
   "docid":"shard_00001_0",
   "source_file":"/mnt/e/corpora/climbmix-400b-shuffle/shard_00001.parquet",
   "row_number":"0",
   "mode":"generated_segment_row"}
  ```
- Mapping we want:
  - `docid`   → external document id  → stored field **`externalId`** (this becomes TREC DOCNO)
  - `contents`→ document body/text    → stored field **`fulltext`** (default query field)
  - All other keys (`id`, `source_file`, `row_number`, `mode`) are ignored.
- Each shard has tens of thousands of records (≈86k+ lines observed in a sample). Note the
  records are document *segments* (`mode: generated_segment_row`), i.e. the retrieval unit is a
  passage, not a full document. That is intended; no need to reassemble.

## 3. Design

Create `ClimbmixJsonlDocumentParser` in
`LucindriIndexer/src/main/java/org/lemurproject/lucindri/indexer/documentparser/`.
It extends `DocumentParser` and:

- On construction, recursively collects every file under `options.getDataDirectory()` (reuse
  the recursive `listFiles` pattern from `TrecTextDocumentParser`, so a directory of shards
  works; a single-file path should also work — guard for that).
- Opens one file at a time as a stream. If the filename ends with `.gz`, wrap the
  `FileInputStream` in `java.util.zip.GZIPInputStream`; otherwise read it plain. Decode UTF-8.
- Streams line by line. For each non-empty line, parse it with Gson and emit a `ParsedDocument`
  with these `ParsedDocumentField`s (all non-numeric):
  - `"externalId"` = value of JSON key `docid`
  - `INTERNALID_FIELD` (`"internalId"`) = a running 1-based counter (nice for debugging)
  - `FULLTEXT_FIELD` (`"fulltext"`) = value of JSON key `contents`
- Constant memory: keep only one `BufferedReader` open at a time; advance to the next file when
  the current one is exhausted.

Robustness:
- Skip blank lines.
- If a record is missing `docid` or `contents`, do not crash the whole run — either skip the
  record (log a count) or substitute a safe default (e.g. `internalId` value for a missing
  `docid`, empty string for missing `contents`). Prefer skip-with-count; the writer already
  skips null-content fields.
- Use `gson.fromJson(line, JsonObject.class)` and pull fields via
  `obj.has("docid") ? obj.get("docid").getAsString() : null`, to tolerate the extra keys.

Gson is already a dependency of `LucindriIndexer` (`com.google.code.gson:gson`), and
`GZIPInputStream` is JDK built-in — **no new Maven dependencies required.**

### Reference sketch (not final; adapt names/style to the module)

```java
package org.lemurproject.lucindri.indexer.documentparser;

import java.io.*;
import java.nio.charset.StandardCharsets;
import java.util.*;
import java.util.zip.GZIPInputStream;
import com.google.gson.Gson;
import com.google.gson.JsonObject;
import org.lemurproject.lucindri.indexer.domain.IndexingConfiguration;
import org.lemurproject.lucindri.indexer.domain.ParsedDocument;
import org.lemurproject.lucindri.indexer.domain.ParsedDocumentField;

public class ClimbmixJsonlDocumentParser extends DocumentParser {

    private final Iterator<File> fileIterator;
    private BufferedReader br;
    private final Gson gson = new Gson();
    private int docNum = 0;

    public ClimbmixJsonlDocumentParser(IndexingConfiguration options) throws IOException {
        List<File> files = new ArrayList<>();
        File root = new File(options.getDataDirectory());
        if (root.isFile()) files.add(root); else listFiles(root, files);
        Collections.sort(files);                 // deterministic shard order
        fileIterator = files.iterator();
        openNext();
    }

    private static void listFiles(File dir, List<File> out) {
        File[] fs = dir.listFiles();
        if (fs == null) return;
        for (File f : fs) { if (f.isFile()) out.add(f); else listFiles(f, out); }
    }

    private void openNext() throws IOException {
        if (fileIterator.hasNext()) {
            File f = fileIterator.next();
            InputStream in = new FileInputStream(f);
            if (f.getName().endsWith(".gz")) in = new GZIPInputStream(in);
            br = new BufferedReader(new InputStreamReader(in, StandardCharsets.UTF_8));
        } else {
            br = null;
        }
    }

    @Override public boolean hasNextDocument() { return br != null; }

    @Override public ParsedDocument getNextDocument() throws IOException {
        String line;
        while (br != null) {
            if ((line = br.readLine()) == null) { br.close(); openNext(); continue; }
            if (line.isEmpty()) continue;
            JsonObject o = gson.fromJson(line, JsonObject.class);
            String docid    = o.has("docid")    ? o.get("docid").getAsString()    : null;
            String contents = o.has("contents") ? o.get("contents").getAsString() : null;
            if (contents == null) continue;      // nothing to index; skip
            docNum++;
            ParsedDocument doc = new ParsedDocument();
            doc.setDocumentFields(new ArrayList<>());
            doc.addDocumentField(new ParsedDocumentField("externalId",
                    docid != null ? docid : String.valueOf(docNum), false));
            doc.addDocumentField(new ParsedDocumentField(INTERNALID_FIELD,
                    String.valueOf(docNum), false));
            doc.addDocumentField(new ParsedDocumentField(FULLTEXT_FIELD, contents, false));
            return doc;
        }
        return null;
    }
}
```


## 6. Test — index one shard end to end

Create a properties file, e.g. `/tmp/climbmix-test.properties`:
```
documentFormat=jsonl
indexingPlatform=lucene
dataDirectory=/ssd-8TB/corpora/climbmix-400b-corpus-jsonl
indexDirectory=/ssd-8TB/indexes
indexName=climbmix_test
indexFullText=true
fieldNames=
stemmer=kstem
removeStopwords=true
ignoreCase=true
```
For a quick smoke test, point `dataDirectory` at a directory containing just ONE shard (copy or
symlink `shard_00001.jsonl.gz` into a scratch dir) so it finishes fast, since the parser reads
every file in the directory.

Run:
```bash
cd /ssd-8TB/git-repos/Lucindri && java -jar -Xmx4G \
  LucindriIndexer/target/LucindriIndexer-1.45-jar-with-dependencies.jar /tmp/climbmix-test.properties
```

### Acceptance criteria
1. Indexer completes without exceptions and reports a document count in the tens of thousands
   for one shard.
2. A search returns our `docid` values in the TREC-format DOCNO column (not blank, not internal
   numbers). To verify, build the searcher too (if not already built) and run a trivial query
   file against the `climbmix_test` index, e.g. a common word:
   ```
   <parameters>
     <index>/ssd-8TB/indexes/climbmix_test</index>
     <trecFormat>true</trecFormat>
     <rule>dirichlet:2000</rule>
     <count>10</count>
     <query><number>1</number><text>#combine(the visions death)</text></query>
   </parameters>
   ```
   Confirm the 3rd column of the TREC output contains ids like `shard_00001_0`.
3. Gzipped input is read directly (no decompressed copies created on disk).
