# TASK-0007 — Fix Lucindri's TREC-text (`trectext`) document parser

- **Status:** Done
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** `TrecTextDocumentParser` does not correctly parse standard TREC-text: it **ignores
  `<DOCNO>`** for standard `<DOC>`-first documents (assigning sequential ids instead), **NPEs when
  `fieldNames` is empty**, and puts **markup tokens into `fulltext`** (it indexes the whole raw
  document buffer, tags and all). Rewrite it to parse standard TREC correctly — external id from
  `<DOCNO>`, markup-free body in `fulltext` — using C++ Indri's TREC parsing as the reference for
  correct behavior.

> Read `../CLAUDE.md` first (esp. the Testing policy and the field-name rules). This is in the
> **`LucindriIndexer`** module (JUnit already wired there from TASK-0001). Found while building the
> [[TASK-0005]] comparison harness (`docs/lucindri-vs-indri.md`).

---

## 1. The bugs (verified)

`LucindriIndexer/.../documentparser/TrecTextDocumentParser.java`, `getNextDocument()`:

1. **`<DOCNO>` is ignored for standard TREC.** DOCNO is extracted only by
   `if (nextLine.startsWith("<DOCNO>"))` on the **outer** line (~line 106), but for a standard
   document the outer line that starts a doc is `<DOC>` (line 110), and the `<DOCNO>` line is then
   consumed by the **inner** loop (117–130) without extraction. So `docno` stays empty and falls
   back to `String.valueOf(docNum)`. **Verified:** a `<DOCNO>aaa</DOCNO>` inside `<DOC>…</DOC>`
   yields external id `"1"`, not `"aaa"`; DOCNO is only picked up if it appears **before** `<DOC>`
   (non-standard). Net effect: TREC output DOCNOs are wrong (sequential ints, not the corpus ids).
2. **NPE on empty `fieldNames`.** With `indexFullText=true` and empty `fieldNames`,
   `options.getIndexFields()` is null, and `fieldsToIndex.contains(BODY_FIELD)` throws NPE — caught
   as "Could not parse document", so **0 documents index**. (Workaround today: set `fieldNames` to a
   dummy value.)
3. **`fulltext` contains markup.** `fulltext` is set to `docBuffer.toString()` — the entire raw
   buffer including `<DOC>`, `<DOCNO>`, the docno value, and `<TEXT>` tags — so tokens like `doc`,
   `docno`, `text` pollute the body and the collection statistics.
4. **Minor:** `docNum++` runs per outer-line iteration (a quasi-line counter), and DOCNO parsing
   (`substring(7, len-8)`) assumes an exact single-line `<DOCNO>x</DOCNO>` with no surrounding
   whitespace.

(Cataloged as caveats in `docs/lucindri-vs-indri.md`; TASK-0005 worked around them by comparing
Lucindri in JSONL — its actual target format. `trectext` is still a claimed `documentFormat` in the
README and should parse standard TREC correctly.)

## 2. Reference: correct TREC parsing (study C++ Indri)

C++ Indri ([[indri-cpp-reference]], `/ssd-8TB/git-repos/indri/`) is the reference for correct
behavior. Study:
- **`src/FileClassEnvironmentFactory.cpp`** — defines the `trectext` file class: which tag delimits
  a document, which tag(s) carry the **docno/metadata**, and which tags are indexed vs. treated as
  structure. This is the authoritative tag mapping.
- **`include/indri/TaggedTextParser.hpp`** and **`src/TaggedDocumentIterator.cpp`** — the tag-based
  document iteration and metadata extraction (how DOCNO is read from within the document and how the
  indexed content excludes markup).

Standard TREC-text shape to support: a document is delimited by `<DOC> … </DOC>`; its id is the
text of `<DOCNO> … </DOCNO>` **inside** the document (trim surrounding whitespace); the indexed
content is the document's text **without** the markup tags.

**Concrete Indri `trectext` config** (read from `FileClassEnvironmentFactory.cpp`): documents are
`<DOC>`…`</DOC>`; the **metadata** tags are `docno`, `title`, `author` — so **`docno` is the
document id, read as metadata, not body**; the indexed **content** comes from include tags, chiefly
`<TEXT>` (also `headline`/`title`/`date`/… when present), tokenized with markup removed. Mapping for
Lucindri's model: **`externalId` = the `<DOCNO>` text; `fulltext` = the `<TEXT>` content,
markup-free.** (`docno`/`title` appearing in both metadata and content is Indri's business; Lucindri
only needs id + markup-free body, plus optional named fields on request.)

## 3. Fix

Rewrite `TrecTextDocumentParser.getNextDocument()` to:
- Accumulate one document from `<DOC>` to `</DOC>`.
- Extract the external id from the `<DOCNO>…</DOCNO>` tag **wherever it occurs within the doc**
  (whitespace-tolerant); set the `externalId` field to it (fallback to a real per-document counter
  only if genuinely absent). Store under field name **`externalId`** (the searcher reads that;
  see `CLAUDE.md`).
- Set **`fulltext` to the markup-free text content** (strip tags), not the raw buffer, gathered
  from the **configurable content tags** (below). Jsoup is already a dependency and can extract text.
- **Configurable content tags.** Add an indexing property **`contentTags`** (comma-separated,
  case-insensitive; **default `text`**) listing which tags inside `<DOC>` contribute to `fulltext`
  — e.g. `contentTags=text,headline,graphic`. The parser gathers the markup-free text of every
  configured tag that is present, **in document order**, and concatenates it into `fulltext`
  (missing tags skipped). This mirrors Indri's include-tags model and keeps the default (`text`
  only) backward-compatible. It is **orthogonal to `fieldNames`** (which stores separate *named*
  fields); `contentTags` shapes the searchable body. Note: concatenation creates a token boundary
  between tags — gather in document order (so proximity positions are sane) and optionally insert a
  small position gap between tags.
- Guard `fieldNames`/`getIndexFields()` being null so empty `fieldNames` **does not NPE**; optional
  fields (title, etc.) are indexed only when requested and remain out of `fulltext`.
- Keep behavior conservative for the other tags; match Indri's `trectext` semantics where they
  differ.

## Implementation plan (ordered)

Test-first, in the **indexer** module (JUnit already wired from TASK-0001; mirror
`ClimbmixJsonlDocumentParserTest`). All work is in `TrecTextDocumentParser`.

1. **Branch** off `master` (e.g. `fix/trectext-parser`).
2. **Null-safe `fieldNames` first (unblocks everything).** In the constructor, guard
   `options.getIndexFields()` being null — default `fieldsToIndex` to an empty list. Verify a
   standard-TREC doc with empty `fieldNames` now indexes instead of throwing "Could not parse
   document" (currently 0 docs).
3. **Fail-first tests** (confirm they FAIL on the current parser before fixing):
   - `<DOC><DOCNO>d-42</DOCNO><TEXT>…</TEXT></DOC>` → `ParsedDocument.externalId == "d-42"`
     (currently `"1"`).
   - `fulltext` contains the body tokens and **not** `doc`/`docno`/`text` (currently raw buffer
     includes markup).
   - `<DOCNO> d-42 </DOCNO>` (whitespace) → `"d-42"`.
   - Two docs in one file → distinct, correct ids.
4. **Rewrite `getNextDocument()`** — accumulate a whole `<DOC>…</DOC>` block into a buffer (keep the
   line-based reader; be tolerant of `<DOC>`/`</DOC>` with surrounding whitespace), then parse the
   block once:
   - **externalId** = the text of the `<DOCNO>` element, trimmed, case-insensitive, **wherever it
     appears within the doc** (not only as the outer line). Fall back to a real per-document counter
     only when `<DOCNO>` is genuinely absent. Store under field name **`externalId`** (`CLAUDE.md`:
     the searcher reads that).
   - **fulltext** = the **markup-free** text of the **configured content tags** (`contentTags`,
     default `text`), gathered in **document order** — NOT `docBuffer.toString()`. Add `contentTags`
     to `IndexingConfiguration` + `IndexOptionsFactory` (default `text`). Jsoup is already a
     dependency and already used here: `Jsoup.parse(block)`, then `getElementsByTag("docno").text()`
     for the id, and for the body select the union of content-tag elements in document order (e.g.
     `doc.select("text, headline, graphic")`) and concatenate `.text()`. (Jsoup lowercases tags, so
     it matches `<DOCNO>`/`<TEXT>`/`<HEADLINE>`.)
   - Fix `docNum` to increment **once per document**, not per outer line.
   - Keep optional named fields (`title`, `heading`, `url`, `body`) only when requested via
     `fieldNames`, null-safe, and each markup-free; never fold markup into `fulltext`.
5. **Verify Jsoup handles the pseudo-SGML** (the §3-step tests gate this). Jsoup's HTML parser can
   mis-nest unknown tags; if the DOCNO/TEXT extraction tests fail because of that, switch to
   whitespace/case-tolerant regex extraction (`(?is)<DOCNO>(.*?)</DOCNO>`, `<TEXT>(.*?)</TEXT>` then
   strip inner tags) — but let the tests decide, don't pre-optimize.
6. **Remaining tests** green (`mvn test` in `LucindriIndexer`).
7. **End-to-end:** index a tiny standard-TREC corpus, run a query, confirm the corpus DOCNOs appear
   in the TREC output (not sequential ints) and that querying `doc`/`text` does not match on markup.
8. **Cross-check vs Indri (optional but ideal):** add a `trectext`-for-Lucindri variant to
   `scripts/diff_indri.sh` (or a one-off), and confirm Lucindri-trectext now agrees with
   Indri-trectext on ids/doc sets — the same corpus both engines, both via trectext. (The main
   harness intentionally uses JSONL for Lucindri; this variant just validates the parser fix.)
9. **Close out:** rebuild `LucindriIndexer-1.45-jar-with-dependencies.jar`; set `Status: Done`;
   progress-log the extraction mechanism chosen and before/after ids; update the
   `docs/lucindri-vs-indri.md` "caveats" note (the trectext quirks are resolved); commit.

## 4. Tests (per the testing policy)

Add to the indexer module (reuse the JUnit setup from TASK-0001; mirror `ClimbmixJsonlDocumentParserTest`):
1. **DOCNO extraction:** a standard TREC doc (`<DOC><DOCNO>d-42</DOCNO><TEXT>…</TEXT></DOC>`) yields
   a `ParsedDocument` with `externalId == "d-42"` (fail-first: currently `"1"`).
2. **Markup-free body:** `fulltext` contains the text tokens and **not** `doc`/`docno`/`text` (assert
   those tag tokens are absent).
3. **Empty `fieldNames` does not NPE** and indexes the document.
4. **Whitespace tolerance:** `<DOCNO> d-42 </DOCNO>` → `externalId == "d-42"`.
5. **Multiple docs** in one file parse with correct, distinct ids.
6. **End-to-end:** index a tiny TREC corpus, run a query, confirm the corpus DOCNOs appear in the
   TREC output (not sequential ints) and that a query for `text`/`doc` does not match on markup.

7. **Configurable content tags:** with `contentTags=text,headline`, a doc's `<HEADLINE>` tokens are
   present in `fulltext` alongside `<TEXT>`; with the default (property absent → `text`), the
   `<HEADLINE>` tokens are **not** in `fulltext`. A `<HEADLINE>` preceding `<TEXT>` is gathered
   first (document order).

## 5. Acceptance criteria

1. Standard TREC-text (`<DOC>`-first, `<DOCNO>` inside) indexes with `externalId` = the real DOCNO
   and `fulltext` = markup-free content; empty `fieldNames` works.
2. `contentTags` works: default is `text` only (backward-compatible); configuring extra tags (e.g.
   `text,headline,graphic`) adds their markup-free text to `fulltext` in document order.
3. Unit tests (§4) green; `mvn test` green in `LucindriIndexer`.
4. Optional: add a `trectext` variant to `scripts/diff_indri.sh` (or a note) showing Lucindri-trectext
   now agrees with Indri-trectext on doc sets/ids.
5. Rebuild `LucindriIndexer-1.45-jar-with-dependencies.jar`.
6. Behavior matches Indri's `trectext` for the standard shape (spot-check a few docs).

## 6. Out of scope

- Full TREC/TIPSTER format coverage (SGML entities, nested tags, multiple metadata fields) beyond
  the common `<DOC>/<DOCNO>/<TEXT>` shape — do the common case correctly; note anything deferred.
- The other document parsers (WSJ, gov2, etc.) — this task is `trectext` only.
- Lucindri's primary corpus path is JSONL (`climbmix`, TASK-0001); this fixes the claimed
  `trectext` format for correctness/parity, not because the climbmix work needs it.
