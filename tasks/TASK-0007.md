# TASK-0007 — Fix Lucindri's TREC-text (`trectext`) document parser

- **Status:** Ready
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** `TrecTextDocumentParser` does not correctly parse standard TREC-text: it **ignores
  `<DOCNO>`** for standard `<DOC>`-first documents (assigning sequential ids instead), **NPEs when
  `fieldNames` is empty**, and puts **markup tokens into `fulltext`** (it indexes the whole raw
  document buffer, tags and all). Rewrite it to parse standard TREC correctly — external id from
  `<DOCNO>`, markup-free body in `fulltext` — using C++ Indri's TREC parsing as the reference for
  correct behavior.

> Read `../CLAUDE.md` first (esp. the Testing policy and the field-name rules). This is in the
> **`LucindriIndexer`** module (JUnit already wired there from TASK-0001). Found while building the
> [[TASK-0005]] comparison harness (`docs/lucindri-vs-indri.md`).

---

## 1. The bugs (verified)

`LucindriIndexer/.../documentparser/TrecTextDocumentParser.java`, `getNextDocument()`:

1. **`<DOCNO>` is ignored for standard TREC.** DOCNO is extracted only by
   `if (nextLine.startsWith("<DOCNO>"))` on the **outer** line (~line 106), but for a standard
   document the outer line that starts a doc is `<DOC>` (line 110), and the `<DOCNO>` line is then
   consumed by the **inner** loop (117–130) without extraction. So `docno` stays empty and falls
   back to `String.valueOf(docNum)`. **Verified:** a `<DOCNO>aaa</DOCNO>` inside `<DOC>…</DOC>`
   yields external id `"1"`, not `"aaa"`; DOCNO is only picked up if it appears **before** `<DOC>`
   (non-standard). Net effect: TREC output DOCNOs are wrong (sequential ints, not the corpus ids).
2. **NPE on empty `fieldNames`.** With `indexFullText=true` and empty `fieldNames`,
   `options.getIndexFields()` is null, and `fieldsToIndex.contains(BODY_FIELD)` throws NPE — caught
   as "Could not parse document", so **0 documents index**. (Workaround today: set `fieldNames` to a
   dummy value.)
3. **`fulltext` contains markup.** `fulltext` is set to `docBuffer.toString()` — the entire raw
   buffer including `<DOC>`, `<DOCNO>`, the docno value, and `<TEXT>` tags — so tokens like `doc`,
   `docno`, `text` pollute the body and the collection statistics.
4. **Minor:** `docNum++` runs per outer-line iteration (a quasi-line counter), and DOCNO parsing
   (`substring(7, len-8)`) assumes an exact single-line `<DOCNO>x</DOCNO>` with no surrounding
   whitespace.

(Cataloged as caveats in `docs/lucindri-vs-indri.md`; TASK-0005 worked around them by comparing
Lucindri in JSONL — its actual target format. `trectext` is still a claimed `documentFormat` in the
README and should parse standard TREC correctly.)

## 2. Reference: correct TREC parsing (study C++ Indri)

C++ Indri ([[indri-cpp-reference]], `/ssd-8TB/git-repos/indri/`) is the reference for correct
behavior. Study:
- **`src/FileClassEnvironmentFactory.cpp`** — defines the `trectext` file class: which tag delimits
  a document, which tag(s) carry the **docno/metadata**, and which tags are indexed vs. treated as
  structure. This is the authoritative tag mapping.
- **`include/indri/TaggedTextParser.hpp`** and **`src/TaggedDocumentIterator.cpp`** — the tag-based
  document iteration and metadata extraction (how DOCNO is read from within the document and how the
  indexed content excludes markup).

Standard TREC-text shape to support: a document is delimited by `<DOC> … </DOC>`; its id is the
text of `<DOCNO> … </DOCNO>` **inside** the document (trim surrounding whitespace); the indexed
content is the document's text **without** the markup tags.

## 3. Fix

Rewrite `TrecTextDocumentParser.getNextDocument()` to:
- Accumulate one document from `<DOC>` to `</DOC>`.
- Extract the external id from the `<DOCNO>…</DOCNO>` tag **wherever it occurs within the doc**
  (whitespace-tolerant); set the `externalId` field to it (fallback to a real per-document counter
  only if genuinely absent). Store under field name **`externalId`** (the searcher reads that;
  see `CLAUDE.md`).
- Set **`fulltext` to the markup-free text content** (strip tags), not the raw buffer. Jsoup is
  already a dependency and can extract text; or drive off the tag structure like Indri.
- Guard `fieldNames`/`getIndexFields()` being null so empty `fieldNames` **does not NPE**; optional
  fields (title, etc.) are indexed only when requested and remain out of `fulltext`.
- Keep behavior conservative for the other tags; match Indri's `trectext` semantics where they
  differ.

## 4. Tests (per the testing policy)

Add to the indexer module (reuse the JUnit setup from TASK-0001; mirror `ClimbmixJsonlDocumentParserTest`):
1. **DOCNO extraction:** a standard TREC doc (`<DOC><DOCNO>d-42</DOCNO><TEXT>…</TEXT></DOC>`) yields
   a `ParsedDocument` with `externalId == "d-42"` (fail-first: currently `"1"`).
2. **Markup-free body:** `fulltext` contains the text tokens and **not** `doc`/`docno`/`text` (assert
   those tag tokens are absent).
3. **Empty `fieldNames` does not NPE** and indexes the document.
4. **Whitespace tolerance:** `<DOCNO> d-42 </DOCNO>` → `externalId == "d-42"`.
5. **Multiple docs** in one file parse with correct, distinct ids.
6. **End-to-end:** index a tiny TREC corpus, run a query, confirm the corpus DOCNOs appear in the
   TREC output (not sequential ints) and that a query for `text`/`doc` does not match on markup.

## 5. Acceptance criteria

1. Standard TREC-text (`<DOC>`-first, `<DOCNO>` inside) indexes with `externalId` = the real DOCNO
   and `fulltext` = markup-free content; empty `fieldNames` works.
2. Unit tests (§4) green; `mvn test` green in `LucindriIndexer`.
3. Optional: add a `trectext` variant to `scripts/diff_indri.sh` (or a note) showing Lucindri-trectext
   now agrees with Indri-trectext on doc sets/ids.
4. Rebuild `LucindriIndexer-1.45-jar-with-dependencies.jar`.
5. Behavior matches Indri's `trectext` for the standard shape (spot-check a few docs).

## 6. Out of scope

- Full TREC/TIPSTER format coverage (SGML entities, nested tags, multiple metadata fields) beyond
  the common `<DOC>/<DOCNO>/<TEXT>` shape — do the common case correctly; note anything deferred.
- The other document parsers (WSJ, gov2, etc.) — this task is `trectext` only.
- Lucindri's primary corpus path is JSONL (`climbmix`, TASK-0001); this fixes the claimed
  `trectext` format for correctness/parity, not because the climbmix work needs it.
