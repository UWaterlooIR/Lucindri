# TASK-0016 — Revisit how natural language is specified in the query language (the `.`/`:` punctuation-vs-notation collision)

- **Status:** Done (implemented 2026-07-06, branch claude/indri-query-language-parsing-90jguk)
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** The Indri/Lucindri query language does not cleanly separate **natural-language input**
  from **structured-query notation**. It applies an analyzer (tokenizer + stemmer + stopper) to terms,
  yet also reserves ordinary punctuation — `.` (period) and `:` (colon) — as *field-restriction*
  syntax, and strips quotes. The result: a user cannot type natural language like `U.S.A.`,
  `google.com`, or `section 3.14` and have it behave sensibly. Worse, the parser tokenizes on this
  punctuation **before** the analyzer, so the *same surface string is tokenized differently at query
  time than at index time* — making some indexed tokens effectively unsearchable, silently. This task
  is to **investigate and propose** how to fix the natural-language/structured-notation interference.
  The solution is not predetermined.

## The problem (owner's framing)

> The whole "said." issue is that the query language does not make a clean distinction between natural
> language input and the query language. There is an interference between the notion that it applies a
> parser, stemmer, and stopper but also wants to respect notation such as `.` and `:`. The user does
> not know how `U.S.A.` is going to be parsed. Will it turn into `usa`, or `u s a`, or something else?
> The user has to know in advance to enter `usa` or `#1(u s a)` or something else — they certainly
> cannot enter `U.S.A.` today and have it be natural-language input. This is an aspect of the Indri
> query language that the creators never grokked; it has always been a frustrating part of using it.

## Evidence (probed 2026-07-05) — index-time and query-time tokenization DIVERGE

Index-time analyzer (what a document's text becomes) vs query-time parser (what the same text becomes
as a query):

| surface text | index-time tokens (analyzer) | query-time (parser) | can they match? |
|---|---|---|---|
| `U.S.A.` | `[u.s.a]` (one token) | term `u` in field `s.a.` | **no** |
| `google.com` | `[google.com]` | term `google` in field `com` | **no** |
| `3.14` | `[3.14]` | term `3` in field `14` | **no** |
| `A.I.` | `[a.i]` | (vanishes: stopword `a` in field `i.`) | **no** |
| `said.` | `[said]` | term `said` in field `` (empty) | **no** |
| `covid-19` | `[covid][19]` | `covid`, `19` | yes (agrees) |

So a term the indexer stores as `u.s.a` is **unreachable**: any query containing `u.s.a` is split on the
first `.` into `u`.`s.a`, which is a term in a nonexistent field that matches nothing — with no error.
Quotes are also silently deleted (`parseQuery` strips `"`/`'`), so phrases can't be typed either.

## Root cause (grounded in `IndriQueryParser`)

- `parseQuery` rewrites the whole query string up front: strips `"` and `'`, turns `+`→space and
  **`:`→`.`**, before any parsing.
- `popTerm` splits each whitespace token on the **first `.`** into `term.field` (field = everything
  after the first dot, lowercased), then hands only the *term* part to the analyzer via
  `tokenizeString`.
- So the parser pre-tokenizes on punctuation that the **index-time analyzer treats as part of a token**
  (Lucene's `StandardTokenizer` keeps `u.s.a`, `google.com`, `3.14` whole). Query-time and index-time
  tokenization therefore disagree for any token containing `.`/`:`. The analyzer is *not* the single
  source of tokenization truth; the parser front-runs it.

## Why this is fundamentally hard (the tension)

The language tries to be two things at once with one character set: a **structured query language**
(operators `#…`, field restriction via `.`/`:`, weights, quotes) **and** a way to type **free natural
language** — and `.`/`:`/`"` are meaningful in both. Any fix must resolve: *when is a `.` field syntax
vs. ordinary punctuation?* And it must keep **query-time tokenization consistent with index-time
tokenization**, or matching silently fails. There is also an **Indri-compatibility** cost: Lucindri
reimplements the Indri query language, so changing the notation diverges from Indri (which has the same
wart) — a deliberate-divergence decision the owner must make.

## Design space (options to evaluate — not yet chosen)

1. **Make the analyzer the single tokenizer.** Don't pre-split terms in the parser at all; hand each
   raw term (e.g. `U.S.A.`) to the same analyzer the indexer uses, so query tokens == index tokens by
   construction. Field restriction then needs an **explicit, non-colliding operator** instead of `.` —
   e.g. `#field(h1 term)` / a dedicated syntax — so `.`/`:` are just characters the analyzer sees.
2. **Context-sensitive delimiter.** Treat `.`/`:` as a field separator **only** when the right-hand
   side is a *known indexed field name*; otherwise pass the token through untouched. Reduces collisions
   but is still ambiguous (a field literally named `com` would re-break `google.com`) and couples the
   parser to the index schema.
3. **Escaping / explicit quoting for literals.** Allow the user to mark literal text so punctuation is
   not interpreted (and restore real phrase quotes as an ordered window). Puts the burden on the user.
4. **Two explicit modes.** A natural-language mode (everything is text, analyzer-tokenized, maybe an
   implicit `#combine`) vs a structured mode (operators/fields honored). A query is one or the other,
   chosen by a flag/wrapper, not mixed ambiguously.
5. **Restore phrase quotes** regardless of the above (`"new york"` → ordered window) — quotes are
   currently just deleted.

Likely the right answer combines (1) + (4) + (5): natural language flows through the analyzer
unchanged, and structured features (fields, phrases) get unambiguous, non-punctuation notation. But
this is exactly the open question to resolve.

## Design decision (2026-07-06) — DECIDED with the owner; implement this

Direction chosen: a refinement of option (1) — **the index analyzer becomes the single tokenization
authority for all query text**, field syntax is **removed**, and text enters the language through
**string literals** (plus a verbatim escape hatch). Discussed and agreed with the owner on 2026-07-06.

### Goals (the owner's invariant)

1. **Primary path:** the user/agent can present *arbitrary untokenized text* and is promised it will be
   transformed into the exact token sequence the index-time analyzer would produce for that text —
   then used for basic matching retrieval. No pre-tokenization tool call required; agents write the
   language directly.
2. **Escape hatch:** a way to present an *already-tokenized* vocabulary entry and say "do not process
   this; look it up verbatim" (for tokens whose surface form can't be reliably re-typed, e.g. `u.s.a`).
3. Queries are authored primarily by an **LLM agent**; the notation must be one an agent produces
   correctly by reflex, favoring **local, familiar rules** (C/JSON string escaping) over global or
   bespoke ones (paren balancing inside a text span, novel escape conventions).
4. There is deliberately **no whole-query bare-string mode**: the top level stays structured. A trivial
   wrapper (out of scope here) can turn a raw text string into `#combine("<escaped text>")`.

### New surface grammar

Exactly two text constructs plus the existing operators — **all query text is quoted; there are no
bare terms** (bare chunks were cut on 2026-07-06, see "What is REMOVED"). In operand position only
three things are legal: a `"..."` literal, a `#operator(...)`, and (inside weighted operators) a bare
number in weight position. *Anything else is a loud syntax error* — unquoted text was the grammar's
silent-typo catch-all, and closing it is the point.

| construct | meaning |
|---|---|
| `"..."` (string literal) | **Analyzed-text splice.** Run the index-time analyzer over the contents; splice the resulting **zero or more** term nodes into the enclosing operator. Escapes: `\"` and `\\` **only**; a backslash before anything else is a syntax error (loud, not clever). Parens/`#`/`.`/`:`/`'` inside the literal are ordinary characters. |
| `#token("..." "..." …)` | **Verbatim splice.** Whitespace-separated **quoted** tokens (same two escapes), each looked up in the vocabulary **as-is** — no tokenization, stemming, stopping, or lowercasing. Unquoted contents are a syntax error (whitespace stays the language's separator — no commas). The Indri idiom `#1("u" "s" "a")` becomes `#1(#token("u" "s" "a"))`. A token containing spaces (`#token("token three")`) is legal by contract but can never match — the tokenized index has no space-containing terms — and simply flows through the OOV floored-background path; do not "helpfully" forbid it. |

Key semantic rules:

- **Splice, not node.** Neither `"..."` nor `#token(...)` is a scoring operator; each expands in place
  to 0+ term operands of the *enclosing* operator. `#combine("Mary has a hi-tech lamb")` ≡
  `#combine(mary hi tech lamb)` (whatever the analyzer actually emits); `#1("hi-tech")` is the
  adjacency phrase of the analyzed tokens; `#uw30("...")` works the same way.
- **Empty splice is legal.** `"the of a"` splices zero tokens (all stopped). Same convention as today:
  only a *literally* empty operator (`#combine()`) is a syntax error; an operator emptied by analysis
  matches nothing, like Indri.
- **OOV `#token` uses existing semantics** — the floored background `P(w|C)=1/(2|C|)` (§2 of
  `docs/indri-query-language.md`, locked by `oovTermContributesFlooredBackgroundNotDropped`). Not
  dropped, not an error. `#token` is *only* a tokenization bypass; it introduces no new scoring.
- **`#token` operands produce positions**, so `#token` is valid inside proximity operators.
- **Every operand chunk must start with `"` or `#`.** Anything else — `cat`, `said."`, `dog.title`,
  `~syn(a b)`, a misspelled operator missing its `#` — is a **syntax error** naming the offending
  chunk and suggesting quotes. Trailing garbage after a closing quote (`"a"b`) is likewise an error.
- A query with no leading operator remains an implicit `#combine` of its operands — so `"dog cat"`
  or `"dog" "cat"` alone is a valid whole query, and so is a whole-query `#token(...)`. Bare words
  (`dog cat`) now error. Mixing a literal and an operator at the top level without an enclosing
  `#combine(...)` stays an error, exactly like today's top-level rule.
- Weighted operators: in `#weight(1.0 "dog cat")` the bare number in weight position is a weight —
  numbers are grammar there, not text. A number as *text* is written quoted: `#combine("3.14")`. A
  quoted string in weight position (`#weight("2.0" ...)`) is an invalid-weight error, as today.
- **Weights and multi-token splices (decided 2026-07-06): the weight is copied per token.** In a
  weighted operator, a splice cannot be literal text substitution (`#weight(2.0 "a b")` →
  `#weight(2.0 a b)` would misparse `b` as a weight), so the rule is: **each spliced term node
  inherits the operand's weight** — `#weight(2.0 "hi-tech lamp" 1.0 "cat")` builds children
  `(2.0 hi) (2.0 tech) (2.0 lamp) (1.0 cat)`. Because `#weight` normalizes by the *sum of child
  weights* (`Σwᵢsᵢ/Σwᵢ`), per-token copying changes the mass split: here the literal's tokens
  jointly get 6/7 of the mass. This is exactly what the current parser already does for a
  multi-token chunk (`#weight(2.0 near-death 1.0 cat)` → `(2.0 near)(2.0 death)(1.0 cat)`,
  normalizer 5) — we **document existing `#weight` behavior, we do not invent a new meaning**. A
  user who wants to weight a *concept* (weight applied once to the group) writes the explicit
  wrapper: `#weight( 2.0 #combine("this is my first concept") 1.0 "cat" )` — the `#combine` gets the
  2.0 and internally averages its tokens' log-beliefs. Both forms are expressible; the unwrapped
  literal means per-token. (The `near-death` example describes the OLD bare-chunk grammar — bare
  chunks are now a syntax error — but its per-token weight mechanism is exactly what literals
  inherit. C++ Indri's handling of a multi-token chunk inside `#weight` is unknown/untested —
  "existing behavior" here means Lucindri's mechanism, which conformance work never exercised.)

### What is REMOVED

- **Field restriction syntax, entirely**: `term.field` and `:` are no longer notation. Rationale:
  it is the sole reason the parser pre-splits on punctuation (the root cause above); field-restricted
  *scoring* was never validated against Indri (TASK-0017, deferred); the owner has no real use for it;
  and a field applied to a spliced token *sequence* is semantically ill-defined anyway (Indri fields
  attach to a single term/proximity node). If field restriction returns, it returns as an explicit
  operator on a single term/window node (e.g. `#field[title](...)`), designed and validated then.
  Default field remains `fulltext` throughout.
- **All of `parseQuery`'s pre-rewrites**: `replace("'","")`, `replace("\"","")`, `replace("+"," ")`,
  `replace(":",".")`, and `popTerm`'s `indexOf('.')` split. The parser becomes a proper little lexer
  (string-literal-aware paren scanning) instead of regex mangling. Apostrophes are ordinary
  characters; whatever the analyzer does with `don't` is by definition correct.
- **Bare (unquoted) term chunks, entirely (owner decision 2026-07-06).** Originally the design kept
  `chunk` ≡ `"chunk"` as a convenience. Cut, because the bare chunk is the grammar's **catch-all**:
  any garbage that isn't an operator silently becomes an analyzed-text query — `~syn(a b)` in operand
  position, `dog(cat)`, legacy `dog.title`, a misspelled operator missing its `#` — i.e. **typos turn
  into queries**. With bare chunks gone, all of those are loud syntax errors, and the lexical rule
  collapses to one line: *all text is quoted; everything unquoted is structure (operators, weights)
  or an error.* The same applies **inside `#token(...)`**: contents are whitespace-separated *quoted*
  tokens only (`#token("token1" "token2" "funky)")`) — whitespace stays the language's separator
  (commas considered and rejected: the language separates with whitespace everywhere else), and the
  quoted form absorbs the paren/space edge cases a bare `#token` chunk would mishandle. Cost
  knowingly accepted: **every legacy Indri bare-term query now errors** (`#combine(dog training)` →
  rewrite as `#combine("dog training")`); the `indri-parity` tag marks the divergence and the
  differential-fuzz renderer quotes the Lucindri dialect.
- **The `~` operator prefix, entirely (owner decision 2026-07-06).** Today the parser silently treats
  `~` as a synonym for `#` (`createOperator` does `.replace("~","")`; the operand dispatch tests
  `charAt(0)=='#' || =='~'`), so `~combine(a b)` ≡ `#combine(a b)`. This is **undocumented, unused (no
  test/fuzz/doc references it), and not part of C++ Indri** (no `~` in `indrilang.g`) — a surprising
  wart. It is removed: the **only** operator prefix is `#`. Concretely, drop `~` from **createOperator's
  `.replace("~","")`** and from **every `charAt(0)=='#' || =='~'`** test in the plan snippets (2d paren
  guard, 2d `hasOperator`, 2e operand dispatch) and from **`isTokenSplice`'s `.replace("~","")`**.
  Resulting behavior (simplified by the bare-chunk removal above): **every** `~` use is now a loud
  error — top-level `~combine(a b)` hits the 2d guard (*"parentheses require a #operator"*), and
  operand-position `~syn(a b)` or bare `~foo` hits the unquoted-text error in `popLiteral`. (An
  earlier revision of this bullet had operand-position `~` silently becoming analyzed text; removing
  bare chunks eliminated that silent channel — one of the reasons the owner cut them.) `~` inside a
  `"..."` literal is ordinary text for the analyzer, consistent with the single-tokenizer principle.

### Query-time analysis configuration is RETAINED as-is (decided 2026-07-06, reversing a same-day proposal)

The searcher's query-XML analysis knobs (`<stemmer>`, `<removeStopwords>`, `<ignoreCase>`) **stay and
are honored exactly as today**. A "self-describing index" alternative (indexer persists its analyzer
config in Lucene commit user data; searcher reads it; knobs removed) was proposed and briefly recorded
here, but the owner decided **against changing the indexer**, so it was reverted. Rationale for being
comfortable with user-declared knobs: both the indexer (`ConfigurableAnalyzerFactory`) and the searcher
(`IndriQueryParser.getConfigurableAnalyzer`) hardwire the **same class**,
`org.lemurproject.lucindri.analyzer.EnglishAnalyzerConfigurable` — tokenizer, English stop set, and the
KStem/Porter/none stemmer menu are all fixed by that class, so the entire possible mismatch surface is
those three knobs, matched by convention (documented in the README) as before. The design's promise is
therefore stated *conditionally*: **given the declared analysis configuration, query text flows through
that analyzer with zero parser interference** — matching the index remains the user's declared-knobs
responsibility, unchanged. A spun-off `#stop(...)` task (briefly TASK-0019) was deleted along with the
reversal — with `removeStopwords` retained as a query-time knob, no capability is lost that it would
have restored.

### Known ergonomic risk (accepted, mitigate by documentation)

Agents habituated to web-search may write `"this is a phrase"` expecting **phrase** semantics; here it
is a bag-of-words splice. The failure mode is graceful — a sensible relaxed retrieval, exactly what web
engines do when they drop quotes — never a silent zero-result. Mitigation: the query-language guide
(and any agent prompt) must carry the one-line cheat sheet: **all text is quoted; phrases are
`#1("...")`; quotes alone mean analyzed text; `#token("...")` means verbatim lookup.**

**2b. String-literal helpers** (private static; `syntaxError` already throws `QueryParseException`):

```java
/** Given s.charAt(start)=='"', returns the index of the literal's closing quote.
 *  Escapes: only \" and \\ are legal. Errors: unterminated literal, unknown escape. */
private static int endOfLiteral(String s, int start) {
    for (int i = start + 1; i < s.length(); i++) {
        char c = s.charAt(i);
        if (c == '\\') {
            if (i + 1 >= s.length()
                    || (s.charAt(i + 1) != '"' && s.charAt(i + 1) != '\\')) {
                syntaxError("unknown escape in string literal (only \\\" and \\\\ are supported)");
            }
            i++; // skip the escaped character
        } else if (c == '"') {
            return i;
        }
    }
    syntaxError("unterminated string literal");
    throw new AssertionError("unreachable"); // syntaxError always throws
}

/** Unescape the CONTENTS of a literal (the text between the quotes). Assumes endOfLiteral validated it. */
private static String unescapeLiteral(String contents) {
    StringBuilder sb = new StringBuilder(contents.length());
    for (int i = 0; i < contents.length(); i++) {
        char c = contents.charAt(i);
        if (c == '\\') {
            i++;
            c = contents.charAt(i);
        }
        sb.append(c);
    }
    return sb.toString();
}
```

**2c. Literal-aware scanning.** Replace `countChars` (delete it) and rewrite

```java
/** Counts '(' and ')' OUTSIDE string literals; validates every literal en route. */
private static int[] countParens(String s) {
    int open = 0, close = 0;
    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        if (c == '"') {
            i = endOfLiteral(s, i);
        } else if (c == '(') {
            open++;
        } else if (c == ')') {
            close++;
        }
    }
    return new int[] { open, close };
}

/** Index of the ')' balancing the left-most '(' outside literals; -1 if none. */
private static int indexOfBalancingParen(String s) {
    int depth = 0;
    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        if (c == '"') {
            i = endOfLiteral(s, i);
        } else if (c == '(') {
            depth++;
        } else if (c == ')') {
            depth--;
            if (depth == 0) {
                return i;
            }
        }
    }
    return -1;
}

/** Index of the first '(' outside literals; -1 if none. */
private static int indexOfFirstParen(String s) {
    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        if (c == '"') {
            i = endOfLiteral(s, i);
        } else if (c == '(') {
            return i;
        }
    }
    return -1;
}
```

```java
/** Pops one "..." literal and splices its analyzed tokens into queryTree. ALL query text is
 *  quoted: an unquoted chunk here is a syntax error (bare terms were removed — they were the
 *  catch-all that turned typos into queries). */
private String popLiteral(String argString, QueryParserOperatorQuery queryTree, Float weight,
        Occur occur) {
    if (argString.charAt(0) != '"') {
        String chunk = argString.split("[ \t\n\r]+", 2)[0];
        syntaxError("unquoted text '" + chunk
                + "' — query text must be quoted, e.g. #combine(\"dog cat\")");
    }
    int end = endOfLiteral(argString, 0);
    if (end + 1 < argString.length() && !Character.isWhitespace(argString.charAt(end + 1))) {
        syntaxError("unexpected character after closing quote (a literal must be its own chunk)");
    }
    String rawText = unescapeLiteral(argString.substring(1, end));
    String remainder = argString.substring(end + 1);

    // The splice: the WHOLE literal goes to the index analyzer; 0..n term nodes come out.
    for (String t : tokenizeString(analyzer, rawText)) {
        QueryParserTermQuery termQuery = new QueryParserTermQuery();
        termQuery.setTerm(t);
        termQuery.setField(defaultField);
        termQuery.setOccur(occur);
        queryTree.addSubquery(termQuery, weight);
    }
    return remainder.trim();
}
```

today's `#combine(the a)`, which is now written `#combine("the a")`).

**2g. Add `popTokenSplice`** — whitespace-separated **quoted** tokens only; verbatim, no analysis of
any kind (no lowercasing!):

```java
/** #token("a" "b" ...): whitespace-separated QUOTED tokens, each a vocabulary entry used AS-IS —
 *  no tokenization, stemming, stopping, or lowercasing. Unquoted contents are a syntax error
 *  (same all-text-is-quoted rule as everywhere else). OOV entries flow through the existing
 *  floored-background path (IndriMissingTermScorer), like any other absent term. */
private String popTokenSplice(String argString, QueryParserOperatorQuery queryTree, Float weight,
        Occur occur) {
    int close = indexOfBalancingParen(argString);
    if (close < 0) {
        syntaxError("unbalanced parentheses in #token");
    }
    String contents = argString.substring(argString.indexOf('(') + 1, close).trim();
    if (contents.isEmpty()) {
        syntaxError("#token has no operands");
    }
    while (!contents.isEmpty()) {
        if (contents.charAt(0) != '"') {
            String chunk = contents.split("[ \t\n\r]+", 2)[0];
            syntaxError("unquoted token '" + chunk
                    + "' in #token — tokens must be quoted, e.g. #token(\"u.s.a\")");
        }
        int end = endOfLiteral(contents, 0);
        if (end + 1 < contents.length() && !Character.isWhitespace(contents.charAt(end + 1))) {
            syntaxError("unexpected character after closing quote in #token");
        }
        String term = unescapeLiteral(contents.substring(1, end));
        contents = contents.substring(end + 1).trim();
        if (term.isEmpty()) {
            // #token("") — a verbatim EMPTY token can never exist in the vocabulary; unlike an
            // analyzed empty splice (legal), this is necessarily a mistake. Loud.
            syntaxError("#token: empty term");
        }
        QueryParserTermQuery termQuery = new QueryParserTermQuery();
        termQuery.setTerm(term);
        termQuery.setField(defaultField);
        termQuery.setOccur(occur);
        queryTree.addSubquery(termQuery, weight);
    }
    return argString.substring(close + 1);
}
```

(Quoting makes parens/whitespace inside a token unambiguous — `#token("funky)")` is fine, the
balancing-paren scan skips literal contents. A space-containing token is legal-but-unmatchable; see
the grammar table.)

#### Pitfalls (read before coding)

- **Never lowercase or analyze `#token` contents** — verbatim means byte-for-byte.
- The old `lastIndexOf(")")` operator-strip is replaced by positional stripping (2d) *because* a
  trailing literal may contain `)`; do not reintroduce it.
- `queryString.split("[(]", 2)` anywhere is a bug now (a `(` may be inside a literal) — all paren
  logic must go through the 2c helpers.
- **Do not re-add a bare-chunk convenience path.** Unquoted text erroring is the design, not an
  oversight — the bare chunk was the catch-all that turned typos (`~syn(a b)`, `dog(cat)`,
  `dog.title`, an operator missing its `#`) into silent queries. If an unquoted chunk "would
  obviously be fine," it still errors; the user quotes it.
- `dog(cat)` errors twice over now — top-level via the 2d paren guard, operand-position via the
  unquoted-text error. Both must stay errors.
- Top-level `#token("a") "cat"` errors ("operator must enclose its entire (sub)query") — same
  top-level rule as `#1("a" "b") "cat"` today; the composed form is `#combine(#token("a") "cat")`.
- `QueryParserTermQuery.setField(defaultField)` stays (field is always the default field now; the
  domain class and `createBooleanClause` are unchanged).
- Keep `IndriQueryParserAnalysisTest` passing untouched — it pins the retained analysis knobs.
- **Known, deliberate redundancy — do not "optimize":** `countParens` (and the other 2c scanners)
  re-validate literals at every recursion level, so a deeply nested literal is re-scanned once per
  level. This is idempotent and bounded (`MAX_PARSE_DEPTH` = 128; queries are small) and is the
  point: each helper is safe on ANY input, with no "caller already validated this" contract between
  methods. Hoisting validation to one up-front pass would make every downstream helper assume
  pre-validated input — the exact kind of implicit contract that produced the original parser's
  out-of-bounds crashes (TASK-0015). Noted in plan review 2026-07-06; accepted.

## Reference: how C++ Indri actually handles quotes, stemming, stopwords, and acronyms (read from source, 2026-07-05)

Any redesign of Lucindri's query-text handling should be informed by exactly what Indri does. Read
directly from the Indri 5.21 source and confirmed empirically:

**Quotes = a single, unprocessed term (NOT a phrase, NOT just "unstemmed").**
- Grammar `src/indrilang.g` (~line 967): `DBL_QUOTE t=rawText DBL_QUOTE { t->setStemmed(true); }`, and
  `rawText` returns a **single** `IndexTerm`. So `"x"` wraps exactly **one** term and flags it "stemmed."
- `src/InferenceNetworkBuilder.cpp` (`after(IndexTerm*)`): `_repository.processTerm()` — which does
  **stemming AND stopword removal together** — is called **only when `getStemmed()==false`**. A quoted
  term skips it entirely. **So a quoted term is neither stemmed nor stopped** (nor normalized). The
  common "quotes turn off stemming" description undersells it: quotes turn off *all* query-side term
  processing.
- Stopwords are implemented as terms that `processTerm` maps to the empty string
  (`stopword = processed.length() == 0`); a quoted term never reaches that check, so it is never dropped.
- Multi-word `"white house"` is a **parse error** in Indri (verified) — quotes are not phrases. Use
  `#1(white house)` for a phrase. `"white"` (single term) is fine.
- Consequence / idiom: to feed already-tokenized text back to Indri without re-processing, quote each
  token, e.g. `#1("u" "s" "a")` keeps `a` from being stopped so the ordered window stays intact.
  **Caveat:** quoting only prevents *query-side* processing; the literal term must still exist **in the
  index** to match (if the index was built with a stopper, `a` isn't there, and an OOV term makes the
  whole `#1` window fail). And `"u.s.a."`/`"U.S.A"` are literal, unnormalized lookups that won't match
  the index's normalized `usa` token.

**Acronym / period tokenization (`src/TextTokenizer.l`, the flex lexer).** The rule ordering is:
```
[A-Z0-9]"."([A-Z0-9]".")*        -> one token (acronym; dots stripped)   e.g. U.S.A. -> usa
[a-zA-Z0-9']+                    -> normal word (apostrophe kept, then stripped: don't -> dont)
"-"[0-9]+("."[0-9]+)?            -> negative number
.                               -> ZAP (discarded; a bare '.' is dropped)
```
- The acronym rule is **uppercase-only** (`[A-Z0-9]`) and needs a period after **each** element,
  including the last: `U.S.A.` -> `usa`, `I.B.M.` -> `ibm`, `U.N.` -> `un`. Lowercase `u.s.a.` fails the
  rule -> `u`,`s`,`a`. `e.g.`, `a.b.c.`, `Ph.D.`, `co.uk` don't collapse; `3.14` -> `3`,`14`.
- So "the user can't predict the tokenization" is real and rooted here: case and trailing punctuation
  change the result, and this is the *same* tokenizer Indri uses at index and query time (so Indri stays
  self-consistent — the divergence is with *Lucindri's* pre-analyzer splitting on `.`/`:`).

**Lucindri gap.** `IndriQueryParser.parseQuery` does `queryString.replace("\"", "")` — it **deletes all
quotes** before parsing, so Lucindri has no unstemmed/unstopped-term operator at all and silently treats
`"a"` as `a`. A redesign should decide whether to implement the quote-as-literal-term operator, reject
quotes, or repurpose quotes for phrases (they are *not* phrases in Indri) — and must keep query-time
tokenization consistent with index-time (§ root cause).
