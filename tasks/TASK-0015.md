# TASK-0015 — Negative/robustness fuzzer: Lucindri must reject invalid queries cleanly (never crash, never silently misparse)

- **Status:** In progress
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** We have a *legal*-query differential fuzzer (TASK-0011). We do **not** have a
  *negative* fuzzer that feeds **invalid** queries and checks that Lucindri rejects them gracefully.
  Preliminary probing shows the parser currently **leaks low-level crashes** (`ArrayIndexOutOfBounds`,
  `StringIndexOutOfBounds`, `NumberFormatException`) on malformed input instead of a clean, documented
  error. Build a grammar-aware invalid-query fuzzer, define and enforce a robustness invariant, and fix
  the parser so every bad query is rejected with a clear, specific `QueryParseException` (a new subtype
  of `IllegalArgumentException`) — never an implementation-leak crash, hang, or silent misparse — and so
  `IndriSearch` reports the bad query and continues the batch.

## The robustness invariant (the oracle)

For **any** input string `s`, `new IndriQueryParser().parseQuery(s)` (and running the result on a small
index) must satisfy:

1. **No implementation-leak throwable ever escapes.** The parser must never surface a low-level
   exception that exposes internals — no `NullPointerException`, `ArrayIndexOutOfBoundsException`,
   `StringIndexOutOfBoundsException`, `NumberFormatException`, `StackOverflowError`, `OutOfMemoryError`.
   Those are the true defects: they give the user a useless message and signal the parser failed to
   validate its input.
2. **A malformed query throws a single, documented, catchable type.** Rejection is a **new
   `QueryParseException` that extends `IllegalArgumentException`** — so it stays *unchecked* (no
   signature churn) and every existing `assertThrows(IllegalArgumentException…)` and the `syntaxError`
   convention keep working, while callers gain a *specific* type they can catch to mean exactly "invalid
   query" without swallowing a genuine internal bug. It carries a clear, actionable message (ideally
   naming the offending substring/position). Raw `IllegalArgumentException` is too generic to be the
   contract — it is thrown by unrelated library code too; `QueryParseException` is the precise signal.
3. **It terminates** — no infinite loop or pathological blow-up (guard with a timeout and a nesting/size
   cap).
4. **No silent misparse of a structurally invalid query** — a query that violates the grammar
   (unbalanced parens, a belief operator used as a proximity operand, a weighted operator with a
   dangling weight, an unknown operator) must be rejected, not quietly reinterpreted as something else.
5. **`IndriSearch` catches it per query and continues.** A single malformed query in a batch must be
   reported (its number + message) and skipped, not abort the whole run. Today `IndriSearch` calls
   `parseQuery` (line ~124) without catching the parse error, so even a clean rejection currently aborts
   the batch — that must be fixed as part of "rejecting nicely."

This invariant is the fuzzer's checkable property; it needs no second engine. (Optionally, cross-check
against C++ Indri: where Indri rejects and Lucindri accepts, or vice versa, flag for review.)

## Evidence — confirmed parser crashes (probed 2026-07-05 on `lucindri_idx`)

| invalid query | current behavior | should be |
|---|---|---|
| `#uw( alpha beta )` (window, no N) | **ArrayIndexOutOfBoundsException** | `QueryParseException` "missing window distance" |
| `#combine( alpha beta` (missing `)`) | **StringIndexOutOfBoundsException** | `QueryParseException` "unbalanced parentheses" |
| `#combine alpha beta )` (missing `(`) | **StringIndexOutOfBoundsException** | `QueryParseException` "unbalanced/misplaced parentheses" |
| `#weight( abc alpha )` (non-numeric weight) | **NumberFormatException** | `QueryParseException` "invalid weight 'abc'" |
| `#9999999999( alpha beta )` (distance overflows int) | **NumberFormatException** | `QueryParseException` "invalid window distance" |
| `#combine( )` (empty operator) | no error, 0 results | decide: reject (`QueryParseException`), or define as empty (documented) |

(`QueryParseException extends IllegalArgumentException`, so these remain catchable as
`IllegalArgumentException` for backward compatibility.)

Cleanly handled already (regression targets to keep green): belief operator inside a proximity operator
(`#1(#combine(peace talks) summit)`) and unknown/unimplemented operators (`#foo(...)`, `#wsyn(...)`, per
TASK-0014) are rejected with a clear message today (raw `IllegalArgumentException` via `syntaxError`);
after this task they throw `QueryParseException` (still an `IllegalArgumentException`), so the existing
`assertThrows(IllegalArgumentException.class, …)` tests stay green.

## Known parser weak points (grounded in `IndriQueryParser.java`)

- **The parenthesis-balance check is commented out.** In `parseQueryString`, the code computes
  "missing/unbalanced/misplaced parentheses" but the `throw` is commented out, so malformed-paren
  queries fall through to `substring(...)`/`lastIndexOf(")")` and crash with
  `StringIndexOutOfBoundsException`. Re-enable it as an `IllegalArgumentException`.
- **`createOperator` proximity parsing is unguarded:** `#uw`/`#near`/`#window` with no number →
  `split(...)[1]` `ArrayIndexOutOfBounds`; `Integer.parseInt(distance)` → `NumberFormatException` on a
  huge/negative/non-numeric N. Validate the distance (positive int in range) and error clearly.
- **`popWeight` uses `Float.valueOf(...)`** on the weight token → `NumberFormatException` for a
  non-numeric weight, and a weighted operator (`#weight`/`#wand`/`#wsum`) with an odd token count leaves
  a dangling weight. Validate.
- **Deep nesting** (`#combine(#combine(#combine(…)))`) risks `StackOverflowError` in the recursive
  descent — cap depth and reject beyond it.

## Implementation plan (detailed, ordered)

Design principle: **validate at the source, don't mask.** The parser must reject bad input by *checking*
for it and throwing `QueryParseException` with a specific message — not by wrapping everything in a
blanket `try/catch` that translates crashes (that would hide real logic bugs). The only blanket catch is
the operational safety net in `IndriSearch` (per-query), so one bad query never aborts a batch. The
negative fuzzer's job is to surface every leak so we fix it at its site.

### Phase 0 — harness first (so we can measure before/after)
0.1 Write the **enforcement harness** before touching the parser, so it red-flags today's crashes.
   Location: a small `main` under `scripts/fuzz/` compiled against the searcher fat jar (mirrors the
   TASK-0013 `LucindriKStem.java` pattern), **and** the reusable core as a test helper so the JUnit test
   can call it. It reads one query per line and for each classifies the outcome:
   - `OK` — `parseQuery` returned and a bounded search on a tiny in-memory index ran.
   - `REJECTED` — threw `QueryParseException` (the desired rejection).
   - `VIOLATION` — any other `Throwable` (record class + message + input), a **timeout**, or a
     **silent-accept** heuristic (see 0.2).
   Run each parse+search on a worker with a per-query **timeout** (e.g. 5 s via a `Future`/`ExecutorService`)
   so an infinite loop / pathological blow-up is caught as a VIOLATION rather than hanging the harness.
   Use a fixed tiny index (a handful of docs of known lowercase non-stopword tokens) so search actually
   exercises scorer-stage crashes, not just the parser.
0.2 Silent-accept detection is heuristic (there is no full grammar oracle). Flag as *suspicious* the
   cases we can define: a query with unbalanced parens that still returns a `Query`; a `#weight`/`#wsum`
   with an odd weight/operand token count that parses; an operator token containing characters outside
   `[a-z0-9]` that parses. These are reviewed, not hard failures, unless they clearly misparse.

### Phase 1 — the exception type
1.1 New file `LucindriSearcher/src/main/java/org/lemurproject/lucindri/searcher/parser/QueryParseException.java`
   (no CMU/Lemur header — author's file, start at `package`): `public class QueryParseException extends
   IllegalArgumentException` with `(String message)` and `(String message, String query)` constructors
   (the latter stores the offending query for context in the message).
1.2 Change `syntaxError(String)` to `throw new QueryParseException("Syntax Error: " + msg)`. Every
   existing rejection (belief-in-proximity, unknown operators from TASK-0014, missing-distance) upgrades
   for free and stays an `IllegalArgumentException` (existing `assertThrows` tests keep passing).

### Phase 2 — parser hardening (`IndriQueryParser`), each at its site
2.1 **Parentheses.** The balance check in `parseQueryString` is present but its `throw` is commented out.
   Re-enable it as `syntaxError(...)`, **but fix the predicate first**: the current condition rejects
   `countChars('(')==0`, which would wrongly reject a legal **bare-term** query (`dog training`, which
   has no parens and defaults to `#combine`). Correct rule: require `count('(') == count(')')` (balanced)
   always; and if there is ≥1 `(`, require the first `(` opens the top operator and the balancing paren
   is the last non-space char. Zero parens (bare terms) is legal. Add a regression test that
   `dog training` still parses.
2.2 **Proximity distance.** In `createOperator`, replace the fragile normalization with validated regex
   matches: `#N`=`\d+`, `#odN`=`od(\d+)`, `#uwN`=`uw(\d+)`. If a proximity spelling has **no** number
   (`#uw`, `#od`) → `syntaxError("#uwN/#odN requires a window size")` (today `"uw".split("uw")[1]`
   throws `ArrayIndexOutOfBounds`). Parse the distance with a bounded check (e.g. `matches("\\d{1,7}")`
   then `Integer.parseInt`), and reject distance `0`, negative, non-numeric, or over a sane cap →
   `syntaxError("invalid window distance …")` (today `Integer.parseInt` on `9999999999` throws
   `NumberFormatException`). Keep the existing `#nearN`/`#windowN`-error behavior.
2.3 **Weights.** In `popWeight`, wrap `Float.valueOf(token)` so a non-numeric weight →
   `syntaxError("invalid weight '…'")` instead of `NumberFormatException`; reject NaN/∞/negative if we
   decide those are illegal (record the decision). In `parseQueryString`, for `#weight`/`#wand`/`#wsum`
   ensure the operands come in matched `weight operand` pairs — a dangling trailing weight or a
   weight with no following operand → `syntaxError`.
2.4 **Empty operators.** Decide and implement: a *literal* empty operator (`#combine()`, `#uw2()`,
   `#syn()`) → `syntaxError("operator #… has no operands")`. Distinguish from an operator that becomes
   empty only **after** stopword removal (`#combine(the a)`), which is legal input and should behave like
   Indri (empty result, not an error). Encode the check on the *parsed operand count from the raw text*,
   not post-analysis, so the two cases are separable.
2.5 **Recursion depth.** Thread a depth counter through `parseQueryString`/`popSubquery` (or an instance
   field reset in `parseQuery`); beyond a cap (e.g. 128) → `syntaxError("query nesting too deep")` so
   deep nesting can't `StackOverflowError`. Also cap total operand count / query length as a cheap DoS
   guard.
2.6 After each fix, re-run the harness on the growing invalid corpus; every VIOLATION must become
   `REJECTED`. **No blanket `try/catch` in `parseQuery`** — fix causes, don't mask.

### Phase 3 — per-query recovery in `IndriSearch`
3.1 In the query loop (`IndriSearch.java` ~line 123–142) wrap each query's `parseQuery(...)` + `search`
   in `try { … } catch (QueryParseException e) { System.err.println("skipping query " +
   query.getNumber() + ": " + e.getMessage()); continue; }`. Optionally also catch a broader
   `RuntimeException` defensively (log as an internal error) so a single query can never abort the batch,
   while still surfacing it. Keep stdout TREC output clean (errors to stderr).

### Phase 4 — the invalid-query generator (`scripts/fuzz/fuzz_invalid.py`)
Seeded, reproducible; emit `label<TAB>query` per line (label = intended violation category, for triage).
4.1 **Grammar-aware invalids** (one broken rule each): unbalanced/misplaced/extra parens; empty operator
   (`#uw2()`, `#syn()`, `#combine()`); missing window size (`#uw()`, `#od()`); bad distance (`#0(a b)`,
   `#99999999999(a b)`, `#-1(a b)`); malformed weights (`#weight(a b)`, `#weight(0.5 a b)`,
   `#weight(abc a)`); belief op as a proximity operand (`#1(#or(a b) c)`); unknown ops/typos (`#combyne`,
   `#wsyn`, `#prior`); field-dot edge cases (`a.`, `.title`, `a.b.c`); empty/whitespace text.
4.2 **Mutation of legal queries:** read the TASK-0011 legal fuzzer's output and corrupt each (delete/
   insert/swap a char; drop a random paren; blank a weight; splice a belief op into a proximity slot).
   This finds crashes the structured generator misses.
4.3 **Adversarial/degenerate:** very deep nesting; thousands of operands; huge tokens; control/Unicode
   chars; only-punctuation. (These target 2.5 and any unicode/tokenizer edge.)

### Phase 5 — lock it down
5.1 New JUnit `InvalidQueryRejectionTest` (`LucindriSearcher`, reuses `TestIndex`): a curated seeded
   corpus — at least one case per evidence row and per category — asserting the invariant via the Phase-0
   helper: each throws `QueryParseException` (an `IllegalArgumentException`) and **no** other throwable
   escapes. Include the fail-before/pass-after crash rows and the already-clean cases
   (belief-in-proximity, unknown operator) as kept-green regressions. Add the `dog training` bare-term
   parse and a mixed valid+invalid batch through `IndriSearch` (Phase 3) asserting the batch completes.
5.2 Run the large random generator (thousands of inputs) through the harness → **zero VIOLATIONs**.
5.3 Re-run the **TASK-0011 legal fuzz** (5k queries) to prove no legal query regressed
   (`diff_fuzz.py` unchanged vs Indri).
5.4 Commit script + harness; record the empty-operator / weight-sign decisions in this task and (if they
   affect user-visible behavior) in `README.md` / `docs/indri-query-language.md`.

### Suggested ordering / sizing
Phase 0 → 1 → 3 (cheap safety net) can land first as a small PR; then 2 (the real hardening) fix-by-fix
against the harness; then 4–5. Each parser fix in Phase 2 is independently testable (fail-before via the
evidence row, pass-after). Estimate: exception type + IndriSearch catch ~small; parser hardening the bulk
(five discrete fixes); fuzzer + tests moderate.

## Acceptance criteria

- A committed, reproducible negative fuzzer + harness under `scripts/`.
- Running it over a large seeded corpus produces **zero invariant violations** after the fixes: no
  implementation-leak throwable escapes, no hang, no silent misparse; every rejection is a
  `QueryParseException`.
- A new `QueryParseException extends IllegalArgumentException`; `syntaxError` throws it.
- The confirmed crash rows above all become clean `QueryParseException`s with clear messages.
- `IndriSearch` catches a bad query, reports it (number + message), and continues the batch — verified
  by a test with a mix of valid and invalid queries in one parameter file.
- A green JUnit `InvalidQueryRejectionTest` that fails on today's parser (the crashes above) and passes
  after; the legal-query behavior and the full searcher suite remain green.
- Decision recorded for the genuinely ambiguous cases (e.g. `#combine()` empty operator): reject, or
  define-and-document.

## Notes / scope

- Scope is the searcher query parser + its scorer entry points (`IndriQueryParser`, and any crash that
  only appears at search time). Indexer/analyzer untouched.
- **Why `QueryParseException extends IllegalArgumentException` (unchecked) and not a checked exception:**
  the essential property is (a) no implementation-leak crash and (b) one specific catchable type meaning
  "invalid query." A dedicated *unchecked* subtype delivers both while staying backward-compatible with
  this codebase's existing convention (`syntaxError` already throws `IllegalArgumentException`;
  `parseQuery` declares no checked exception; the existing tests assert `IllegalArgumentException`). A
  *checked* `ParseException` (as Lucene's own `QueryParser` uses) would force callers to handle bad
  input and is arguably more "correct" for a library, but converting this unchecked-throughout code is a
  larger, against-the-grain refactor; revisit only as a separate decision.
- Rationale (owner, 2026-07-05): a search engine must not crash or silently misinterpret a user's
  malformed query — it should reject it with a clear message. This complements TASK-0014 (which stopped
  *unknown operators* from silently degrading); this task hardens the parser against *all* malformed
  input.
- Related: TASK-0014 (reject unknown operators), TASK-0011 (legal-query differential fuzzer whose corpus
  feeds the mutation strategy).
