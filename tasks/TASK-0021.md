# TASK-0021 — Parent/reactor POM + bump all modules to 2.0

- **Status:** In progress (implemented + verified on branch `task-0019-http-service`, 2026-07-07; optional plugin-version centralization deferred)
- **Created:** 2026-07-07
- **Owner:** (unassigned)
- **Summary:** Add a top-level **Maven reactor** (a parent/aggregator `pom.xml` at the repo root) so a
  single `mvn install` builds **all** modules in the correct dependency order (the "just build it"
  workflow), and **bump every module to a unified `2.0`** to mark the line past Lucindri 1.5 (query-language
  extensions beyond Indri parity, the TASK-0020 index-format change, and the forthcoming HTTP service).
  Today the three modules are standalone Maven projects with **no parent POM** and mismatched versions
  (`analyzer` 1.5, `LucindriSearcher` 1.5, `LucindriIndexer` **1.45**), built one-by-one in a hand-documented
  order. This task makes the module dependency graph explicit in the build and unifies the version.

## Context / why

The build order is currently tribal knowledge (`analyzer` → `searcher` → `indexer`, each
`mvn clean install`) because nothing captures the inter-module dependencies for Maven to act on. Maven's
**reactor** is exactly the "one command, correct order" mechanism: a root `pom.xml` with
`<packaging>pom</packaging>` and a `<modules>` list; `mvn install` there computes the order from the
modules' dependencies and builds them together (you can still build one with `mvn -pl <module> -am`). This
task adds that, and while we're re-touching every POM's version, unifies them at **2.0** (a reactor also
naturally wants a single inherited version, so the two changes are really one).

> Honest scope note: the reactor gives correct **ordering** and a one-command build; it is **not** a
> make-grade incremental engine (a plain reactor build rebuilds all modules — finer change-detection needs
> flags/add-ons). Ordering + one-command is the win here.

CLAUDE.md already flags this as deliberately deferred ("A parent/reactor POM to centralize the above is a
deferred option … revisit only as a separate decision"). This task is that decision.

## Design decisions (flag any you'd change)

1. **Idiomatic reactor with version inheritance.** The parent declares `<version>2.0</version>`; each child
   declares `<parent>` (groupId `org.lemurproject.lucindri`, artifactId `lucindri-parent`, version `2.0`,
   `<relativePath>../pom.xml</relativePath>`) and **omits its own `<groupId>`/`<version>`** (inherited). Next
   bump = one edit. *Alternative (rejected):* keep explicit `<version>2.0</version>` in every child — more
   places to drift.
2. **Inter-module deps use `${project.version}`.** `searcher`→`analyzer` and `indexer`→`analyzer` reference
   `<version>${project.version}</version>` (all reactor modules share the version), so they never need
   manual bumping again.
3. **Unified version `2.0`.** All three modules → `2.0` (indexer jumps from the odd `1.45`). Advertised jar
   names become `Lucindri{Analyzer,Searcher,Indexer}-2.0[-jar-with-dependencies].jar`.
4. **Keep artifactIds as they are** (`analyzer`, `LucindriSearcher`, `LucindriIndexer`). Renaming is a
   nice-to-have that changes dependency coordinates everywhere; deferred (Risks).
5. **Optional (recommended if clean): centralize pinned plugin/test versions in the parent.** The modules
   pin identical `maven-compiler-plugin` 3.8.1, `maven-surefire-plugin` 2.22.2, `junit-jupiter` 5.8.2, and
   Java 11 — hoisting these to the parent's `<properties>`/`<pluginManagement>`/`<dependencyManagement>` is
   the "capture it in the build infra" spirit and kills drift. Do it **only if** each module still resolves
   and tests stay green; otherwise leave module POMs as-is and just add the parent aggregation. (Lucene
   8.10.0 version management across modules is explicitly optional — don't force it.)

## Implementation plan

1. **Add the root parent POM** `pom.xml` (repo root): `org.lemurproject.lucindri:lucindri-parent:2.0`,
   `<packaging>pom</packaging>`, `<modules>` listing the three module directories. (If doing Decision 5,
   add the shared `<properties>`/`<pluginManagement>`/`<dependencyManagement>` here.)
2. **Each module POM:** add `<parent>` (→ `lucindri-parent:2.0`, `relativePath ../pom.xml`); remove its own
   `<groupId>` and `<version>` (inherit `2.0`). Change each `analyzer` dependency's `<version>` to
   `${project.version}`.
3. **Verify the reactor:** from the repo root, `mvn -o clean install` builds all three **in order**
   (analyzer → searcher → indexer) with **all module tests green**, and produces
   `Lucindri{Analyzer,Searcher,Indexer}-2.0` artifacts + the two `-jar-with-dependencies` fat jars named
   `-2.0`. Also confirm a single-module build still works: `mvn -pl LucindriSearcher -am -o install`.
4. **Smoke the fat jars run at 2.0:** `scripts/getdoc-smoke.sh` already globs the jar names
   (`LucindriIndexer-*-jar-with-dependencies.jar`), so it is version-agnostic — run it to confirm the 2.0
   jars still index + `getdoc` end-to-end. (Guards against a botched assembly/mainClass after the bump.)
5. **README:** replace the per-module "build in this order" instructions with "run `mvn clean install` at
   the repo root (the reactor builds all modules in order); to build one, `mvn -pl <module> -am install`";
   and update the run-command jar filenames to `-2.0` (`LucindriIndexer-2.0-…`, `LucindriSearcher-2.0-…`,
   incl. the `getdoc` example). Leave the historical "post-1.5" prose alone.

## Risks / notes

- **artifactId inconsistency** (`analyzer` vs `LucindriSearcher`/`LucindriIndexer`) persists. Renaming
  `analyzer` → `LucindriAnalyzer` would be tidy and is low-risk (it's a local-only artifact referenced only
  by the two dependents), but it changes coordinates in three POMs and any external reference; **deferred**
  unless the owner wants it folded in.
- **trec-car-tools / jitpack** (indexer) and all Lucene/Solr/Gson deps are untouched by a version bump.
- **Downstream jar-name references** live in the README (updated here) and `scripts/getdoc-smoke.sh` (already
  glob-based, so safe). Grep the repo for `-1.5-` / `-1.45-` before finishing to catch any stragglers.
- **Offline build:** use `mvn -o` (the deps are already cached from prior builds); the reactor doesn't add
  new dependencies.
