# TASK-0004 — Establish a shared test framework across the Lucindri modules

- **Status:** Ready
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** Stand up one consistent JUnit 5 test setup across all Lucindri modules, plus a
  shared searcher test-support helper (controlled tiny-index builder + query runner), so
  individual tasks write **tests**, not test-framework plumbing. Foundation for [[TASK-0002]] and
  [[TASK-0003]].

> Read `../CLAUDE.md` first. This task touches module `pom.xml`s, adds a small shared test-support
> package, and documents the testing standard in `CLAUDE.md`.

---

## 1. Motivation

TASK-0001 added JUnit 5 + `maven-surefire-plugin` to `LucindriIndexer` and wrote the first tests.
TASK-0002 and TASK-0003 both target `LucindriSearcher` and each independently specified "add
JUnit + surefire to the searcher pom" **and** "build a tiny in-memory Lucene index in the test."
That is duplicated plumbing and duplicated fixtures. Establish it once.

## 2. Structural decision (do the lightweight option unless told otherwise)

The three modules currently build **independently** (`mvn clean install` each, in order
Analyzer → Searcher → Indexer) and carry **different versions** (analyzer 1.5, searcher 1.5,
indexer 1.45). A Maven **parent/reactor POM** is the textbook way to centralize test deps
(`dependencyManagement` + `pluginManagement`), but it changes the build model and is awkward with
per-module versions.

- **Recommended (this task):** lightweight standardization — the **same** JUnit + surefire
  coordinates and config in each module `pom.xml`, pinned to shared constants, documented as the
  project standard. Minimal disruption; matches the pattern TASK-0001 already set.
- **Deferred option:** introduce a parent POM later if we decide to unify the build. Note it in
  `CLAUDE.md` as a possible future consolidation; do **not** do it here without a separate decision.

## 3. Deliverables

### 3a. Standardize the test wiring in every module
- Pin the versions once, as a project standard: **`org.junit.jupiter:junit-jupiter:5.8.2`**
  (test scope) and **`maven-surefire-plugin:2.22.2`**.
- `LucindriIndexer/pom.xml` — already has these (TASK-0001); just confirm they match the pinned
  versions.
- `LucindriSearcher/pom.xml` — add them.
- `LucindriAnalyzer/pom.xml` — add them (the analyzer is small but is the tokenization contract;
  a couple of tests there are cheap insurance and keep the setup uniform).
- Confirm `src/test/java/...` is the test root in each and that `mvn -q test` runs Jupiter tests.

### 3b. Shared searcher test-support helper (the reused fixture)
Add a small, dependency-free test-support package in the searcher module, e.g.
`LucindriSearcher/src/test/java/org/lemurproject/lucindri/searcher/testutil/`, used by BOTH
TASK-0002 and TASK-0003 so neither rebuilds it:

- **`TestIndex`** — build a controlled Lucene index in a JUnit `@TempDir` that faithfully mirrors
  production index-time config: `LMDirichletSimilarity` (so norms encode as in production),
  `IndexOptions.DOCS_AND_FREQS_AND_POSITIONS`, stored fields, field names `fulltext` /
  `externalId`. API roughly:
  - `TestIndex.builder(Path dir).analyzer(Analyzer).add(externalId, fulltextText)... build()`
    → returns something exposing a `DirectoryReader` / `IndriIndexSearcher`.
  - Default analyzer = a **whitespace, no-stemming, no-stopword** analyzer so token counts and
    term frequencies are exactly known (needed for TASK-0003's closed-form scoring and
    known-corpus assertions); allow overriding with the project analyzer.
  - A variant that builds **N sub-indexes** and wraps them in a `MultiReader`, to mirror the
    production 8-part layout (used by TASK-0002 and TASK-0003 multi-segment tests).
- **`Queries`** (or similar) — run an Indri query string through `IndriQueryParser` +
  `IndriIndexSearcher` and return ranked `(externalId, score)` pairs, so operator/scoring tests
  are one-liners.

Keep it in the searcher module (both dependent tasks live there) — no cross-module test-jar needed
now. If the indexer later wants shared helpers, revisit.

### 3c. Document the standard AND the policy in `CLAUDE.md`
Add a "Testing" section to `CLAUDE.md` with two parts:

- **Policy (mandate).** State clearly that **all development work on this project shall include
  appropriate automated tests** — every bug fix ships with a regression test that fails before /
  passes after, every new feature (e.g. a new document parser or query operator) ships with tests
  covering its contract, and a change is not "done" until `mvn test` is green in the affected
  module(s). This is a standing rule for any future Claude session or contributor, phrased as an
  instruction (it lives in the file that overrides default behavior).
- **How testing is done here.** Pinned versions (`junit-jupiter:5.8.2`, `maven-surefire-plugin:2.22.2`),
  the `src/test/java/...` layout, `mvn test` to run, "reuse the `searcher/testutil` helpers
  (`TestIndex` + query runner) rather than rebuilding fixtures," where the indexer's parser tests
  live as an example (TASK-0001), and the note that a parent/reactor POM is a deferred option.

## 4. Acceptance criteria

1. `mvn -q test` is green in all three modules and actually executes Jupiter tests (a trivial
   smoke test in Searcher and Analyzer is fine as proof the wiring runs).
2. The `searcher/testutil` helpers exist and are demonstrated by at least one test that builds a
   tiny index and runs a query end to end.
3. Versions are identical across the three module poms.
4. `CLAUDE.md` contains the testing **policy** (all development work ships with appropriate
   tests; not done until `mvn test` is green) **and** the how-to (versions, layout, running,
   reuse `searcher/testutil`).
5. No change to the existing independent build order or module versions.

## 5. Relationship to other tasks

- **[[TASK-0002]]** and **[[TASK-0003]]** now **depend on this task**. Once this lands, their test
  sections reduce to "write tests using `searcher/testutil`" — they must NOT re-add JUnit/surefire
  wiring or rebuild the tiny-index fixture. (Their specs have been updated to point here.)

## 6. Out of scope

- A parent/reactor POM (deferred; see §2).
- Broad test coverage of existing code — this task establishes the framework + shared fixtures and
  a couple of smoke tests, not a full test suite.
