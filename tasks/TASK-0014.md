# TASK-0014 — Alias `#odN` to `#N`, and reject unknown `#operator`s instead of silently degrading to `#and`

- **Status:** Ready
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** Lucindri's query parser recognizes only a fixed set of operators. Any `#word` operator
  it does not recognize — including Indri's very common **`#odN`** ordered-window spelling — is
  **silently degraded to `#and`/`#combine`** instead of being executed or rejected. A user porting
  Indri queries gets wrong results with no warning. Fix: (1) treat `#odN` as an alias for `#N` (the
  ordered window), and (2) make the parser **reject** any operator not on an explicit allow-list of
  implemented-and-tested operators, with a clear error. Silent degradation must be impossible.

## Evidence (verified 2026-07-05, on the t45mCR indexes)

`#odN` is Indri's canonical ordered-window spelling; `#N` is shorthand for the same operator. In
**Indri** they are identical and both differ from `#combine`:

| query | Indri top doc:score | Lucindri top doc:score |
|---|---|---|
| `#5(peace talks)` | FBIS3-48064 : −5.79 | FBIS3-48064 : −5.71 |
| `#od5(peace talks)` | FBIS3-48064 : −5.79 **(= #5)** | FBIS4-4440 : −4.69 **(= #combine)** |
| `#combine(peace talks)` | FBIS4-4440 : −4.94 | FBIS4-4440 : −4.69 |

So in Lucindri `#od5` produces byte-identical results to `#combine` — it degraded. The ordered-window
*semantics* are correct and Indri-matching (see the `#5` rows, identical rankings) and were tested in
TASK-0010; only the `#odN` **surface spelling** is unwired, and unknown operators fail silently.

## Root cause (in `LucindriSearcher/.../parser/IndriQueryParser.java`)

- **`createOperator(...)`** (~line 167) normalizes surface spellings to internal operator names: pure
  digits `#N` → `near/N`; `#uwN` → `window/N`. There is **no** case for `#odN`, so `od5` stays the
  literal string `od5`.
- **`createBooleanClause(...)`** (~lines 425–483) dispatches known names (`or`, `wsum`, `max`, `wand`,
  `near`, `window`, `band`, `syn`, `not`, `scoreif`, `scoreifnot`) to real query classes. The legit
  belief-combine names (`combine`, `and`, `weight`) intentionally fall through to the final branches:
  ```
  } else if (clauses.size() == 1) {   // single operand -> return the child
      query = clauses.get(0).getQuery();
  } else {                            // >=2 operands -> IndriAndQuery
      query = new IndriAndQuery(clauses);
  }
  ```
  **Any unrecognized name reaches these same two branches**, so `#od5(a b)` → `IndriAndQuery` and
  `#bogus(a)` → just `a`. That is the silent-degradation bug, for `#odN` and for every unimplemented
  operator (`#wsyn`, `#prior`, `#less`, `#greater`, `#between`, `#equals`, `#date:*`, typos, …).

## Plan

1. **Alias `#odN` → ordered window.** In `createOperator`, add a normalization case: an operator of
   the form `od` followed by digits (`od\d+`) becomes `near/<digits>` (exactly like the pure-digit
   `#N` case). Require the digits — `#od` with no number is invalid. After this, `#od5` builds the same
   `IndriNearQuery(distance=5)` as `#5`. (`#uwN` already covers the unordered spelling; there is no
   `#odN`-vs-`#N` semantic difference to preserve.)
2. **Introduce an explicit allow-list and reject everything else.** Define the set of operators the
   parser actually implements **and** has passing tests for:
   `and, combine, weight, wand, wsum, or, not, max, syn, band, near, window, scoreif, scoreifnot`
   (surface `#N`/`#odN` → `near`, `#uwN` → `window`). Validate the normalized operator name against
   this set and `syntaxError(...)` (the existing `IllegalArgumentException` path) with a clear message
   for anything else — e.g. `Unknown or unsupported operator: #od  (did you mean #odN / #N ?)` and
   `Operator #wsyn is not implemented in Lucindri`. Validation must run **before** the
   single-operand/`IndriAndQuery` branching so a single-operand unknown (`#bogus(dog)`) cannot slip
   through the `clauses.size()==1` shortcut either.
   - Keep the final `IndriAndQuery` branch reachable **only** for `combine`/`and`/`weight` (and the
     single-operand shortcut only for those). The simplest robust structure: reject unknown names up
     front (in or right after `createOperator`), so by the time dispatch runs, every name is known.
3. **Preserve today's correct behavior.** Do not change how any currently-valid operator scores; this
   is a parser gate, not a scoring change. Watch the edge case where `near`/`window` currently require
   `clauses.size() > 1` (a 1-operand `#5(x)` yields a null query today) — decide whether that should be
   a clear error too, and if so make it one (silent null is its own foot-gun).
4. **Update docs.** In `README.md` "Not (yet) implemented", change the wording from "will either error
   or be silently ignored" to state these operators now **error** (they are rejected). Add `#odN` to
   the ordered-window description as an accepted alias for `#N`, and remove the note that `#odN`
   degrades to `#and`. (If `docs/indri-query-language.md` still exists, fix or drop its matching claim.)

## Acceptance criteria / tests (required, per repo policy)

Add regression tests (extend `ProximityOperatorTest` and the `parser` tests; reuse the `TestIndex`
fixture). Each new gate ships with a test that fails before / passes after:
- **`#odN` alias:** on the shared fixture, `#od5(a b)` returns the **same** hits/scores as `#5(a b)`,
  and **differs** from `#combine(a b)` (the exact inversion of today's bug). Also `#od1` ≡ `#1`.
- **Unknown operator rejected:** `#bogus(a b)` and single-operand `#bogus(a)` both throw
  `IllegalArgumentException` with a message naming the operator — they must NOT return `#and`/the child.
- **Unimplemented Indri operators rejected:** `#wsyn(...)`, `#prior(...)`, `#less(...)` (a representative
  sample) throw rather than silently degrade.
- **No regression:** every implemented operator (`#combine`, `#and`, `#weight`, `#wand`, `#wsum`, `#or`,
  `#not`, `#max`, `#syn`, `#band`, `#N`, `#uwN`, `#scoreif`, `#scoreifnot`) still parses and scores as
  before; full searcher suite green.
- Rebuild the searcher fat jar; optionally re-confirm on t45mCR that `#od5` now equals `#5` and the
  overall Indri-vs-Lucindri agreement is unchanged or improved.

## Notes

- Scope is the searcher parser only (`IndriQueryParser`); the indexer/analyzer are untouched.
- Rationale (owner, 2026-07-05): "Users of the system must not be allowed to shoot themselves in the
  foot silently." A rejected query with a clear message is always better than a query that runs the
  wrong operator.
