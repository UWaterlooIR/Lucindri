# TASK-0020 — Fix docno (externalId) correctness across all document parsers + make it exact-lookupable

- **Status:** Ready
- **Created:** 2026-07-07
- **Owner:** (unassigned)
- **Summary:** Two coupled defects in the indexer's docno handling, fixed together. **(1) Name bug:** the
  base constant `DocumentParser.EXTERNALID_FIELD` is `"id"`, but the searcher prints the TREC DOCNO from
  a stored field literally named `"externalId"` — so the **9 of 15 parsers** that emit their docno under
  `"id"` (8 inheriting the base, 1 redefining it wrong) produce a **blank DOCNO column** in TREC output
  today. **(2) Type bug:** every field, including `externalId`, is indexed **tokenized**, so a docno
  cannot be looked up by an exact `TermQuery` — blocking any "fetch document by docno" feature. Fix (1) at
  the root (base constant → `"externalId"`, remove per-parser redefinitions) so **all** parsers emit the
  docno under `"externalId"`; fix (2) by indexing `externalId` as a non-analyzed keyword `StringField`
  (stored, indexed, not tokenized). This is a prerequisite that lets **TASK-0019** (the HTTP service)
  implement `/document` as a routine one-term lookup, cleanly.

## Context / why (the audit)

The searcher reads the DOCNO from the stored field named `externalId`
(`IndriSearch.EXTERNALID_FIELD = "externalId"`, `doc.get(EXTERNALID_FIELD)`). But in the indexer, the base
class `DocumentParser` declares `public final static String EXTERNALID_FIELD = "id"` (line 15). Auditing
every parser's docno-emit line (`new ParsedDocumentField(EXTERNALID_FIELD, <docno>, false)`) against the
field name that symbol resolves to:

| docno lands in field | parsers | status |
|---|---|---|
| `"externalId"` (local constant, correct) | Climbmix, ClueWeb12, ClueWeb22, IndriGov2, TrecText, WARC | ✅ non-blank DOCNO |
| `"id"` (inherits base constant) | CAR, Gov2, Json, MARCO, MARCOFull, Text, WSJ, WashingtonPost | ❌ **blank DOCNO** |
| `"id"` (redefines the local constant *to* `"id"`) | ClueWeb09 | ❌ **blank DOCNO** |

So **9 of 15 parsers emit an unusable (blank) DOCNO** in TREC output right now — a pre-existing, latent
correctness bug, independent of the HTTP service. `ClueWeb09` is the giveaway that this was drift, not
design: it sits beside `ClueWeb12`/`ClueWeb22`, copies their shape, and defines its own
`EXTERNALID_FIELD` — set to `"id"` while they set `"externalId"`. CLAUDE.md already notes the blank-DOCNO
symptom for `JsonDocumentParser` as if it were a one-off; the audit shows it is the majority case.

Every parser *does* intend to emit a docno (all 15 build a field through the `EXTERNALID_FIELD` symbol,
and separately an `internalId` through `INTERNALID_FIELD`/`ID_FIELD`), so the design intent is uniform —
only the execution drifted. This task makes the execution uniform and correct, and makes the docno
exact-lookupable at the same time.

**Proof of functionality.** As the end-to-end demonstration that the fix works on a *real*
LucindriIndexer-built index (not just a unit fixture), this task also delivers a small command-line
program, **`LucindriGetDoc`**, that takes a Lucindri index and an external id (docno) and prints that
document's stored contents (`fulltext`) to stdout — the manual, scriptable proof that exact docno lookup
works end-to-end, and a foreshadowing of TASK-0019's `/document`.

### Verified facts the plan relies on

- Every parser `extends DocumentParser` **directly** (no intermediate class), so fixing the base constant
  flows to all inheritors.
- **No non-parser code references `EXTERNALID_FIELD`** (`grep` over `LucindriIndexer/src/main` outside
  `documentparser/` is empty). Renaming the base constant changes only the emitted docno field name.
- **Nothing anywhere reads a field literally named `"id"`** — the searcher only reads `"externalId"` and
  `"fulltext"`; the query parser's default-field picker skips any field whose name contains the substring
  `"id"` (true for both `"id"` and `"externalId"`, so unchanged). Renaming is behavior-safe.
- Document parsers do **not** create Lucene `Field`s — they emit `ParsedDocumentField(name, content,
  numeric)` triples; the sole place a Lucene `FieldType` is chosen is `LuceneDocumentWriter` (which already
  branches on `numeric` to emit `NumericDocValuesField`). So the type change (2) is localized to the writer
  and needs **no parser edits**.
- Existing indexer tests are unaffected: `TrecTextDocumentParserTest` and `ClimbmixJsonlDocumentParserTest`
  already assert on `"externalId"` (both parsers are already correct); `LuceneDocumentWriterExactLengthTest`
  builds an `externalId` text field but only asserts on `fulltext_len`/`year_len`, never on an
  `externalId_len`, so making `externalId` a keyword (no `_len`) leaves it green.

## Root cause

A wrong single source of truth plus copy-paste duplication: the base `EXTERNALID_FIELD = "id"` is simply
the wrong name, and because most parsers redundantly re-declare `EXTERNALID_FIELD`/`ID_FIELD` locally
instead of using the base, the value drifted (some to `"externalId"`, one to `"id"`, the rest inheriting
the wrong base). The fix is to make the base correct and let every parser use it.

## Scope

**In scope**
- Correct the docno field name to `"externalId"` for **all** parsers (root-constant fix + remove local
  redefinitions).
- Index `externalId` as a non-analyzed keyword `StringField` (stored + indexed, not tokenized) in the
  standalone Lucene indexer, so `new TermQuery(new Term("externalId", docno))` is an exact single-doc
  lookup.
- Tests proving (a) the docno round-trips exactly as a single non-analyzed term even for docnos containing
  tokenizer-splitting characters, and (b) the parsers we can synthesize input for emit `"externalId"`.
- **`LucindriGetDoc`** (searcher module): a CLI that, given an index and a docno, prints the document's
  `fulltext` to stdout — the end-to-end proof of functionality.
- **`TestIndex` fixture faithfulness** (searcher module): index `externalId` as a keyword `StringField`
  (no `externalId_len`), mirroring the production writer change, so the searcher tests stay faithful.
- Consequence: this task touches **both** modules (`LucindriIndexer` writer + parsers; `LucindriSearcher`
  `LucindriGetDoc` + `TestIndex`). Per CLAUDE.md, `mvn test` must be green in **both**.

**Out of scope**
- The `/document` HTTP endpoint and the `LucindriSearchService` core — that is **TASK-0019**; `LucindriGetDoc`
  is a standalone utility (its own `main`), not that service. TASK-0019 may later reuse the same one-term
  lookup, but this task does not build the service.
- `internalId` handling — it stays a tokenized/stored field (its value `"internalId"` is already
  consistent between the base `INTERNALID_FIELD` and the per-parser `ID_FIELD`, and nothing reads it). Not
  worth churning here; note only.
- The **Solr** indexing path (`SolrDocumentWriter`) — untested/unsupported per README; the constant change
  flows to it harmlessly, but we make no Solr-specific keyword change.
- Reindexing any production corpus — operational, done when TASK-0019 stands up its served index.

## Design decisions (each is a decision, not a discovered constraint — veto any)

1. **Fix the name at the root, delete local redefinitions.** Set `DocumentParser.EXTERNALID_FIELD =
   "externalId"` and **remove every parser-local `EXTERNALID_FIELD` redefinition** (the 6 redundant
   `"externalId"` copies + ClueWeb09's wrong `"id"`), so there is exactly one definition. This prevents the
   drift that caused the bug. *Minimal alternative (rejected):* fix only the base + ClueWeb09 and leave the
   6 redundant copies — smaller diff, but leaves the same future-drift landmine the owner explicitly wants
   removed.
2. **Keyword by field name, in the writer (name-based, Option A).** `LuceneDocumentWriter` treats the field
   named `DocumentParser.EXTERNALID_FIELD` as a keyword `StringField`. *Rejected alternative (Option B):*
   add a `keyword` kind to `ParsedDocumentField` and have each parser mark its id field — more general but
   touches `ParsedDocumentField` + many parsers for no present benefit; the writer already keys behavior
   off the field identity (it special-cases `numeric`), and the searcher already treats `"externalId"` as a
   magic literal, so a name-based rule is consistent, not a new hack.
3. **Replace, don't duplicate.** `externalId` becomes a `StringField` (stored + indexed-not-tokenized),
   fully replacing its current tokenized `Field`. It is not additionally kept as a tokenized field, because
   nothing queries `externalId` (behavior-neutral for search) and a second field wastes space.
4. **Keyword field carries no norms/positions/`_len`.** `StringField` indexes `DOCS`-only and omits norms
   by default; the writer must **not** run token-counting/`<field>_len` (TASK-0012) on the keyword field.
5. **`LucindriGetDoc` does an exact `TermQuery`, with no analyzer.** Because `externalId` is now a keyword,
   the lookup is `new TermQuery(new Term("externalId", rawDocno))` — the docno is used **verbatim**, never
   analyzed. (Not needing an analyzer at all is precisely the property this task creates; if you find
   yourself analyzing the docno to make the lookup work, the keyword change didn't take.) It uses a plain
   `IndexSearcher` (no similarity/scoring needed — it's a membership lookup) over the same single-or-comma-
   separated index handling `IndriSearch` uses (`MultiReader` for multiple dirs).
6. **`LucindriGetDoc` needs no packaging change to run.** The class ships inside the existing searcher
   `jar-with-dependencies`; invoke it with an explicit main class
   (`java -cp LucindriSearcher-1.5-jar-with-dependencies.jar org.lemurproject.lucindri.searcher.LucindriGetDoc …`).
   A dedicated second assembly/`mainClass` is an optional convenience, not required.

## Implementation plan

### Step 1 — Root docno-name fix (parsers)

1. `LucindriIndexer/.../documentparser/DocumentParser.java:15` — change
   `public final static String EXTERNALID_FIELD = "id";` → `= "externalId";`.
2. Remove the redundant/incorrect **local** `EXTERNALID_FIELD` declarations so each parser uses the
   inherited base constant (unqualified `EXTERNALID_FIELD` references keep compiling and now resolve to
   `"externalId"`):
   - Delete local `EXTERNALID_FIELD` in: `ClimbmixJsonlDocumentParser` (line ~51), `ClueWeb12DocumentParser`
     (~33), `ClueWeb22DocumentParser` (~33), `IndriGov2DocumentParser` (~28), `TrecTextDocumentParser`
     (~36), `WARCDocumentParser` (~32) — these were `"externalId"` (no behavior change).
   - Delete local `EXTERNALID_FIELD` in `ClueWeb09DocumentParser` (~33), which was **`"id"`** — this is the
     behavior fix (blank → correct DOCNO).
   - Leave the per-parser `ID_FIELD = "internalId"` declarations alone (out of scope; consistent value).
   - **Do not touch** the 8 parsers with no local `EXTERNALID_FIELD` (CAR, Gov2, Json, MARCO, MARCOFull,
     Text, WSJ, WashingtonPost) beyond the base fix — they inherit the now-correct constant automatically.
3. Compile the module (`mvn -q -pl LucindriIndexer compile`) to confirm every unqualified
   `EXTERNALID_FIELD` still resolves (it will, via the base class).

### Step 2 — Keyword `externalId` (writer)

In `LucindriIndexer/.../documentwriter/LuceneDocumentWriter.java`, `writeDocuments(...)` field loop
(around lines 134–153):

- Add a first branch, before the `!numeric` text branch, that recognizes the docno field by name:
  ```java
  if (DocumentParser.EXTERNALID_FIELD.equals(docField.getFieldName())) {
      // Docno: exact-match keyword, not analyzed text. Stored (for TREC output + /document) and
      // indexed as a single term (for exact TermQuery lookup). No norms, no positions, no _len.
      luceneDoc.add(new org.apache.lucene.document.StringField(
              docField.getFieldName(), docField.getContent(), Field.Store.YES));
      continue; // skip the tokenized-Field path AND the exactDocumentLength token count
  }
  ```
  (`continue` requires the loop body be structured so the numeric/text branches are the `else` path; keep
  the existing `numeric` handling unchanged for all other fields.)
- Ensure the `fieldLengths` token-counting (TASK-0012) is **not** reached for the keyword field (the
  `continue` above guarantees it; double-check no `externalId_len` is written).
- Import `org.apache.lucene.document.StringField` (and reference `DocumentParser.EXTERNALID_FIELD`).

Rationale recap: `StringField` is `indexed (DOCS) + not tokenized + optionally stored`; with `Store.YES`
the stored value that `IndriSearch`/`/document` reads is unchanged, while the indexed term is the whole
docno, so exact lookup works regardless of characters (underscore, hyphen, period, etc.).

### Step 3 — Keep the searcher `TestIndex` fixture faithful

In `LucindriSearcher/src/test/.../testutil/TestIndex.java` (builder, ~lines 213–217): index `externalId`
as a keyword `StringField` (stored, not tokenized) instead of the tokenizing `fieldType`, and stop
emitting the `externalId_len` NumericDocValues. Leave `fulltext` exactly as it is. This mirrors the
Step‑2 production writer so searcher tests remain production-faithful (CLAUDE.md), and it lets
`LucindriGetDoc`'s unit test (Step 5) exercise real keyword lookup — including docnos with
tokenizer-splitting characters. Existing searcher tests use single-token docnos (`d1`, `d2`) and read the
*stored* `externalId`, so they are unaffected.

### Step 4 — `LucindriGetDoc` program (searcher module)

New class `org.lemurproject.lucindri.searcher.LucindriGetDoc`, alongside `IndriSearch`:

- **CLI:** `LucindriGetDoc <indexDir[,indexDir2,…]> <docno>`. Open the index exactly as `IndriSearch`
  does (one `DirectoryReader`, or a `MultiReader` over comma-separated dirs).
- **Lookup:** `new IndexSearcher(reader).search(new TermQuery(new Term("externalId", docno)), 1)`. No
  analyzer, no similarity.
- **Output contract (Unix-friendly, so it is scriptable):**
  - found → print the document body to **stdout** and exit **0**. Use
    `String.join("\n", reader.document(hit).getValues("fulltext"))` so a multi-valued body isn't truncated
    (`doc.get` returns only the first value).
  - not found → print nothing to stdout, a `no document with externalId: <docno>` line to **stderr**,
    exit **1**.
  - wrong/missing args → usage to stderr, exit **2**.
  - (optional) more than one hit for a docno → print the first, warn to stderr (a duplicate docno is a
    data problem, not a crash).
- **Testability:** factor the lookup into a static, side-effect-free method
  `static String fetch(IndexReader reader, String docno) throws IOException` returning the body or `null`;
  keep `main` a thin wrapper doing arg-parsing, reader-open, and stdout/stderr/exit. Tests call `fetch`
  directly (no stdout capture needed).

### Step 5 — Tests (per CLAUDE.md: `mvn test` green in every module touched)

Add to `LucindriIndexer/src/test/.../documentwriter/`:

1. **Keyword round-trip / exact-lookup capability test** (the stand-in for the un-synthesizable parsers).
   Build a tiny index via `LuceneDocumentWriter` with several docnos that *would* be split by the text
   analyzer — e.g. `"shard_00000_0"`, `"wsj-90-01-01"`, `"doc.42"` — each with a distinct `fulltext`. Open a
   `DirectoryReader` and assert:
   - `new IndexSearcher(reader).search(new TermQuery(new Term("externalId", "wsj-90-01-01")), 10)` returns
     **exactly one** hit, and it is the right document (check its stored `fulltext`).
   - The stored docno round-trips: `reader.document(hit).get("externalId").equals("wsj-90-01-01")`.
   - `externalId` is a **single indexed term equal to the whole docno** (not tokenized): e.g. via
     `MultiTerms.getTerms(reader, "externalId")` the term `"wsj-90-01-01"` exists and `"wsj"`/`"90"` do
     **not**. This is the assertion that proves the class-of-bug fix generally, not just for underscores.
2. **No `externalId_len` even with `exactDocumentLength=true`** — extend/mirror
   `LuceneDocumentWriterExactLengthTest`: with the flag on, `readLengths(..., "externalId_len", n)` is
   `null`, while `fulltext_len` is unchanged. (Guards decision 4.)
3. **Parser docno-name tests where input is synthesizable.** Confirm the emitted docno field name is
   `"externalId"` for the parsers we can feed without a corpus:
   - `TextDocumentParser` (uses filenames as docno — was inheriting `"id"`): add a small test that parses a
     temp file and asserts `field(doc, "externalId")` is the filename and is non-blank. This is a *behavior
     fix* regression test (blank → correct).
   - `JsonDocumentParser` (was `"id"`): if a JSON doc can be synthesized as in existing tests, assert its
     docno lands in `"externalId"`.
   - `ClimbmixJsonlDocumentParser` / `TrecTextDocumentParser`: already covered by existing tests (already
     correct) — re-run to confirm still green.
   - **Not unit-testable without real corpora** (accept, per owner): ClueWeb09/12/22, WARC, Gov2/IndriGov2,
     CAR, WaPo, MARCO/MARCOFull. Their correctness now rests on the single base constant + Step 1; the
     capability test (Step 5.1) proves the writer half. Note this coverage boundary in the PR.
4. Run the whole indexer suite (`mvn -pl LucindriIndexer test`) — expect all existing tests green.

Add to `LucindriSearcher/src/test/.../`:

5. **`LucindriGetDoc` unit test.** Over a `TestIndex`-built index (now keyword `externalId`, Step 3), with
   at least one docno containing tokenizer-splitting characters (e.g. `wsj-90-01-01`): assert
   `LucindriGetDoc.fetch(reader, "wsj-90-01-01")` returns that document's exact `fulltext`; a known
   single-token docno returns its body; an **unknown** docno returns `null`; and multi-index
   (`MultiReader`) resolves a docno in any sub-index. Run `mvn -pl LucindriSearcher test` green.

**Cross-module end-to-end proof (script, not a unit test).** Because `LucindriIndexer` does not depend on
`LucindriSearcher` (and vice-versa), no single Maven module can both *build* an index with the real
`LuceneDocumentWriter` and *read* it with `LucindriGetDoc`; the two unit tests above cover each half. The
genuine end-to-end proof is a small committed script (e.g. `scripts/getdoc-smoke.sh`, in the style of
TASK-0019's conformance harness) that:

1. builds a tiny index from a synthetic **trectext** file with the real `LucindriIndexer` jar (a format
   we can hand-write, and one whose docno path this task fixes),
2. runs `LucindriGetDoc <index> <a known DOCNO>` from the searcher jar,
3. asserts stdout equals that document's body and that an unknown docno exits non-zero.

Document the exact invocation in the task/PR so the proof is reproducible. This is the objective "it works
on a real Lucindri index" gate.

## Behavioral changes & compatibility

- **TREC output changes for the 9 previously-broken formats:** their DOCNO column goes from **blank →
  correct**. This is the intended fix. The 6 already-correct parsers are byte-identical.
- **`externalId` is no longer a tokenized/searchable field.** Behavior-neutral: nothing queries it, and the
  default-field picker still skips it. Stored value (hence TREC output and `/document`) unchanged.
- **Requires reindexing** to take effect (both the name fix and the keyword type are index-time). Any index
  built before this task keeps its old behavior; the 9 broken formats' old indexes were already
  blank-DOCNO, so nothing of value regresses. TASK-0019's served index must be built with the post-TASK-0020
  indexer.
- **`internalId`** is unchanged (still tokenized/stored). If exact `internalId` lookup is ever wanted, it is
  a trivial follow-up mirroring this task — out of scope now.

## Risks / treat-with-suspicion notes

- **Un-testable parsers (ClueWeb/WARC/gov2/CAR/WaPo/MARCO).** We cannot exercise them end-to-end without
  their corpora. Mitigation: they all now derive the docno name from the single base constant (Step 1) and
  the writer half is proven by the capability test (Step 5.1). The change is *highly likely to only make
  them correct* (blank → non-blank docno) and cannot make a currently-correct one worse (the 6 correct
  parsers keep emitting `"externalId"`). Owner has accepted this coverage boundary.
- **Someone builds a docno longer than Lucene's max term (32766 bytes).** `StringField` would throw at index
  time. Docnos are short identifiers; not a real risk, but the failure is loud (not silent), which is
  acceptable.
- **Multi-valued `externalId`** (a parser emitting the docno twice) would index two keyword terms and store
  two values; `doc.get` returns the first. No real parser does this; note only.
- **Verify no hidden reader of `"id"`.** Confirmed by grep now, but the implementer should re-grep
  (`"id"` as a Lucene field name) after any rebase before merging, since the whole safety of the rename
  rests on it.
