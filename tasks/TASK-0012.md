# TASK-0012 — Store exact document length for Indri-parity scoring (bypass Lucene's lossy norm)

- **Status:** Done (2026-07-05)
- **Created:** 2026-07-05
- **Owner:** Claude (branch `task-0012-exact-document-length`)
- **Summary:** Lucindri stores document length in a Lucene **norm**, which is quantized to a single
  byte (SmallFloat). Indri uses the **exact** length. On realistic document lengths this makes every
  Dirichlet/JM score differ by a small amount (~0.02 in log at μ=2000 for ~700-token docs), which
  shifts rankings. **Direction (owner, 2026-07-05):** offer an **index-time configuration choice**
  between the default 1-byte norm and storing the **exact length in `NumericDocValues`**, so we can
  index+query with exact doc lengths, compare to Indri, and then decide whether the extra storage is
  worth it. The scoping below assesses the disruption.

## Evidence (found in TASK-0011)

On a tokenizer-neutral integer collection from LATimes (20k docs) where `|C|`, `cf`, and positions
align **exactly** between the engines, single-term scores still diverge:

- doc `d13365` (Indri exact length **716**), term `45`: Indri `−4.85425` vs Lucindri `−4.83491`
  (Δ ≈ 0.019). Backing the length out of Lucindri's score gives **≈ 664**, i.e. 716 SmallFloat-rounded
  down — the classic 1-byte norm quantization. Tiny toy docs (C1/C2, ≤ 5 tokens) matched exactly
  because SmallFloat is lossless in that range; the error only appears at realistic lengths.

The quantization is amplified through proximity backgrounds and belief combinations, and it is the
**dominant remaining term-level divergence** between the engines once the operator bugs are fixed
(TASK-0010/0011). It also caps the achievable TREC agreement.

## Root cause

`IndriSimilarity.computeNorm` returns `numTerms + 1`, which Lucene stores as a 1-byte norm;
`IndriSimilarity.BasicSimScorer.getLengthValue(norm)` decodes it via `SmallFloat.byte4ToInt` — the
standard lossy 1-byte length. Returning the raw stored norm does **not** help (it is already the
encoded byte — verified: it makes scores worse). So exact lengths require storing the length
elsewhere.

## The decision / options

1. **Store exact length behind an index-time config flag** (recommended): default keeps the 1-byte norm;
   `exactDocumentLength=true` additionally writes a per-doc `NumericDocValues` length, which the scorers
   auto-detect and use instead of the norm. See **Scoping / disruption** below for the concrete touch
   points (moderate effort, low risk, ~8 files). Lets us index+query exact lengths, measure vs Indri,
   and decide whether the storage is worth keeping.
2. **Accept & catalog** the quantization as a known small divergence (like the tokenizer, TASK-0008).
   Cheapest; leaves rankings slightly off from Indri on long documents.

## Scoping / disruption (assessed 2026-07-05 by reading the code)

**Favorable architecture:** every score gets its length through a *single* seam —
`IndriSimilarity.BasicSimScorer.score(freq, norm)` → `score(stats, freq, getLengthValue(norm))`. All
scorers hold a stock Lucene `LeafSimScorer` (reads the 1-byte norm) and only ever call
`docScorer.score(docId, freq)`; the stats are encapsulated in the SimScorer, so **only the length comes
from the norm.** And `LuceneDocumentWriter` already writes `NumericDocValuesField` (for numeric fields),
so the write mechanism exists.

**Design: gate on an index-time flag `exactDocumentLength` (default off = today's behavior).**

Index side — *low* disruption (~2 files + config):
- Add `exactDocumentLength` to `IndexingConfiguration` / `IndexOptionsFactory`.
- In `LuceneDocumentWriter.writeDocuments`, when enabled, for each indexed (non-numeric) field, tokenize
  the content with the analyzer to **count tokens** and add `NumericDocValuesField(field + "_len", count)`
  next to the existing `Field`. Cost: one extra analysis pass per field at index time (only when on).
  (There is no side-channel from `computeNorm` to DocValues, so a pre-count pass is required.)

Query side — *moderate* but contained to the scoring path (~6 files):
- `IndriSimilarity.BasicSimScorer`: add `scoreWithLength(freq, exactLength)` (skips `getLengthValue`).
- Introduce a one-method length-source abstraction (`float score(int docId, float freq)`) with two impls:
  **norm-based** (wraps `LeafSimScorer`, current) and **exact** (reads the `_len` `NumericDocValues` and
  calls `scoreWithLength`). The scorers only call that one method, so the interface is tiny.
- Change the 3 scorers (`IndriTermScorer`, `IndriTermOpScorer`, `IndriMissingTermScorer`) to take the
  abstraction instead of the concrete `LeafSimScorer`.
- Change the 2 weights (`IndriTermQuery`, `IndriTermOpWeight`) to pick the exact source **when the `_len`
  DocValues exists** for the field, else the norm source — i.e. **auto-detected, no query-side config.**

**Risk: low.** Purely additive and gated — an index without `_len` DocValues auto-detects to the norm
path and is **bit-for-bit identical to today**. Only the per-doc length input changes; matching,
iteration, operators, and query parsing are untouched. The scorer change is a mechanical refactor behind
a one-method interface.

**Costs to weigh (the point of the config):**
- Storage: ~8 bytes/doc/indexed-field for the `_len` `NumericDocValues`, but Lucene delta/GCD-compresses
  these heavily (usually much less). On the full climbmix index (553M docs, fulltext) ≤ ~4.4 GB before
  compression vs a 1.4 TB index — ~0.3%.
- Index time: the extra per-field analysis pass.

**Effort:** index side ~½ day; query side ~1–1.5 days (abstraction + 6 files + regression test). No
ripple into the query language or operators.

**Semantic caveat (decision, not architecture):** the count must match Indri's `|d|` — `numTerms` with
**no `+1`** (the norm uses `numTerms+1`), and the removed-stopword question is **[[TASK-0009]]**. The
feature removes the SmallFloat quantization regardless of that decision, but exact Indri parity needs
TASK-0009 settled too. Scope the first cut to `fulltext` (field-restricted/per-field exact length ties
to the deferred field work, TASK-0017).

  off); with it **off** the index and scores are **bit-for-bit unchanged** vs today (no-regression via

  tokens) matches Indri to ~1e-4 (not just the toy short-doc case), and the TASK-0011 integer-collection

  informed. If "accept": a catalog entry quantifying the effect.
- **Either way, update `docs/indri-query-language.md` §6B** (the Indri-vs-Lucindri differences guide),
  which currently documents norm quantization as an open decision: record the outcome, and if exact
  length was implemented, change §6B to state that Lucindri now scores with the exact document length
  (no SmallFloat quantization).

## Implementation plan (2026-07-05)

**Decisions locked (owner, 2026-07-05):**
- **Field scope:** write the exact-length DocValues for **all indexed text (non-numeric) fields**, not just
  `fulltext`. The query side auto-detects per field, so this is future-proof for TASK-0017 field scoring
  and is the same code either way.
- **Verification depth:** **full acceptance in this task** — implement + unit tests **and** run the C++
  Indri differential on the LATimes integer collection (700+ token doc to ~1e-4) and report the measured
  storage delta before closing.

### Design invariants (must hold)

1. **Flag off ⇒ bit-for-bit unchanged.** An index built without the exact-length DocValues, or a build
   with `exactDocumentLength=false`, takes the existing norm path and produces identical scores. Auto-
   detection is by presence of the `<field>_len` DocValues at query time — **no query-side config**.
2. **The stored length equals Indri's `|d|` = `numTerms` with _no_ `+1`.** Index-time similarity is stock
   `LMDirichletSimilarity`, whose `computeNorm` stores `SmallFloat.intToByte4(numTerms)` with **no `+1`**
   (`IndriSimilarity.computeNorm`'s `numTerms+1` is **dead code** for search — query-time norms are never
   recomputed; do not "fix" it). So the un-quantized exact length must be exactly `numTerms`. Concretely
   `numTerms = state.getLength() - state.getNumOverlap()` = **count of analyzer-emitted tokens whose
   `positionIncrement > 0`** (removed stopwords are not emitted; synonym/overlap tokens have posIncr 0 and
   don't count). Counting this way makes short/lossless-range docs bit-identical to the norm path and long
   docs the un-quantized `numTerms`.
3. **Stopword semantics are out of scope here** — governed by [[TASK-0009]]. This feature removes the
   SmallFloat quantization regardless; whatever the analyzer emits today is what we count.

### Single length seam (confirmed by reading the code)

Every score reaches its length through exactly one call: the three scorers (`IndriTermScorer`,
`IndriTermOpScorer`, `IndriMissingTermScorer`) each hold a stock Lucene `LeafSimScorer docScorer` and only
ever call `docScorer.score(docId, freq)` (and `docScorer.score(docId, 0)` for `smoothingScore`).
`LeafSimScorer` reads the per-doc **norm** (`NumericDocValues`) and calls
`IndriSimilarity.BasicSimScorer.score(freq, norm)` → `score(stats, freq, getLengthValue(norm))`. Length is
the **only** input coming from the norm; stats are encapsulated in the SimScorer. So the change is: give
the scorers a one-method length source with two impls (norm vs exact), and pick the impl by whether the
`_len` DocValues exists.

### Index side (LucindriIndexer) — ~3 files

1. **`domain/IndexingConfiguration.java`** — add `private boolean exactDocumentLength;` with
   `isExactDocumentLength()` / `setExactDocumentLength(boolean)`. Default `false`.
2. **`factory/IndexOptionsFactory.java`** — read it:
   `options.setExactDocumentLength(Boolean.valueOf(properties.getProperty("exactDocumentLength")));`
   (`Boolean.valueOf(null)` is `false`, so absence = off — preserves today's behavior).
3. **`documentwriter/LuceneDocumentWriter.java`** —
   - Store `private final boolean exactDocumentLength;` (from options) and a suffix constant
     `public static final String EXACT_LENGTH_SUFFIX = "_len";`.
   - In `writeDocuments`, when the flag is on, **accumulate token counts per field name** into a
     `Map<String,Long>` while iterating `parsedDoc` fields (a Lucene doc may carry the same field name
     multiple times — multi-valued — and the norm aggregates them; only **one** `NumericDocValues` per
     field per doc is legal, so accumulate then add once). For each non-numeric field with non-null
     content, count tokens with the index analyzer:
     ```java
     long count = 0;
     try (TokenStream ts = analyzer.tokenStream(fieldName, content)) {
         PositionIncrementAttribute posIncr = ts.addAttribute(PositionIncrementAttribute.class);
         ts.reset();
         while (ts.incrementToken()) {
             if (posIncr.getPositionIncrement() > 0) count++;
         }
         ts.end();
     }
     ```
     After the field loop, for each entry add `new NumericDocValuesField(field + EXACT_LENGTH_SUFFIX, count)`.
   - Numeric fields never get a `_len`. Flag off ⇒ this whole block is skipped (no new DocValues, index
     bit-identical).
   - Cost: one extra analysis pass per text field, **only when the flag is on** (there is no side channel
     from `computeNorm` to DocValues, so a pre-count pass is required).

### Query side (LucindriSearcher) — ~6 files

4. **`similarities/IndriSimilarity.java`** — add an exact-length scoring entry point that skips the
   SmallFloat decode:
   ```java
   public interface ExactLengthSimScorer { float scoreWithExactLength(float freq, long exactLength); }
   ```
   Implement it on `BasicSimScorer`:
   `return (float) IndriSimilarity.this.score(stats, freq, exactLength);` and on `MultiSimScorer` (sum over
   subs, each cast to `ExactLengthSimScorer`). `backgroundSimScorer(...)` already returns a `BasicSimScorer`,
   so OOV/background paths get exact length for free.
5. **New file `searcher/IndriLengthSource.java`** — the one-method abstraction + a static factory that
   encapsulates auto-detection:
   ```java
   abstract class IndriLengthSource {
       final Similarity.SimScorer simScorer;
       abstract float score(int docId, float freq) throws IOException;
       Similarity.SimScorer getSimScorer() { return simScorer; }
       static IndriLengthSource create(LeafReaderContext ctx, String field,
                                       Similarity.SimScorer s, boolean needsScores) throws IOException {
           if (s instanceof IndriSimilarity.ExactLengthSimScorer) {
               NumericDocValues len = ctx.reader().getNumericDocValues(
                       field + LuceneDocumentWriter-style "_len");   // literal "_len"
               if (len != null) return new ExactLengthSource((IndriSimilarity.ExactLengthSimScorer) s, len);
           }
           return new NormLengthSource(new LeafSimScorer(s, ctx.reader(), field, needsScores));
       }
   }
   ```
   - `NormLengthSource` wraps a `LeafSimScorer` and delegates `score(docId, freq)` — **today's behavior**.
   - `ExactLengthSource` holds the `ExactLengthSimScorer` + the `_len` `NumericDocValues`; on `score`:
     `long len = lengths.advanceExact(docId) ? lengths.longValue() : 0L; return scorer.scoreWithExactLength(freq, len);`.
     This mirrors exactly how `LeafSimScorer` advances the **norms** iterator (`advanceExact` in increasing
     docId order), so the access pattern is proven safe — the DAAT merge queries each leaf's docs
     monotonically for both `score()` and `smoothingScore()`. (`advanceExact==false` means the doc lacks the
     field; a matching doc always has the term ⇒ the field ⇒ a `_len`, so 0 is only reached for the
     degenerate smoothing-on-absent-field case.)
   - The `"_len"` literal is duplicated across modules by necessity (searcher does not depend on indexer);
     add a cross-referencing comment on both constants so they stay in sync.
6. **`IndriTermScorer.java`**, **`IndriTermOpScorer.java`**, **`IndriMissingTermScorer.java`** — change the
   field/ctor param from `LeafSimScorer docScorer` to `IndriLengthSource docScorer`. Bodies are unchanged:
   `docScorer.score(docId, freq)` / `docScorer.score(docId, 0)` have the same signature. In
   `IndriTermScorer`, the `ImpactsDISI` still needs a `Similarity.SimScorer` — build it from
   `docScorer.getSimScorer()` (getMaxScore returns 0, so impacts never actually skip; the source is
   irrelevant to correctness, only shape).
7. **`IndriTermQuery.java`** (`IndriTermWeight.scorer` + the OOV branch + `explain`) — replace each
   `new LeafSimScorer(simScorer, ctx.reader(), field, needsScores)` with
   `IndriLengthSource.create(ctx, field, simScorer, needsScores)` and pass it to the scorer. The OOV/missing
   branch gets exact length too (its background `smoothingScore` divides by `|d|`, so length matters there).
8. **`IndriTermOpWeight.java`** (`getScorer` + `backgroundScorer`) — same substitution: build the length
   source via the factory and hand it to `IndriTermOpScorer` / `IndriMissingTermScorer`.

**Risk: low.** Purely additive and gated. With no `_len` DocValues present, `create(...)` returns the
norm source and the path is byte-identical to today. Matching, iteration, operators, and parsing are
untouched; the scorer edits are a mechanical type change behind one method.

### Tests (required by the CLAUDE.md testing policy)

- **Indexer — `LuceneDocumentWriterExactLengthTest` (new).** Build a tiny index with the flag on and read
  back the DocValues:
  - `fulltext_len` equals the count of analyzer-emitted `posIncr>0` tokens for a plain doc.
  - stopword-gap doc (e.g. `"cat the dog"`) ⇒ length 2 (stopword not counted), and it equals the doc's
    stored norm-decoded length in the lossless range (cross-check).
  - multi-valued field (same field name twice) ⇒ **one** `_len` DV = sum of the two counts.
  - flag **off** ⇒ `getNumericDocValues("fulltext_len") == null` (no regression, no new DocValues).
- **Searcher — extend `testutil/TestIndex`.** Add `Builder.exactDocumentLength(boolean)` that, when on,
  writes `FULLTEXT + "_len"` as a `NumericDocValuesField` computed by the same `posIncr>0` count (replicated
  in the fixture, cross-referenced to `LuceneDocumentWriter`). Faithful to production: still
  `LMDirichletSimilarity` index-time so the **norm is also written** — the two paths differ only by which
  length the scorer reads. Then:
  - `flagOff_scoresUnchanged` — a fixture **without** `_len` gives byte-identical scores to a golden value
    (auto-detection ⇒ norm path).
  - `exactLength_longDoc_matchesClosedForm` — index one **700+ token** doc (padding tokens + a target term)
    whose true `|d|` SmallFloat-quantizes; assert the single-term score equals the closed form
    `log((freq + mu*p(w|C)) / (|d|_exact + mu))` to ~1e-6, and that it **differs** from the norm-path score
    on the same doc (proving quantization is gone).
  - `autoDetect_perLeaf` — a `MultiReader` where one part has `_len` and one does not still scores each part
    by its own available length source (per-leaf detection).
- Whole searcher + indexer suites stay green (`mvn test` in each touched module).

### Clarify the "dead" `IndriSimilarity.computeNorm` (from the +1 discussion, 2026-07-05)

While scoping this task we traced the `+1` in `IndriSimilarity.computeNorm` (line 181, `return numTerms +
1;`). Findings:
- It is **inherited upstream** (initial import `f8a3782`), undocumented. Indri itself uses **raw `|d|`, no
  `+1`** (verified: `DirichletTermScoreFunction.hpp:45`, `(occurrences + μ·cf) / (contextSize + μ)`), so the
  `+1` is a divergence, not parity. Likely a "never encode a zero-length norm" guard and/or an add-one
  smoothing reflex (the same file's collection-model Javadoc mentions a non-existent `+1` too).
- The method is **not invoked in the standalone Lucindri pipeline**: the indexer binds
  `LMDirichletSimilarity` (`LuceneDocumentWriter`), whose `SimilarityBase.computeNorm` stores
  `SmallFloat(numTerms)` with no `+1`. Confirmed no `IndexWriter` in main or tests binds an Indri similarity.
- **BUT it is reachable via Solr:** `solr/IndriDirichletSimilarityFactory.getSimilarity()` returns
  `IndriDirichletSimilarity`, and Solr uses the schema `SimilarityFactory` at **index** time to compute
  norms. So in a Solr deployment using that factory, this `computeNorm` (the `+1` **and** its
  `getPosition()` length that counts stopword gaps) is live. Therefore **do not** make it `throw` and **do
  not** silently remove the `+1` here — that would change/break the Solr path and also intersects the
  stopword-length decision.

**Decision (owner, 2026-07-05):** leave the method's behavior unchanged; add an **orienting comment** on
`IndriSimilarity.computeNorm` stating: it runs only when this Similarity is the index-time similarity (the
Solr `IndriDirichletSimilarityFactory` path); the standalone indexer uses `LMDirichletSimilarity`, so there
this method is not invoked and the stored norm is `SmallFloat(numTerms)` with no `+1`; and the `+1` /
`getPosition()` length diverge from Indri's raw `|d|` — see TASK-0009 (stopword length) and this task
(exact length). **Aligning/removing the `+1` is deferred to [[TASK-0009]]** (it changes Solr norms and is
part of the stopword-length question), recorded there.
