# TASK-0012 — Store exact document length for Indri-parity scoring (bypass Lucene's lossy norm)

- **Status:** Draft
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** Lucindri stores document length in a Lucene **norm**, which is quantized to a single
  byte (SmallFloat). Indri uses the **exact** length. On realistic document lengths this makes every
  Dirichlet/JM score differ by a small amount (~0.02 in log at μ=2000 for ~700-token docs), which
  shifts rankings. **Direction (owner, 2026-07-05):** offer an **index-time configuration choice**
  between the default 1-byte norm and storing the **exact length in `NumericDocValues`**, so we can
  index+query with exact doc lengths, compare to Indri, and then decide whether the extra storage is
  worth it. The scoping below assesses the disruption.

## Evidence (found in TASK-0011)

On a tokenizer-neutral integer collection from LATimes (20k docs) where `|C|`, `cf`, and positions
align **exactly** between the engines, single-term scores still diverge:

- doc `d13365` (Indri exact length **716**), term `45`: Indri `−4.85425` vs Lucindri `−4.83491`
  (Δ ≈ 0.019). Backing the length out of Lucindri's score gives **≈ 664**, i.e. 716 SmallFloat-rounded
  down — the classic 1-byte norm quantization. Tiny toy docs (C1/C2, ≤ 5 tokens) matched exactly
  because SmallFloat is lossless in that range; the error only appears at realistic lengths.

The quantization is amplified through proximity backgrounds and belief combinations, and it is the
**dominant remaining term-level divergence** between the engines once the operator bugs are fixed
(TASK-0010/0011). It also caps the achievable TREC agreement.

## Root cause

`IndriSimilarity.computeNorm` returns `numTerms + 1`, which Lucene stores as a 1-byte norm;
`IndriSimilarity.BasicSimScorer.getLengthValue(norm)` decodes it via `SmallFloat.byte4ToInt` — the
standard lossy 1-byte length. Returning the raw stored norm does **not** help (it is already the
encoded byte — verified: it makes scores worse). So exact lengths require storing the length
elsewhere.

## The decision / options

1. **Store exact length behind an index-time config flag** (recommended): default keeps the 1-byte norm;
   `exactDocumentLength=true` additionally writes a per-doc `NumericDocValues` length, which the scorers
   auto-detect and use instead of the norm. See **Scoping / disruption** below for the concrete touch
   points (moderate effort, low risk, ~8 files). Lets us index+query exact lengths, measure vs Indri,
   and decide whether the storage is worth keeping.
2. **Accept & catalog** the quantization as a known small divergence (like the tokenizer, TASK-0008).
   Cheapest; leaves rankings slightly off from Indri on long documents.

## Scoping / disruption (assessed 2026-07-05 by reading the code)

**Favorable architecture:** every score gets its length through a *single* seam —
`IndriSimilarity.BasicSimScorer.score(freq, norm)` → `score(stats, freq, getLengthValue(norm))`. All
scorers hold a stock Lucene `LeafSimScorer` (reads the 1-byte norm) and only ever call
`docScorer.score(docId, freq)`; the stats are encapsulated in the SimScorer, so **only the length comes
from the norm.** And `LuceneDocumentWriter` already writes `NumericDocValuesField` (for numeric fields),
so the write mechanism exists.

**Design: gate on an index-time flag `exactDocumentLength` (default off = today's behavior).**

Index side — *low* disruption (~2 files + config):
- Add `exactDocumentLength` to `IndexingConfiguration` / `IndexOptionsFactory`.
- In `LuceneDocumentWriter.writeDocuments`, when enabled, for each indexed (non-numeric) field, tokenize
  the content with the analyzer to **count tokens** and add `NumericDocValuesField(field + "_len", count)`
  next to the existing `Field`. Cost: one extra analysis pass per field at index time (only when on).
  (There is no side-channel from `computeNorm` to DocValues, so a pre-count pass is required.)

Query side — *moderate* but contained to the scoring path (~6 files):
- `IndriSimilarity.BasicSimScorer`: add `scoreWithLength(freq, exactLength)` (skips `getLengthValue`).
- Introduce a one-method length-source abstraction (`float score(int docId, float freq)`) with two impls:
  **norm-based** (wraps `LeafSimScorer`, current) and **exact** (reads the `_len` `NumericDocValues` and
  calls `scoreWithLength`). The scorers only call that one method, so the interface is tiny.
- Change the 3 scorers (`IndriTermScorer`, `IndriTermOpScorer`, `IndriMissingTermScorer`) to take the
  abstraction instead of the concrete `LeafSimScorer`.
- Change the 2 weights (`IndriTermQuery`, `IndriTermOpWeight`) to pick the exact source **when the `_len`
  DocValues exists** for the field, else the norm source — i.e. **auto-detected, no query-side config.**

**Risk: low.** Purely additive and gated — an index without `_len` DocValues auto-detects to the norm
path and is **bit-for-bit identical to today**. Only the per-doc length input changes; matching,
iteration, operators, and query parsing are untouched. The scorer change is a mechanical refactor behind
a one-method interface.

**Costs to weigh (the point of the config):**
- Storage: ~8 bytes/doc/indexed-field for the `_len` `NumericDocValues`, but Lucene delta/GCD-compresses
  these heavily (usually much less). On the full climbmix index (553M docs, fulltext) ≤ ~4.4 GB before
  compression vs a 1.4 TB index — ~0.3%.
- Index time: the extra per-field analysis pass.

**Effort:** index side ~½ day; query side ~1–1.5 days (abstraction + 6 files + regression test). No
ripple into the query language or operators.

**Semantic caveat (decision, not architecture):** the count must match Indri's `|d|` — `numTerms` with
**no `+1`** (the norm uses `numTerms+1`), and the removed-stopword question is **[[TASK-0009]]**. The
feature removes the SmallFloat quantization regardless of that decision, but exact Indri parity needs
TASK-0009 settled too. Scope the first cut to `fulltext` (field-restricted/per-field exact length ties
to the deferred field work, TASK-0017).

## Acceptance criteria

- A written decision. If "store exact length": the `exactDocumentLength` index flag exists (default
  off); with it **off** the index and scores are **bit-for-bit unchanged** vs today (no-regression via
  the TASK-0011 legal fuzz); with it **on**, a single-term score on a **long** document (e.g. 700+
  tokens) matches Indri to ~1e-4 (not just the toy short-doc case), and the TASK-0011 integer-collection
  term scores align to tolerance. Report the measured storage delta so the keep/drop decision is
  informed. If "accept": a catalog entry quantifying the effect.
- **Either way, update `docs/indri-query-language.md` §6B** (the Indri-vs-Lucindri differences guide),
  which currently documents norm quantization as an open decision: record the outcome, and if exact
  length was implemented, change §6B to state that Lucindri now scores with the exact document length
  (no SmallFloat quantization).

## Reproduce

Build the LATimes integer collection (TASK-0011 Phase A) in both engines and compare single-term
scores on a long document; the residual after all TASK-0010/0011 operator fixes is this quantization.
