# TASK-0019 — Long-running HTTP/JSON search service (warm JVM + index)

- **Status:** Ready
- **Created:** 2026-07-07
- **Owner:** (unassigned)
- **Summary:** Add a **long-running HTTP/JSON service** to Lucindri that opens the index once and stays
  warm, so an interactive client can issue queries one at a time at sub-second latency (the batch CLI
  pays JVM startup + Lucene index-open per query, which is unusable in an agent loop). The service
  exposes `/search` (ranked docnos + scores, with **opt-in** query-biased summaries), `/document`
  (full stored text by docno), and `/healthz`. **Recommended architecture (see §1):** extract the
  reusable search core out of `IndriSearch`, keep the batch CLI as a thin driver over it, and add a
  **separate `LucindriServer` module** (a thin HTTP entry point over the same core) so the batch fat
  jar is not bloated with a web framework. This spec is implementation-ready.

## Context / why

A companion project (UWaterloo **Cottontail**'s ISJ search-agent stack) will drive Lucindri as one of
several interchangeable "searcher" backends behind an LLM agent, in a synchronous
query → judge → next-query loop. It needs warm sub-second queries (persistent JVM + open
`IndexReader`) and, for judging, the ability to fetch full documents by docno. Because Lucindri's
`ClimbmixJsonlDocumentParser` already indexes the climbmix `docid`/`contents` JSONL (docid →
`externalId`, contents → `fulltext`), a Lucindri index over that corpus shares docnos with any other
system's index of it — so a Lucindri-backed searcher is fully self-contained on the Lucindri index.

## 1. Architecture recommendation (investigated) — extract a shared core; dedicated thin programs

**Finding.** `IndriSearch.main` inlines the whole pipeline — read the params file, parse the XML into
`JsonIndriQueryWrapper`, open the `DirectoryReader`/`MultiReader`, build `IndriIndexSearcher` +
`IndriDirichletSimilarity` (honoring `rule`), build `IndriQueryParser` from the analysis config, then
loop the queries running `parseQuery` + `searcher.search` + TREC formatting. **There is no reusable
search-core object.** But the config model already separates the two halves cleanly:
`JsonIndriQueryWrapper` holds the **index/analysis config** (`index`, `rule`, `stemmer`,
`removeStopwords`, `ignoreCase`) *and* the **query list** (`queries`) as distinct fields. The
index/analysis half is exactly the server's *startup* config; each query is a *per-request* input.

**Recommendation: extract the core into `LucindriSearcher`, then have two thin entry points over it.**
This answers the owner's framing directly — we *do* have a clean way to share components, so dedicated
programs are the right call (not one program with a `--server` flag), because the HTTP dependency
(Jetty/Javalin) should **not** be dragged into the batch searcher's fat jar, and one `main` doing both
batch-file-reading and HTTP-serving conflates two responsibilities.

Concretely:

- **New class `LucindriSearchService`** (in `LucindriSearcher`, package `...searcher.service` or
  similar): constructed from the index/analysis config, it opens the reader once and holds a warm
  `IndexReader` + `IndriIndexSearcher` (+ the configured `Similarity`) and an `IndriQueryParser`.
  Exposes exactly two operations (see §2 for signatures/behavior): `search(queryText, count,
  wantSummaries)` and `document(docno)`. This is the single place the retrieval, summary, and
  document-lookup logic live; both entry points call it. **No behavior change to retrieval** — it is
  the same `IndriQueryParser` + `IndriIndexSearcher` + similarity path that `IndriSearch.main` uses
  today, just lifted into a reusable object.
- **`IndriSearch` (batch) becomes a thin driver:** parse the XML `JsonIndriQueryWrapper` → build a
  `LucindriSearchService` from its index/analysis config → for each query call
  `service.search(text, count, /*summaries=*/false)` → print the existing TREC line
  (`number Q0 externalId rank score lucene`). **Same jar, same mainClass, same output, same tests** —
  this is a pure refactor; guard it with the existing/added searcher tests so TREC output is
  byte-identical to today.
- **New module `LucindriServer`** (a fourth Maven module): depends on the `LucindriSearcher` artifact
  + a lightweight embedded HTTP framework; its `main` constructs one `LucindriSearchService` from
  startup config and serves the endpoints in §3. Its own `jar-with-dependencies` + `mainClass`. The
  batch fat jar stays free of the web framework.

**Rejected alternative:** a `--server` mode inside `IndriSearch`. It would pull the HTTP framework
into the searcher jar (bloat for every batch user) and overload one program with two jobs. The
extract-core approach gives the same code sharing with none of that.

## 2. `LucindriSearchService` (the shared core)

Constructed once from index/analysis config (the `JsonIndriQueryWrapper` fields minus `queries`):
`index` (one dir or comma-separated → `MultiReader`, as today), `rule` (similarity, default
`IndriDirichletSimilarity`), `stemmer`/`removeStopwords`/`ignoreCase`. It opens the reader and builds
the searcher + parser **once**.

- `List<SearchResult> search(String queryText, int count, boolean wantSummaries)`
  - `parseQuery` → `searcher.search(query, count)`; per `ScoreDoc`, read the stored `externalId`
    (docno) and score. If `wantSummaries`, also compute the summary (§4). Malformed query → throw a
    typed exception carrying the parser message (the entry points map it to a 4xx / stderr).
  - `SearchResult` = `{ String docno; float score; String summary /*null if not requested*/ }`.
- `String document(String docno)` — return the stored `fulltext` for that docno, or null if unknown
  (see §5 for the docno→docid lookup, which is the one real implementation subtlety).
- Thread-safety: a Lucene `IndexReader`/`IndexSearcher` is safe for concurrent search, so a single
  shared service instance serves all requests; no per-request reopen.

## 3. HTTP endpoints (the wire contract — agree these with the Cottontail consumer, keep minimal)

- **`POST /search`** — request `{ "query": string, "count": int, "summaries": bool }`
  (`summaries` optional, **default false**). Response
  `{ "results": [ { "docno": string, "score": number, "summary": string? } ] }`.
  `summary` present only when `summaries=true`. A **malformed query** → HTTP **400** with
  `{ "error": string }` carrying the `QueryParseException` message (the consumer bounces it to its LLM
  for self-correction, so keep the message informative — reuse the message surfaced by TASK-0015).
- **`POST /document`** — request `{ "docno": string }` → `{ "docno": string, "fulltext": string }`,
  or **404** `{ "error": "unknown docno" }` if not found. (Owner requirement, 2026-07-07: Lucindri
  serves full documents so a Lucindri-backed searcher needs no other system's copy of the corpus.)
- **`GET /healthz`** → `200 { "ok": true }` once the index is open (the consumer polls this on start).
- Bind **loopback (127.0.0.1) by default**; host/port configurable. Auth optional/deferred (dev use).
- Bad JSON / missing required field → 400 with a clear message. Unknown route → 404.

## 4. Summaries (opt-in) — `UnifiedHighlighter`

When `summaries=true`, produce a query-biased extractive summary per result with `UnifiedHighlighter` +
a sentence `BreakIterator` (base recipe in
[`docs/query-biased-summaries.md`](../docs/query-biased-summaries.md)): plain-text output (no `<b>`
markup), sentences joined by `" ... "`, `maxPassages` small (≈2) and request-tunable. **Default the
flag off** — the highlighter **re-analyzes each returned document**, a real per-result cost a
query-only caller (batch eval, a snippet-less UI) should not pay. The ISJ consumer sends
`summaries=true`.

**Getting the highlight terms from a (possibly complex) structured query — do this by walking the
parsed query tree, NOT by tokenizing the raw query string.** The highlighter needs the query's content
*terms*; Indri query classes don't implement `Query.visit()`, so they can't be enumerated directly. The
naive shortcut — `tokenizeString(analyzer, rawQueryString)` — is **wrong for structured queries**:
`tokenizeString` just runs the analyzer over whatever string it's given, so on
`#weight(0.2 "president" 0.8 "obama")` it emits the operator word `weight`, the numeric weights, etc.,
polluting the highlight terms. Instead, use the parser's own tree:

- `IndriQueryParser` parses to a `QueryParserQuery` tree: `QueryParserOperatorQuery` nodes
  (`getOperator()`, `getSubqueries()`) and `QueryParserTermQuery` leaves (`getTerm()`). The parser only
  ever builds term leaves from the text **inside quotes** (analyzer-applied), so **the leaf terms are
  exactly the analyzed content terms** — operators/weights/windows are structure, never leaves.
- Add a small public helper to `IndriQueryParser` (e.g. `List<String> queryTerms(String queryString)`)
  that parses to the tree (reuse the currently-private `parseQueryString`) and recurses collecting
  `QueryParserTermQuery.getTerm()`. Build the highlighter's `BooleanQuery` of `TermQuery`s from that
  set. This is complexity-proof: `#weight`, `#uwN`, `#syn`, `#not`, arbitrary nesting all reduce to
  their clean leaf terms.
- **Exclude terms reached through negation/rejection** — you never want to highlight a term the query
  is trying to *avoid*. Concretely, when recursing, do **not** collect leaf terms under:
  - `#not(...)` — a negated belief; its terms count *against* relevance.
  - the **condition** argument `C` of `#scoreifnot(C S)` / `#filrej(C S)` — documents are kept
    *because they lack* `C`, so its terms are absent from every result by construction (highlighting
    them is a guaranteed no-op — safe to skip, and clearer to skip explicitly). Still collect the
    scoring subquery `S`.

  Positive structure contributes its terms normally: `#combine`, `#weight`/`#wand`/`#wsum`, `#or`,
  `#max`, `#syn` (include **all** alternates as equal terms), the proximity operators' constituents,
  and the **require** condition `C` of `#scoreif(C S)` / `#filreq(C S)` (those terms *are* present in
  the results). Weights are irrelevant to the highlighter's own passage scorer — ignore them
  (optionally map to `BooleanClause` boosts; polish, not required).

  **Worked examples (Indri query → highlight-term set):**

  | Indri query | highlight terms | why |
  |---|---|---|
  | `#combine("president" #not("obama"))` | `president` | `obama` is under `#not` → excluded |
  | `#scoreifnot("parton" #combine("dolly cloning"))` | `dolly`, `cloning` | `parton` is the *rejected* condition → excluded (and would be a no-op anyway — reject means no result contains it); the scoring subquery `#combine("dolly cloning")` contributes its terms |
  | `#scoreif("climbing" #combine("rope" "anchor"))` | `climbing`, `rope`, `anchor` | `#scoreif` *requires* `climbing` (present in results) → included, along with the scoring terms |
  | `#weight(0.2 "president" 0.8 #syn("obama" "barack obama"))` | `president`, `obama`, `barack` | weights dropped; `#syn` alternates all kept; the quoted phrase `"barack obama"` tokenizes to `barack`, `obama` (deduped) |
  | `#combine(#uw8("mountain" "rescue") "avalanche")` | `mountain`, `rescue`, `avalanche` | proximity constituents become independent highlight terms (the term-vs-structure caveat below) |
- **Caveat (acceptable for v1):** term-based highlighting weights terms, not structure, so a proximity
  query (`#uwN`, `#band`) highlights its constituent terms independently, not their adjacency. Fine for
  readable snippets; proximity-faithful highlighting would need span extraction (the "ephemeral index"
  alternative in `docs/query-biased-summaries.md`) — out of scope here.

## 5. `/document` docno→docid lookup (the one real subtlety — resolve explicitly)

Per `CLAUDE.md`, `LuceneDocumentWriter` stores **and tokenizes** every field
(`DOCS_AND_FREQS_AND_POSITIONS`), so `externalId` is analyzed — a docno like `shard_00000_5` is not
cleanly exact-matchable with a single `TermQuery`. Two ways to make `/document` an O(1) exact lookup;
**pick one and record the choice in the progress log:**

- **(A, recommended) Index `externalId` as a non-analyzed keyword field** (a `StringField` — indexed,
  not tokenized, stored), in addition to / instead of its current tokenized form, so
  `new TermQuery(new Term("externalId", docno))` finds the doc exactly. This is a small
  `LucindriIndexer` change (the writer / the climbmix parser's field options) and requires reindexing —
  which is acceptable since the corpus is being (re)indexed into Lucindri for this work anyway. Cleanest
  and O(1) with no extra RAM. **Note:** the searcher's TREC output reads `externalId` as a *stored*
  value (`doc.get(...)`), which is unaffected by making it non-analyzed, so batch behavior is unchanged.
- **(B, no-reindex fallback) Build a `docno → Lucene docid` map at service startup** by scanning stored
  `externalId` values across the reader. O(N) startup time and RAM (~a string + int per doc), which is
  fine for the dev-scale index but heavy at full corpus scale. Use only if a reindex is undesirable.

## 8. Consumer conformance harness (provided) — the "answers the client correctly" gate

The §7 `mvn test` suite checks the implementation from the inside. This section is the **black-box
contract check**: a self-contained script the implementer runs **against a running server**, with no
consumer (Cottontail) code, to confirm the wire contract the ISJ adapter depends on. **The server is
"done for the consumer" when this passes.** It is corpus-agnostic — it discovers a real docno from a
search and reuses it — so it needs only a host/port and one Indri query that matches ≥1 document in
the served index.

**Invariants it enforces (the contract, plainly):**

1. `GET /healthz` → 200, body `{"ok": true}`.
2. `POST /search {query, count}` → 200, `{"results":[...]}`; each result has a **non-empty string
   `docno`** and a **numeric `score`**; `len(results) ≤ count`.
3. Results are in **rank order** — scores **non-increasing** (Indri scores are negative
   log-probabilities, best-first; do not assume positivity).
4. **Determinism** — the same request twice yields identical `(docno, score)` lists.
5. `summaries` omitted/false → **no `summary` field**; `summaries=true` → each result has a **non-empty
   string `summary`**.
6. A **malformed query** → **400** with a non-empty `{"error": ...}` message (the consumer bounces it
   to its LLM, so the message must be present).
7. A request **missing `query`** → 400.
8. `POST /document {docno}` for a docno returned by `/search` → 200, `{"docno","fulltext"}` with the
   **docno echoed** and **non-empty `fulltext`** — this is the ISJ judge's full-document read path.
9. `POST /document` for an unknown docno → **404**.
10. Cross-endpoint invariant (implied by 2+8): **every docno `/search` returns is resolvable via
    `/document`.**

**The harness (`scripts/conformance.py` or similar — stdlib only, no dependencies):**

```python
#!/usr/bin/env python3
"""Lucindri server conformance harness (TASK-0019): black-box checks that a RUNNING
server satisfies the ISJ/Cottontail wire contract. Stdlib only.

  python3 conformance.py --port 8080 --query '#combine("mountain")'

--query must be a VALID Indri query that matches >=1 document in the served index (the
implementer picks one for their test index). Use an explicit operator with quoted text,
e.g. #combine("mountain"), and shell-quote the whole value as shown. Exits non-zero on
any failure.
"""
import argparse, json, sys, urllib.request, urllib.error

def req(base, path, body=None, method="GET"):
    data = json.dumps(body).encode() if body is not None else None
    r = urllib.request.Request(base + path, data=data, method=method,
                               headers={"Content-Type": "application/json"})
    try:
        with urllib.request.urlopen(r, timeout=120) as resp:
            return resp.status, json.loads(resp.read() or b"null")
    except urllib.error.HTTPError as e:
        try: return e.code, json.loads(e.read() or b"null")
        except Exception: return e.code, None

RESULTS = []
def check(name, ok, detail=""):
    RESULTS.append(ok)
    print(f"[{'PASS' if ok else 'FAIL'}] {name}" + (f"  -- {detail}" if detail and not ok else ""))

def finish():
    n, p = len(RESULTS), sum(RESULTS)
    print(f"\n{p}/{n} checks passed")
    sys.exit(0 if p == n else 1)

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--host", default="127.0.0.1")
    ap.add_argument("--port", type=int, required=True)
    ap.add_argument("--query", required=True,
                    help="a valid Indri query (text quoted) matching >=1 doc in the served index")
    a = ap.parse_args()
    base = f"http://{a.host}:{a.port}"

    st, b = req(base, "/healthz")
    check("GET /healthz -> 200 {ok:true}", st == 200 and isinstance(b, dict) and b.get("ok") is True, f"{st} {b}")

    st, b = req(base, "/search", {"query": a.query, "count": 5}, "POST")
    res = b.get("results") if isinstance(b, dict) else None
    ok = st == 200 and isinstance(res, list) and len(res) >= 1
    check("POST /search -> 200 with >=1 result", ok, f"{st} {b}")
    if not ok:
        print("cannot continue without a search result; pick a --query that matches your index")
        finish()
    first = res[0]
    check("result.docno is a non-empty string", isinstance(first.get("docno"), str) and first["docno"] != "", f"{first}")
    check("result.score is numeric", isinstance(first.get("score"), (int, float)) and not isinstance(first.get("score"), bool), f"{first}")
    check("no summary when not requested", first.get("summary") in (None,) or "summary" not in first, f"{first}")
    check("count respected (<=5)", len(res) <= 5, f"got {len(res)}")
    scores = [r.get("score") for r in res]
    check("results non-increasing by score", all(scores[i] >= scores[i+1] for i in range(len(scores)-1)), f"{scores}")

    st2, b2 = req(base, "/search", {"query": a.query, "count": 5}, "POST")
    d1 = [(r["docno"], r["score"]) for r in res]
    d2 = [(r["docno"], r["score"]) for r in (b2.get("results") if isinstance(b2, dict) else [])]
    check("same query -> identical results (determinism)", st2 == 200 and d1 == d2, "results differ")

    st, b = req(base, "/search", {"query": a.query, "count": 3, "summaries": True}, "POST")
    top = (b.get("results") or [{}])[0] if isinstance(b, dict) else {}
    check("summaries=true -> non-empty summary", isinstance(top.get("summary"), str) and top.get("summary") != "", f"{st} {b}")

    st, b = req(base, "/search", {"query": "#combine(", "count": 5}, "POST")
    check("malformed query -> 400 + error message", st == 400 and isinstance(b, dict) and bool(b.get("error")), f"{st} {b}")

    st, b = req(base, "/search", {"count": 5}, "POST")
    check("missing query -> 400", st == 400, f"{st} {b}")

    st, b = req(base, "/document", {"docno": first["docno"]}, "POST")
    ok = (st == 200 and isinstance(b, dict) and b.get("docno") == first["docno"]
          and isinstance(b.get("fulltext"), str) and b["fulltext"] != "")
    check("POST /document(known docno) -> 200, docno echoed, non-empty fulltext", ok, f"{st} {b}")

    st, b = req(base, "/document", {"docno": "__definitely_not_a_docno__"}, "POST")
    check("POST /document(unknown) -> 404", st == 404, f"{st} {b}")

    finish()

if __name__ == "__main__":
    main()
```

Ship this script in the repo (e.g. under `scripts/`) so the acceptance step is reproducible:
`python3 scripts/conformance.py --port <p> --query '#combine("<a matching term>")'` must print all PASS. It is
the objective definition of "the server responds correctly to the consumer's requests."

## 9. Non-goals

- Not a public/production service (loopback dev target; auth deferred).
- No change to the query language, scoring model, or index format **except** the optional keyword
  `externalId` field in §5(A).
- Not implementing the consuming agent (that is the Cottontail side).
