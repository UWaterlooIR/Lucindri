# TASK-0019 — Query-time stopword removal as an explicit operator (`#stop(...)`) — future discussion

- **Status:** Draft
- **Created:** 2026-07-06
- **Owner:** (unassigned)
- **Summary:** TASK-0016 removes the searcher's user-configurable query-time analysis parameters
  (`<stemmer>`/`<removeStopwords>`/`<ignoreCase>`): the query analyzer is read from the index so that
  query tokenization always matches index tokenization. That deliberately also removes the one
  *legitimate* use those knobs had — asking for stopwords to be dropped from a **query** even though
  the **index** kept them. This task is to discuss and design bringing that capability back as an
  explicit query-language construct, e.g. `#stop("William to be or not to be")`, in a future feature
  release. Discussion only for now; no direction is decided.

## Motivation (owner, 2026-07-06)

> Currently the query time analyzer separately allows stop word removal via
> `<removeStopwords>true</removeStopwords>`. I think this should be retained in some form in a future
> feature release. For example, we could provide a means to perform stop word removal at query time,
> e.g. `#stop( "William to be or not to be")`.

Context that makes this useful: an index built **with stopwords kept** (`removeStopwords=false` — the
configuration TASK-0009 identified as the Indri length-parity workaround, and how the climbmix index
is being built) tokenizes queries with stopwords kept too, post-TASK-0016. Stopwords in a `#combine`
drag every document's belief around for terms that carry little signal. `#stop(...)` would let the
query author opt into dropping them per span, without any index/query tokenization mismatch — the text
is still tokenized *exactly* as the index was; tokens are only *removed afterward*.

When the index itself was built with stopword removal, the (index-derived) query analyzer already
stops — `#stop` would be a harmless no-op there.

## Proposed shape (starting point, not decided)

`#stop(...)` as a third **splice** construct, alongside TASK-0016's two:

| construct | analysis | splice |
|---|---|---|
| `"..."` / bare chunk | index analyzer | all produced tokens |
| `#token(...)` | none (verbatim) | the given tokens |
| `#stop(...)` | index analyzer, **then drop stopwords** | surviving tokens |

`#stop("William to be or not to be")` → analyze with the index's analyzer → drop stop tokens →
splice (e.g. just `william`). Zero survivors = legal empty splice, same convention as `"the of a"`.

## Questions to settle before implementation

1. **Which stoplist?** The analyzer module's built-in English list (what `removeStopwords=true` uses
   today) is the obvious default. Should the list be configurable (per-index metadata? per-query?),
   or fixed? Note Indri parity concerns from TASK-0009 (Indri's own stoplist is whatever the user's
   `<stopper>` block said — there is no single canonical list).
2. **Stemming order.** The current analyzer chain applies stopping at a fixed point relative to
   stemming/lowercasing. `#stop` must reproduce the same net effect as `removeStopwords=true` for
   the same input, or it becomes a third tokenization behavior. Probably implement by running the
   index analyzer chain with the stop filter enabled, not by post-filtering tokens against a list —
   this needs a look at `EnglishAnalyzerConfigurable`'s filter order.
3. **Inside proximity operators.** `#1(#stop(...))` changes the token sequence (and hence the
   phrase). Legal-but-documented, or discouraged? (Indri's stopping inside windows is exactly the
   `#1("u" "s" "a")` pain point documented in TASK-0016's reference section.)
4. **Interaction with `#token`.** None expected (`#token` bypasses all processing, including `#stop` —
   nesting them should probably be a syntax error), but state it.
5. **Is an operator the right vehicle at all?** Alternatives: a per-query flag in the query XML
   (weaker: whole-query granularity, and TASK-0016 just removed per-query analysis knobs on purpose);
   or leaving it to the wrapper/agent to pre-strip stopwords (weak: the wrapper would need the
   stoplist + analyzer, recreating the mismatch problem).

## Dependencies / sequencing

- Blocked on **TASK-0016** landing (it defines the splice model, the string-literal grammar, and the
  index-derived analyzer this builds on).
- Related: **TASK-0009** (whether removed stopwords count toward document length — a scoring
  decision) is orthogonal but shares the "how do stopwords behave" theme; decide them consistently.
