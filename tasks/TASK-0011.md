# TASK-0011 — Large-scale differential fuzzing (Lucindri vs C++ Indri) on a realistic integer collection

- **Status:** Ready
- **Created:** 2026-07-04
- **Owner:** (unassigned)
- **Summary:** Build a *realistic* integer collection by replacing each analyzed word in a real
  corpus (LATimes, or a subset of `t45minusCR`) with a unique integer id — preserving term counts and
  positions — so tokenization/stemming/stopwords are neutralized but the term distribution matches
  real English. Then **fuzz a large variety of legal Indri queries** and diff per-document scores
  between Lucindri and C++ Indri to surface any remaining scoring/query-semantics divergences.

## Why

TASK-0010 verified operators on tiny toy collections (C1/C2). This scales it up: a realistic
distribution (Zipfian term frequencies, real doc lengths, real co-occurrence and proximity patterns)
exercises the scorers the way real queries do, while **integers keep the input identical across
engines** (the tokenizer/stemmer/stopword residuals from TASK-0008 are removed by construction, so
*any* divergence is a real bug — not analysis). Random legal queries reach corners hand-written tests
miss. This is differential testing: Indri is the reference oracle, but per TASK-0010 we also sanity
-check surprising results against hand computation / internal consistency (Indri is not infallible).

## Build the integer collection (via Indri's tokenization + forward index)

Let Indri do the tokenization and read the tokens back from its **forward index** (document vectors),
keeping **all** tokens (no stopword removal).

1. Choose a source: **LATimes** (~130K docs) or a tractable subset of `t45minusCR` (e.g. 10k–50k docs).
2. Index it in Indri with **no stopper** (keep all tokens) and Krovetz stemming (or none — the integer
   output is stem-invariant either way): `<stemmer><name>krovetz</name></stemmer>`, no `<stopper>`.
3. Build a **global** `term → integer id` map from `dumpindex <idx> v` (the vocabulary; assign ids by
   descending `cf` so common words get small ids, Zipfian like a real index — id values are just
   labels). Note: `dumpindex dv`'s numeric id column is **document-local**, so map by the term
   **string** (3rd column), not that id.
4. For each document, `dumpindex <idx> dv <docid>` → the term strings in position order (all tokens,
   incl. what would be stopwords). Emit trectext with `<TEXT>` = those tokens' global integer ids:
   `<DOC><DOCNO>orig-docno</DOCNO><TEXT>id id id …</TEXT></DOC>`. Order = positions, repetition = tf.
   (dv-per-doc is O(numDocs) dumpindex calls — fine for a subset; use the Indri API for full scale.)
5. Index the integer corpus in **both** engines with analysis disabled so the integers pass through
   unchanged: Indri `word` tokenizer + no stemmer + no stopper; Lucindri whitespace/no-op tokenizer,
   `stemmer=none` (or kstem — a no-op on ints), `removeStopwords=false`. Integers have no punctuation,
   so both tokenize identically, and since all tokens are kept both agree on `|d|`/`|C|`.
6. **Verify Phase-0 alignment** (must be exact before comparing): `numDocs`, `|C|`, per-id `cf`/`df`,
   per-doc `|d|`, and a spot-checked id's positions match between engines. Because all tokens are kept
   (no stopword removal), document lengths are the full realistic lengths — this also exercises the
   distribution that TASK-0009 (stopword-in-`|d|`) concerns, though here both engines see identical
   integers so they agree.

## Fuzz legal Indri queries

Generate N random queries (seeded, reproducible) from a grammar that respects Indri's rules:

- **Leaves:** integer ids, biased toward frequent ids (so results are non-empty) with some rare/OOV.
- **Belief operators** (operands may be anything): `#combine`, `#weight`/`#wsum` (random positive
  weights), `#or`, `#not` (nested), `#max`, `#syn`, plus nesting to a random depth.
- **Proximity operators** (operands must be terms / proximity / `#syn` — NOT belief operators):
  `#1`/`#odN`, `#uwN` (random N), `#band`; include compound operands (`#syn(...)`, nested windows).
- **Filters:** `#filreq`/`#filrej` (Indri) ↔ `#scoreif`/`#scoreifnot` (Lucindri) — emit the correct
  name per engine; `(condition scored)` argument order.
- Enforce the nesting constraints so every generated query is legal in both engines (avoid the known
  "belief operand inside a proximity" error). Emit each query in **both dialects** (operator-name map).
- Sweep smoothing too: Dirichlet at a few μ, and Jelinek–Mercer at a few λ.

## Differential harness

- Run each query on both engines (`IndriRunQuery` vs `IndriSearch`, `trecFormat`, `count` ≥ a few k).
- Compare **per-document scores** (tolerance ~1e-3 on the shared docs) and the retrieved **sets**
  (overlap@k). Flag any query with a score delta above tolerance or a set mismatch.
- For each divergence: **auto-minimize** (shrink the query and/or restrict to a few docs until it
  still diverges), classify, and check the smaller case against hand computation. Then fix (with a
  regression test, per the testing policy) or catalog with the minimal repro.
- Report: number of queries run, % exact-match, and a ranked list of distinct divergence signatures.

## Deliverables

- `scripts/trec-comparison/build_integer_corpus.*` — real corpus → integer trectext + `term↔id` map.
- `scripts/trec-comparison/fuzz_queries.*` — seeded legal-Indri-query generator (both dialects).
- `scripts/trec-comparison/diff_fuzz.sh` — run both engines, diff scores/sets, minimize divergences.
- Findings appended to `docs/lucindri-vs-indri-scores.md`; any bug fixes ship with regression tests
  (reuse the count/score-level single-doc technique where possible).

## Acceptance criteria

- A realistic integer collection built and Phase-0-aligned between the engines (exact `|C|`/`cf`/`|d|`).
- ≥ a few thousand fuzzed legal queries run through both engines with a reported exact-match rate.
- Every distinct divergence signature is root-caused to a minimal repro and either fixed (with a
  regression test) or cataloged as a known difference (with rationale, e.g. an Indri quirk).
- Reproducible from a fixed seed.
