# TASK-0009 — Decide: should document length count removed stopwords (Indri parity)?

- **Status:** Draft
- **Created:** 2026-07-04
- **Owner:** (unassigned)
- **Summary:** Indri counts removed stopwords in the document length used for Dirichlet smoothing;
  Lucindri does not. Decide whether to match Indri or accept the difference.

## Background

Discovered during TASK-0008 (see `docs/trec-comparison.md`) using a tokenizer-free integer
collection, so this is *not* a tokenizer artifact. With stopword removal enabled (the normal
configuration), the two engines assign different document lengths to the same document:

Document `"5 the 7 and 5 9 the 7"` — 8 raw tokens, 5 content tokens, 3 stopwords (`the`, `and`):

| engine | document length reported |
|---|---|
| C++ Indri (`dumpindex … s`, total terms) | **8** (counts the removed stopword positions) |
| Lucindri (`sumTotalTermFreq(fulltext)`) | **5** (indexed content tokens only) |

Both engines *do* preserve the position gaps for removed stopwords (so proximity is unaffected —
verified), but they disagree on the length `|d|`.

## Why it matters

Dirichlet smoothing scores every term as `log( (tf + μ·p(w|C)) / (|d| + μ) )`. `|d|` is the
document length. Because Indri's `|d|` includes stopwords and Lucindri's does not, **every score
differs systematically whenever a document contains stopwords** — i.e. essentially always on real
text. In the TASK-0008 comparison this is one of the reasons term-only agreement was ~0.72 rather
than ~1.0 (the other being the tokenizer, which was accepted and catalogued). It is independent of,
and additive to, the tokenizer difference.

Note the collection length `|C|` is likewise affected (Σ|d|), which shifts `p(w|C)` as well.

## The decision

1. **Match Indri** — make Lucindri's document length include removed-stopword positions. This is
   the faithful choice for Indri parity but is non-trivial in Lucene: `sumTotalTermFreq` /
   norms count only indexed tokens. Options to investigate:
   - Index stopwords as positioned but effectively non-scoring tokens (keep them in the postings so
     length counts them) — changes the vocabulary and `|C|`, and would need care so they never match
     queries. Likely undesirable.
   - Store a separate per-document "raw length" field (pre-stopword token count) and have the
     Dirichlet similarity use it instead of the Lucene norm. Requires a custom length source in the
     similarity/collection-statistics path.
   - Disable index-time stopword removal entirely and rely on query-side stopping only (Indri
     effectively keeps stopword *positions*; it removes them from *matching* but not from *length*).
     Investigate whether this reproduces Indri's `|d|` and `|C|` exactly.
2. **Accept and catalog** — document the difference (like the tokenizer) as a known,
   quantified Category-B divergence and move on. Cheapest; keeps Lucindri's simpler model.

## Acceptance criteria

- A written decision (match vs. accept) with rationale, recorded here and in
  `docs/lucindri-vs-indri.md`.
- If "match": an implementation + a regression test proving `|d|` (and single-term Dirichlet score)
  equals Indri's on a stopword-bearing document; re-measure TASK-0008 term-only agreement.
- If "accept": a catalog entry quantifying the effect (e.g. the term-only overlap delta).
- **Either way, update `docs/indri-query-language.md` §6A** (the Indri-vs-Lucindri differences guide),
  which currently documents this as an open decision: record the outcome, and if "match" was
  implemented, change §6A to state that Lucindri's `|d|`/`|C|` now include removed-stopword positions.

## Update 2026-07-04 — measured impact + a config-only path

Query-time analysis is now configurable (`<removeStopwords>`/`<stemmer>`/`<ignoreCase>` in the query
params), which enables a clean **keep-all-tokens** comparison: index both engines with no stopword
removal (Indri: no `<stopper>`; Lucindri: `removeStopwords=false`) and stop neither the docs nor the
queries. Results (`docs/trec-comparison.md` §7b):

- Collection length aligns to **0.66 %** (Indri 253.37M vs Lucindri 251.70M; residual = tokenizer),
  vs the ~154M-vs-253M gap when Lucindri removed stopwords.
- **Term-only agreement rose overlap@10 0.716 → 0.790, RBO 0.705 → 0.798, top-1 33 → 40/50.** So the
  doc-length gap was a real contributor; the remaining term-only gap is essentially the tokenizer.
- Full-topic agreement barely moved (proximity backgrounds still dominate).

Implication for the decision: "keep all tokens on both sides" is a **configuration-only** way to
neutralize the doc-length divergence (no code change) — at the cost of stopwords being searchable.
The stricter "match Indri exactly" (stopwords count toward `|d|` but stay non-searchable, as Indri's
OOV does) still needs the code work described above. Decide between: (a) accept, (b) recommend the
keep-all-tokens config for Indri-comparison work, or (c) implement true OOV-length parity.

## Note 2026-07-05 — the dead `IndriSimilarity.computeNorm` is the natural seam for "match Indri" (from TASK-0012)

While scoping [[TASK-0012]] (exact document length) we traced `IndriSimilarity.computeNorm`
(`LucindriSearcher/.../similarities/IndriSimilarity.java:172`). It is directly relevant to **this**
decision:

- Its length branch uses **`state.getPosition() - state.getNumOverlap()`**, i.e. it counts **position
  slots** — which **includes removed-stopword gaps** — and then adds **`+1`**. That is *closer to Indri's
  `|d|`* (stopwords counted) than the length actually stored today. In other words, someone already wrote
  the "count stopword positions" logic here; it just isn't wired into the live pipeline.
- **It does not run in the standalone pipeline.** The indexer binds `LMDirichletSimilarity`
  (`LuceneDocumentWriter`), whose `SimilarityBase.computeNorm` stores `SmallFloat(state.getLength() -
  numOverlap)` = indexed-token count, **no gaps, no `+1`**. That is why Lucindri's `|d|` = 5 (not 8) in the
  probe above.
- **It *is* reachable via Solr:** `solr/IndriDirichletSimilarityFactory` supplies `IndriDirichletSimilarity`
  as the schema similarity, and Solr computes norms with it at index time. So changing this method's length
  logic (dropping the `+1`, or the `getPosition` vs `getLength` choice) **changes Solr-hosted index norms**
  — which is exactly why the `+1` removal was deferred **here** rather than done ad hoc in TASK-0012.

**Implication for the decision:** if this task chooses **"match Indri"** via a stored per-doc length (the
"separate raw-length field" option in §36 / the DocValues seam TASK-0012 builds), the count should be the
**position-based length including stopword gaps, with no `+1`** — and this `computeNorm` is the reference
for the intended-but-unwired semantics. TASK-0012 (for its narrower goal of de-quantizing the *current*
length) deliberately counts only emitted tokens and leaves the stopword question here; if TASK-0009 later
says "count gaps," the exact-length count and this `computeNorm` should be reconciled together. For now
TASK-0012 only adds an **orienting comment** to `computeNorm` (no behavior change).

## Finding 2026-07-05 — the reference (Indri **5.21**) always counts stopwords; there is no `countStopWords` knob in it

The Lemur docs (`lemurproject.org/lemur/indexing.php`) describe a **`countStopWords`** parameter — *"If true,
count stopwords in document length."* We checked **our** reference/oracle: the git clone at
`/ssd-8TB/git-repos/indri` is `Indri 5.21` (`configure.ac: AC_INIT([Indri],[5.21])`,
`indri-platform.h: "Indri release 5.21"`), the **same version** as the installed oracle binary at
`/ssd-8TB/installs/indri-5.21` we measured `|d|=8` against. **Neither the 5.21 source nor the compiled
`IndriBuildIndex` contains `countStopWords` (any casing).** So that configurable flag is **not** in 5.21;
it belongs to a different/newer Indri (or the older Lemur `BuildIndex` app). In 5.21 the behavior is
**hardcoded ON**.

Proof in `src/MemoryIndex.cpp` (`addDocument`):
- A stopword is nulled by `StopperTransformation` (`terms[i]=0`) but its **slot stays** in the terms vector.
  In `addDocument`: `if(!word||*word==0){ _termList.addTerm(0); continue; }` — the position is kept, but
  `indexedTerms` is **not** incremented.
- `_corpusStatistics.totalTerms += words.size();` — the **collection length `|C|` counts every position,
  stopwords included.**
- `_writeDocumentStatistics(offset, byteLength, indexedTerms, int(words.size()), uniqueTerms);` stores **two**
  per-doc lengths: `indexedLength=indexedTerms` (no stopwords) **and** `totalLength=words.size()` (with
  stopwords). `documentLength()` **returns `totalLength`**, and the Dirichlet scorer's `contextSize` (`|d|`)
  is fed from it.

**Consequences for this decision:**
- "Match Indri" is unambiguous and two-sided: count stopword positions in **both** `|d|` **and** `|C|`. It is
  not configurable in 5.21 — it is unconditionally true.
- This confirms the [[TASK-0012]] note above: `IndriSimilarity.computeNorm`'s `getPosition()-numOverlap+1`
  reproduces Indri's `totalLength` (= `words.size()`), i.e. it was an unwired attempt at exactly this parity.
  (It also corrects an earlier guess that the `+1` was a "never-store-zero"/smoothing guard — with
  `getPosition()` it is the position-index→slot-count conversion that yields Indri's stopword-inclusive `|d|`.)

## Design 2026-07-05 — feasibility of a configurable `countStopwordsInLength` index flag

Owner asked: can our indexer be configured with a count-stopwords flag; feasible on **both** the Lucene-norm
and the exact-doc-length ([[TASK-0012]]) paths; and what is the storage/performance impact? **Yes**, and the
cost is dominated by the length path, not by the stopword accounting. Two quantities must change together (or
`|d|` and `|C|` go inconsistent, which is worse than today):

**A. Per-doc `|d|` — feasible on both paths:**
- **Norm path (1-byte norm).** The norm is set by the *index-time* similarity's `computeNorm`. Today that's
  stock `LMDirichletSimilarity` → `SimilarityBase.computeNorm = SmallFloat(getLength()-numOverlap)` (content
  count), and it is `final` (can't subclass to change). `IndriSimilarity.computeNorm` already returns
  `getPosition()-numOverlap+1` = Indri's slot count, so: when the flag is on (and `exactDocumentLength` off),
  bind `IndriDirichletSimilarity` (not `LMDirichletSimilarity`) in `LuceneDocumentWriter`. **Cost: 0 extra
  storage** (still 1 byte), **0 extra index passes** (`getPosition()` is already tracked in the indexing
  `FieldInvertState`). Downsides: still SmallFloat-quantized, and **undercounts trailing stopwords** (see
  caveats).
- **Exact path (`_len` DocValues).** Change TASK-0012's `countTokens` to count with a **no-stopword** analysis
  pass (tokenizer+lowercase+stemmer, no `StopFilter`) = the raw token count = Indri's `words.size()`,
  **including trailing stopwords**. Same one extra pass TASK-0012 already pays, different analyzer.
  **Cost: same ~2 B/doc** as TASK-0012; accurate incl. trailing.

**B. Collection `|C|` — the one shared new piece (small):** both paths need `|C|` overridden (it can't stay
`sumTotalTermFreq`, which excludes stopwords). Accumulate `Σ` stopword-inclusive length during indexing and
store one number per field in the **index commit `userData`** (`IndexWriter` commit metadata; a
`Map<String,String>`); `IndriIndexSearcher.collectionStatistics` reads it and substitutes it for
`sumTotalTermFreq`. **No extra documents, no doc-count pollution, a few bytes, and arguably *faster* than
today** (read one number vs. summing per-leaf stats). Avoid the vestigial `document_total_count` /
`_totallen` approach (`IndriConstants`, commented `writeTotalDocLens`) — it writes an extra `Document` that
inflates `maxDoc`/`docCount` and can match queries (probably why it was disabled). A simpler-but-slower
alternative on the exact path only: sum the `_len` DocValues at query startup (O(N) scan; adds to the ~27s
full-scale stat cost).

**Impact summary:**

| combo | index storage | index time | query time |
|---|---|---|---|
| norm + countStopwords | ~0 (1-byte norm) + a few bytes commitData | ~0 (reuses `getPosition()`) | negligible (faster `|C|`) |
| exact + countStopwords | ~2 B/doc (`_len`, = TASK-0012) + commitData | 1 analysis pass (= TASK-0012, no-stop variant) | negligible |

**Caveats to fix the acceptance target before implementing:**
- **Trailing stopwords.** Lucene's `StopFilter` drops a *final* stopword with no following token to carry the
  gap, so the norm/`getPosition` route undercounts docs ending in stopwords; Indri keeps them. The **exact
  path's no-stop counting pass avoids this** — the concrete reason to prefer exact for true parity.
- **Tokenizer difference ([[TASK-0008]], accepted).** Even counting all tokens, Lucene's tokenizer splits
  some text differently from Indri (e.g. numbers), so `|d|`/`|C|` won't be *bit*-identical to Indri on real
  text; the stopword component (the systematic dominant part) does align.

**Flag matrix (all feasible), `exactDocumentLength × countStopwordsInLength`:**

| | content length | stopword-inclusive |
|---|---|---|
| **norm** | today | index-similarity swap; quantized; trailing undercount |
| **exact** | TASK-0012 | **best Indri parity** (exact + stopword-inclusive, incl. trailing) |

**Recommendation:** add `countStopwordsInLength` as a **separate** index-time flag (orthogonal to
`exactDocumentLength`; don't fold them together), default off = today's behavior. Support both paths for
completeness, but the intended "match Indri" combo is **exact + countStopwords** (only that avoids both the
quantization and the trailing-stopword undercount). The single shared new component is the commitData `|C|`
total. This makes option 1 ("Match Indri") the concrete design; option B (config-only keep-all-tokens,
§ "Update 2026-07-04") remains the zero-code alternative if stopwords being searchable is acceptable.
