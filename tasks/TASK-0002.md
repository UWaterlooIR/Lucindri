# TASK-0002 — Fix Lucindri term/proximity operator bugs (NPE, wrong results, position-0 miss)

- **Status:** Done
- **Created:** 2026-07-04
- **Owner:** Mark D. Smucker
- **Summary:** Lucindri's term/proximity operators (`#band`, `#N`, `#uwN`, `#syn`) fail in three
  ways: (1) a **belief** operand (`#or`/`#combine`) causes a **NullPointerException**; (2) **≥2
  compound term-op operands** (`#syn`/`#1`/`#uw`) give **wrong results** (usually 0); (3) ordered
  window `#N` **misses a phrase at document position 0**. Fix all three, with controlled
  regression tests built on the shared `searcher/testutil.TestIndex` fixture.

> indexer is unaffected. Bugs 1–2 were found rendering Cottontail/GCL cover queries in the Indri

> `TestIndex` fixture.

## 1. Background / where this bites

The natural Indri rendering of a multi-facet "cover" query puts a disjunctive facet inside a
proximity/Boolean-window operator. The correct idiom for a disjunctive facet **inside** a
proximity operator is `#syn` (which produces a merged position list), e.g.

```
#band( #syn(backpack hiker trekker) #syn(bear grizzly) #syn(food store) #syn(canister hanging) )
```

Both the buggy behaviors below strike exactly this shape, so positional cover queries are
currently unusable. The belief-composed alternative `#combine(#or(...) ...)` works correctly and
is the only reliable multi-facet path today.

## 2. The bugs

### Bug 1 — NullPointerException on a belief operand

Giving a proximity/term operator a **belief** operand throws:

```
java.lang.NullPointerException
  at org.lemurproject.lucindri.searcher.IndriBandWeight.createInvertedList(IndriBandWeight.java:33)
  at org.lemurproject.lucindri.searcher.IndriTermOpWeight.getProximityIterator(IndriTermOpWeight.java:86)
  at org.lemurproject.lucindri.searcher.IndriTermOpWeight.getScorer(IndriTermOpWeight.java:74)
  ...
```

**Reproducer** (against the 10% index, see §4):
`#band(#or(backpack hiker trekker) #or(bear grizzly) #or(food store) #or(canister hanging))`

**Root cause:** `IndriTermOpWeight.getScorer` (`IndriTermOpWeight.java:55-68`) builds the operand
iterator list like this:

```java
for (Weight w : weights) {
    Scorer scorer = w.scorer(context);
    if (scorer != null) {
        IndriDocAndPostingsIterator iterator = null;
        if (scorer.iterator() instanceof IndriTermOpEnum) {
            iterator = ((IndriTermOpEnum) scorer.iterator());
        } else if (scorer.iterator() instanceof PostingsEnum) {
            iterator = new IndriPostingsEnumWrapper((PostingsEnum) scorer.iterator());
        }
        iterators.add(iterator);   // <-- adds NULL when neither branch matched
    }
}
```

A belief operator's scorer (e.g. `IndriOrScorer`) exposes a disjunction iterator that is neither
`IndriTermOpEnum` nor `PostingsEnum`, so `iterator` stays `null` and a `null` is added to the
list. The `createInvertedList` methods then call `iterators.get(0).nextDoc()` → NPE.

### Bug 2 — wrong results (usually 0) with ≥2 compound term-op operands

`#band` (and, by shared design, almost certainly `#N`/`#uwN`) computes **incorrect** results when
two or more operands are themselves term-ops (`#syn`/`#1`/`#uw`/`#N`). Measured on the 10% index
(`count=20`), which makes the bug provable because a **broader** query returns **fewer** hits:

| query | hits | note |
|---|---:|---|
| `#syn(backpack hiker trekker)` (standalone) | 20 | `#syn` itself is fine |
| `#band(backpack bear food canister)` (4 plain terms) | 20 | plain-term `#band` is fine |
| `#band(#syn(backpack) bear)` (1 term-op + 1 plain) | 11 | suspect (unverified vs truth) |
| `#band(#syn(backpack) #syn(bear))` (2 term-ops) | **0** | **wrong** |
| `#band(#syn(backpack hiker trekker) #syn(bear grizzly) #syn(food store) #syn(canister hanging))` | **0** | **wrong** — strictly broader than the 4-plain-term query above, yet 0 < 20 |

A correct AND can never return fewer results for a strictly broader query, so this is a
correctness defect, not a data/granularity artifact.

**Root cause (hypothesis to confirm):** the window-merge in
`IndriBandWeight.createInvertedList` (`IndriBandWeight.java:56-99`) assumes each operand yields
**single-token** postings via `nextPosition()`/`endPosition()`/`freq()` like a raw
`PostingsEnum`. A `#syn`/proximity operand is an `IndriTermOpEnum` backed by a materialized
`IndriInvertedList` of **intervals** (`start != end`, its own `freq`/iteration semantics). The
`previousTermPosition` tracking and the `TreeMap` keyed by start position miscount against
interval-valued operands, so the "smallest window containing all operands" scan fails to find
windows. The same shared `getScorer` path feeds `IndriNearWeight` and `IndriWindowWeight`, so
`#N`/`#uwN` are very likely affected too — **verify and cover them.**

### Bug 3 — ordered window `#N` misses a phrase at document position 0

`#N` (ordered window, e.g. `#1(cat dog)`) fails to match an occurrence that begins at **position
0** of a document. Reproducer (found via the TASK-0004 `TestIndex` fixture): with docs
`d1="cat dog"` (phrase at pos 0), `d2="filler cat dog"` (phrase at pos 1), `d3="dog cat"`
(reversed), `#1(cat dog)` returns **only `d2`** — it misses `d1` and correctly skips `d3`.

**Root cause:** in `IndriNearWeight.createInvertedList`, the follower term's `nextStartPositions[j]`
starts at its array default `0`, and the inner advance loop is guarded by
`nextStartPositions[j] < nextEndPositions[j-1]`. When the first term is at position 0 its
end position is 0, so `0 < 0` is false, the follower is never advanced to its real position, and
the adjacency check fails. Effect: silent under-counting — every ordered-window occurrence that
starts at doc position 0 is missed. Rare on long docs, but real (and it's why the fixture's
smoke test keeps the phrase off position 0). Fix the initialization/guard so position-0 phrases
match; add a regression test that indexes a phrase at position 0.

## 3. Fixes

Three independent fixes; do all three.

### Fix A — Bug 1: reject (or support) belief operands, never NPE

Proximity/term operators are defined over **positions**; a belief operator (`#or`, `#combine`,
`#wand`, `#wsum`, `#max`, `#not`, `#wand`) has no position list, so it is not a valid operand of
`#band`/`#N`/`#uwN`/`#syn`. Indri itself disallows this. Minimum acceptable fix: in
`IndriTermOpWeight.getScorer`, when an operand's `scorer.iterator()` is neither `IndriTermOpEnum`
nor `PostingsEnum`, **throw a clear `IllegalArgumentException`** naming the operator and the
unsupported operand (instead of adding `null`). Do **not** silently drop the operand.

(Note the correct user idiom is `#syn`, not `#or`, for a disjunctive facet inside a proximity
operator — a good error message should hint at this.)

### Fix B — Bug 2: correctly consume `IndriTermOpEnum` operands

Make the window-merge in `IndriBandWeight.createInvertedList` (and the analogous
`IndriNearWeight`/`IndriWindowWeight` logic, if confirmed broken) correct when operands are
`IndriTermOpEnum` interval producers, not just token-level `PostingsEnum`. Options, pick per the
code:
- Normalize every operand to a uniform interval-iteration contract (`start`, `end`, `freq`,
  `nextPosition`/`endPosition`) so plain terms and term-ops are handled identically, then audit
  the merge loop against that contract; or
- Fix the specific assumptions (the `TreeMap` keyed by `start`, and the `previousTermPosition`
  "new term for this window" test) to account for `end != start` intervals and for an operand
  contributing overlapping intervals.

The result set for `#band(#syn A) …` must be a **superset** of the plain-term version and must
never shrink as a facet is broadened.

### Fix C — Bug 3: match ordered windows that begin at document position 0

In `IndriNearWeight.createInvertedList`, fix the follower-term initialization/guard so an ordered
window whose first term is at position 0 is not skipped — the inner advance loop must read the
follower's real position even when the leader's end position is 0. Preserve the existing distance
semantics. Check whether `IndriWindowWeight` (`#uwN`) has an analogous position-0 issue and fix it
if so. Add a regression test that indexes the phrase at document position 0 (§5 case 6).

## Implementation plan (ordered)

Test-first, using the shared `searcher/testutil.TestIndex` fixture ([[TASK-0004]]); where the
"correct" answer is not obvious by construction, pin it against the C++ Indri gold oracle
([[indri-cpp-reference]], `IndriBuildIndex`/`IndriRunQuery`). Fix simplest → deepest so each step
is independently verifiable. Note: searcher Surefire runs with assertions disabled (Indri scores
are negative log-probs that trip Lucene's `assert score >= 0`), so **assert on doc sets / rank
order, not raw scores.**

1. **Branch** off `master` (e.g. `fix/term-operator-bugs`).
2. **Bug 1 (NPE) — cheapest.** Failing test first: `#band(#or(a b) c)` must throw a clear
   `IllegalArgumentException` (use `assertThrows`), not NPE. Apply **Fix A** in
   `IndriTermOpWeight.getScorer` (reject an operand whose iterator is neither `IndriTermOpEnum`
   nor `PostingsEnum`). Green.
3. **Bug 3 (position-0) — localized.** Failing test first: docs with `a b` at position 0 and at
   position 1; `#1(a b)` must match **both**. Apply **Fix C** in `IndriNearWeight`; extend to
   `IndriWindowWeight` if it shares the issue. Green.
4. **Bug 2 (compound-operand miscount) — deepest.** First establish "correct": build a small
   hand-crafted corpus where `#band(#syn(a b) #syn(c d))` (and `#uwN`/`#N` variants) truth is
   unambiguous by construction; optionally confirm the expected set with C++ Indri on the same
   corpus. Failing tests for `#band`/`#uwN`/`#N` with ≥2 `#syn`/proximity operands (superset
   property + exact set). Apply **Fix B** in the `createInvertedList` window-merge(s). Green.
5. **Regression** (§5 cases 1, 5): plain-term `#band`/`#uwN`/`#N` and standalone `#1`/`#uw`/`#syn`
   stay correct.
6. **`mvn test`** in `LucindriSearcher` → green (per the CLAUDE.md testing policy).
7. **Rebuild** `LucindriSearcher-1.5-jar-with-dependencies.jar`.
8. **End-to-end confirm** on the 10% index (`climbmix_first654_10pct`): `#band(#or …)` errors
   clearly; `#band(#syn …)` and the 4-facet syn cover return non-empty, superset-correct sets;
   regression **hit counts** match [[climbmix-10pct-scout-index]] (TASK-0003 changed scores, not
   hit counts, so compare counts/sets, not score values).
9. **Close out:** set `Status: Done`, dated progress-log entry (which bugs fixed, before/after
   reproducer results), commit.
