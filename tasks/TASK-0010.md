# TASK-0010 — Systematic score-level conformance study (Lucindri vs C++ Indri) on an integer collection

- **Status:** In progress
- **Created:** 2026-07-04
- **Owner:** (unassigned)
- **Summary:** Using a tiny artificial collection of **integers** (no stopwords, no stemming, no
  tokenizer ambiguity), compare **per-document scores** between Lucindri and C++ Indri operator by
  operator across the Indri query language, then their interactions, to find exactly where the two
  scorers diverge — and verify both against the hand-computed Dirichlet values.

## Why integers

Integers eliminate every analysis confound identified in TASK-0008:
- **Tokenizer:** integers have no punctuation to split; Lucene `StandardTokenizer` and Indri's `word`
  tokenizer produce the identical single token. (Verified: positions match exactly.)
- **Stemmer:** integers are stem-invariant.
- **Stopwords:** the vocabulary contains none; index and query with `removeStopwords=false` on both.

Therefore **any score difference is a pure query-language / scoring-logic difference**, not analysis.
And because the collection is tiny and known, every expected score is analytically computable from the
Dirichlet formula, so each engine can be checked against ground truth, not just against each other.

## Ground-truth formula

Dirichlet-smoothed query-likelihood, per term, per document:

```
score(w, d) = log( ( tf(w,d) + mu * cf(w)/|C| ) / ( |d| + mu ) )
```

where `tf(w,d)` = term frequency in the doc, `cf(w)` = collection frequency, `|C|` = total tokens in
the collection, `|d|` = document length, `mu` = Dirichlet parameter. Belief operators combine these
per-term log-probabilities:
- `#combine(a b …)` = arithmetic mean of the child scores.
- `#weight(wa a wb b …)` = weighted mean `Σ wᵢ·sᵢ / Σ wᵢ`.
- `#wsum`, `#wand`, `#max`, `#or`, `#not`, `#syn`, proximity windows — each has its own rule
  (to be pinned down and checked).

For a document that does **not** contain a child, the child contributes its **background** score
(`tf = 0`). Getting these backgrounds right is where mismatches are expected.

## Method

1. Build one integer collection in **both** engines with matched settings: Krovetz/KStem (no-op on
   integers), `removeStopwords=false`, `ignoreCase=true`, Dirichlet `mu` set explicitly and equal.
2. **Verify statistics align exactly first** (Phase 0). If `|C|`, `cf`, `|d|`, `numDocs`, or a term's
   positions differ, stop — that is the first divergence and must be explained before scoring.
3. For each query, run it on both engines with `count >= numDocs`, extract the per-document score,
   and produce a table: `docno | analytic | Indri | Lucindri | Δ(I−L)`. Flag any `|Δ|` above a small
   tolerance (start `1e-3`; single-term scores are expected to match to ~5 decimals).
4. Work **one operator at a time in isolation**, then **interactions/nesting**. When a divergence is
   found, shrink to the smallest query/collection that still shows it, read the relevant scorer
   source (Lucindri classes; stock-Lucene Indri classes on GitHub), and record the root cause.

## Starter collection C1 (5 docs, vocab {10,20,30,40,50,90})

```
d1: 10 20 30
d2: 10 10 20
d3: 10 90 20
d4: 20 10
d5: 30 40 50 90 90
```

Derived stats (|C| = 16): cf(10)=5, cf(20)=4, cf(30)=2, cf(40)=1, cf(50)=1, cf(90)=3;
|d| = {d1:3, d2:3, d3:3, d4:2, d5:5}. Positions are known, so proximity windows are hand-checkable,
e.g. `#1(10 20)` occurs in d1 (10@0,20@1) and d2 (10@1,20@2), not d3 (gap), not d4 (reversed).
Design additional per-operator micro-collections as needed (e.g. to isolate background/`tf=0`
behavior, or window collection-frequency).

## Phased plan (work through individually, then interactions)

- **Phase 0 — Alignment.** numDocs, `|C|`, per-term `cf`/`df`, per-doc `|d|`, and a term's positions
  match exactly across engines. (Note the TASK-0009 finding: with stopword removal these would
  already differ; the integer collection sidesteps it, but keep `removeStopwords=false` to be safe.)
- **Phase 1 — Single term.** `#combine(10)` (and a bare term). Compare to analytic; sweep `mu`
  (e.g. 0, 1, 100, 2000, 10000). Establishes the scoring baseline. Also JM smoothing.
- **Phase 2 — Belief operators, isolated.** `#combine`, `#or`, `#not`, `#syn`, `#max`, `#wsum`,
  `#wand`, `#weight`. For each: hand-compute from child scores (incl. backgrounds for missing
  children) and compare both engines. Nail the exact combination rule and normalization each uses.
- **Phase 3 — Proximity, isolated.** `#1`/`#odN`, `#uwN`, `#band`. Verify the window's `tf`, `df`,
  `cf`, and its Dirichlet score (windows are terms with their own collection statistics). (`#uwN`
  width was fixed in TASK-0008; re-verify at score level.)
- **Phase 4 — Filters.** `#filreq`/`#filrej` (`#scoreif`/`#scoreifnot`), `#not` in context.
- **Phase 5 — Interactions / nesting.** term+proximity under `#combine`/`#weight`; compound operands
  inside windows (`#1(#syn(..) ..)`); multi-level nesting; boosts on operator children; the mixed
  term/proximity case that diverged in TASK-0008 (§5) — now re-examined at the score level with all
  analysis confounds removed, to see whether a residual scoring difference remains.

## Deliverables

- `scripts/trec-comparison/score_probe.sh` — run a query on both integer indexes and print the
  per-doc `analytic | Indri | Lucindri | Δ` table.
- `docs/lucindri-vs-indri-scores.md` — the conformance table per operator (match / diverge, with the
  root cause and smallest repro for each divergence found).
- A committed builder for collection C1 (and any micro-collections) so the study is reproducible.
- Any bugs found get their own fix + regression test (per the project testing policy), or are filed
  as follow-up tasks with the minimal repro.

## Acceptance criteria

- Every operator in Phases 1–4 has a score-level verdict vs Indri **and** vs the analytic formula,
  recorded in the conformance doc.
- Phase 5 interactions covered at least for the combinations used by real Indri topics
  (`#weight`/`#combine`/`#1`/`#uwN`), with any divergence root-caused to a specific scorer.
- Reproducible: a fresh checkout can rebuild the collection and regenerate the tables.
