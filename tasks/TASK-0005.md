## 1. Goal & scope

Lucindri is a **subset** of Indri. The goal is a triage-ready catalog of where Lucindri diverges
from Indri **within that subset**.

**In scope — defined by Lucindri's own claims in the repo `README.md`** ("Lucindri Query
Language" section), which is the authoritative statement of what Lucindri is designed to
implement. As listed there:
- Terms and field restriction (`term`, `term.field`).
- Belief operators: `#combine` (= `#and`), `#or`, `#not`, `#wand`, `#wsum`, `#max`, `#scoreif`
  (filter require), `#scoreifnot` (filter reject).
- Term operators: `#band`, `#N` (= `#nearN` / `#windowN`, ordered window), `#uwN` (unordered
  window), `#syn`.
- Smoothing rules: Dirichlet, Jelinek-Mercer.

(`docs/indri-query-language.md` is a **derived, annotated companion I wrote from reading the
code** — useful as an aid, but NOT the source of the claims. If the two ever disagree, the
README wins; note any README-vs-code-vs-my-guide mismatches as their own catalog rows. E.g.
`#weight` is recognized by the parser but is **not** claimed in the README — treat it as
not-claimed / out of scope unless we decide otherwise.)

**Out of scope** — anything Indri implements that the Lucindri README does **not** claim (do not
report as defects): wildcards (`term*`), `#weight`, `#wsyn`, `#any:field`, `#prior`,
passage/field-extent retrieval (`#combine[field]`, `#combine[passageN:M]`), numeric/date field
operators, two-stage smoothing, and any other Indri operator absent from the README list above.

## 2. Methodology

The core method: **same documents, same query, both engines, diff the results.**

### 2.1 Shared corpus, matched analysis
- Author a small **TREC-text** corpus (both engines index `trectext`: Lucindri via
  `documentFormat=trectext`, Indri via `<corpus><class>trecweb/trectext</class>`), so both see the
  same documents with the same `DOCNO`s (Lucindri stores it as `externalId`).
- **Align analysis so differences are query-semantics, not tokenization.** Lucindri hardcodes the
  query-time analyzer to **KStem + English stopwords + lowercase**; configure Indri to match:
  `<stemmer><name>krovetz</name></stemmer>` + the **same stopword list** + default lowercasing.
- **Isolate operator semantics from residual analysis differences:** for the operator tests, use
  tokens that are lowercase, non-stopword, and **stem-invariant under both KStem and Krovetz**
  (verify with a probe — KStem and Krovetz are the same family but not identical; we already know
  KStem leaves `dogs`/`mice` unstemmed). Catalog analysis differences **separately** (§3, Category B).

### 2.2 Run both engines to TREC output
- Lucindri: `IndriSearch` on a queries `.xml` (`<rule>dirichlet:2000</rule>`, `<trecFormat>true</trecFormat>`).
- Indri: `IndriRunQuery` on its parameter file (`<rule>method:dirichlet,mu:2000</rule>`,
  `<trecFormat>true</trecFormat>`, `<query><type>indri</type>...`). **Align smoothing parameters**
  (same μ / λ) — note the two engines use different `<rule>` spellings.
- Both emit `qid Q0 docno rank score run`. Normalize and diff.

### 2.3 What to compare (three dimensions, decreasing strictness of "must match")
1. **Retrieved doc set** (ignore rank/score): a mismatch is a **correctness** difference. This is
   the primary gate.
2. **Rank order**: differences here with matching sets indicate a **scoring/combination**
   divergence.
3. **Scores** (within a small tolerance): catalog deltas; small numeric drift may be acceptable,
   large/systematic deltas are a finding. (Lucindri's Dirichlet already matches the closed form and
   TASK-0003 corrected the collection length — so scores *should* be close; verify against Indri.)

### 2.4 Harness
Add `scripts/diff_indri.sh` (record-keeping; depends on the external Indri install, so it lives in
`scripts/`, not the JUnit suite): given a corpus + a query string + smoothing params, it builds
both indexes (once, cached), runs the query in both engines, and prints a side-by-side / diff of the
normalized TREC output. Reusable for ad-hoc checks and for regenerating the catalog.

## 3. Catalog (the deliverable)

Produce `docs/lucindri-vs-indri.md` — a triage table with one row per in-scope element:

| element | probe query | Indri behavior | Lucindri behavior | match? | severity | notes / follow-up task |

Group rows into:
- **Category A — operator semantics** (one or more probe queries per operator; the main body).
- **Category B — analysis/tokenization** (KStem vs Krovetz per a word list; stopword-list
  differences; hyphen/possessive handling).
- **Category C — scoring/collection model** (Dirichlet & JM scores on identical matched docs;
  collection-statistics parity).

**Severity rubric:**
- **Critical** — wrong/empty results, crash, or a silently-ignored operator (changes which
  documents are retrieved).
- **Major** — same doc set but materially different ranking (combination/weighting divergence).
- **Minor** — small score deltas, cosmetic output differences, or benign analysis quirks.

### 3.1 Seed the catalog with what we already know (re-verify each against Indri)
- `#scoreif` / `#scoreifnot` — Lucindri **does not filter** (silent no-op; degrades to `#combine`).
  Expect **Critical** vs Indri's `#filreq`/`#filrej` semantics. (See `docs/indri-query-language.md`.)
- `#wand` / `#weight` — Lucindri applies per-clause boosts but AND-combines (`IndriAndQuery`);
  compare ranking to Indri's weighted belief. Likely **Major** if weighting differs.
- `#band` / `#uwN` / `#N` with `#syn`/proximity operands and at document position 0 — **fixed in
  TASK-0002**; confirm they now **match** Indri (regression guard, expect no difference).
- Dirichlet/JM scores — **fixed in TASK-0003** (collection-length double-count); confirm scores now
  match Indri within tolerance.
- KStem vs Krovetz — Category B; e.g. `dogs`/`mice` unstemmed by KStem; catalog per a probe word set.

## 4. Acceptance criteria

1. Every **in-scope** element (§1) has at least one differential probe run in both engines and a
   corresponding catalog row in `docs/lucindri-vs-indri.md` with a severity.
2. `scripts/diff_indri.sh` reproduces any row (build-both, run-both, diff) from a corpus + query.
3. Known items in §3.1 are re-verified against Indri with current (post-TASK-0002/0003) Lucindri;
   the catalog states match/differ, not stale assumptions.
4. **Criticals are triaged:** each Critical/Major row names a follow-up fix task (create the
   `TASK-000N` stubs) or a rationale for accepting the difference.
5. Analysis is aligned (Krovetz + same stopwords) so Category-A differences are query-semantics,
   not tokenization; Category-B captures the analysis differences separately.

## 5. Out of scope

- Fixing the differences found — this task **catalogs and triages**; fixes are separate tasks
  (e.g. a `#scoreif` filtering fix). It may create those task stubs.
- Indri-only features (§1 "out of scope"): do not log them as Lucindri defects.
- Effectiveness/quality evaluation (relevance judgments, metrics) — this is a *behavioral
  conformance* comparison, not an IR-effectiveness study.
- Performance comparison — separate concern.

## 6. Notes / pointers

- Indri `<rule>` uses `method:dirichlet,mu:NNNN` / `method:linear,collectionLambda:X`; Lucindri
  uses `dirichlet:NNNN` / `jm:X`. Keep μ/λ identical across the two.
- Indri query text needs `<type>indri</type>`; Lucindri assumes Indri-language text.
- Doc-id parity: index the same TREC `DOCNO`s; Lucindri prints them from `externalId`.
- Use a corpus small enough to eyeball, plus a handful of docs that exercise each operator's edge
  (adjacency, unordered windows, disjunctive facets inside proximity, negation, filters).
- The installed Indri and its build notes: [[indri-cpp-reference]].
