# TASK-0017 — Differentially test field scoring vs C++ Indri (fielded docs + duplicate/repeated fields)

- **Status:** Deferred (2026-07-05)
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** All prior Lucindri-vs-Indri conformance work (TASK-0008/0010/0011) used a **single content
  field** (`fulltext`). We have **never** differentially tested **field-restricted scoring**
  (`word.field`), documents that carry **multiple named fields**, or documents with **repeated/duplicate
  fields of the same name** (e.g. several `<H1>` blocks indexed as field `h1`). We already have evidence
  that at least one behavior diverges: Lucindri concatenates repeated same-name fields with **no position
  gap**, so a phrase/window can match **across two disjoint field values**, whereas Indri (which models
  fields as extents) is expected **not** to count such a cross-field match. Build fielded test
  collections, index in both engines, and diff scores + match sets to establish ground truth and
  catalog/fix the divergences.

## What we already know (confirmed on the Lucindri side)

- **Field-restricted term scoring is field-scoped.** `dog.title` builds `IndriTermQuery(Term("title",
  "dog"))`; the weight uses `searcher.collectionStatistics("title")` and a `LeafSimScorer` over the
  **`title`** field's norm — so `tf`, `cf`, `|C|`, and `|d|` all come from the title field:
  `score = log((tf_title + μ·cf_title/|C_title|) / (|d_title| + μ))`. (Verified from code, not vs Indri.)
- **Repeated same-name fields are merged.** `LuceneDocumentWriter` adds one Lucene `Field` per
  `ParsedDocumentField`, so N `h1` values become N `Field`s of name `h1`; Lucene merges them into one
  inverted list. Effect: `tf` sums across the values, `|d_h1|` = total tokens across all values.
- **Position gap between values is 0.** `EnglishAnalyzerConfigurable` does not override
  `getPositionIncrementGap` (Lucene default 0), so repeated field values are concatenated with no
  positional separation → `#1(a b)` / `#uwN(...)` can match with `a` at the end of one value and `b` at
  the start of the next.

## What is UNKNOWN / to be measured (the Indri side)

The following were reasoned about but **not** confirmed against C++ Indri — this task measures them:

1. **Field-restricted `|d|`:** does Indri use the document's **field length** or its **total length** as
   `|d|` when scoring `dog.title`? And is the background `p(w|C)` the field's `cf/|C|` or the global one?
   (Lucindri uses field length + field collection stats; confirm Indri matches.)
2. **Repeated fields → length & tf:** does Indri sum `tf` and field length across repeated same-name
   extents the way Lucindri does?
3. **Cross-field proximity (the identified divergence):** for a doc with two disjoint `h1` values, does a
   phrase/window that spans the boundary match in Indri? Owner's expectation: **no** (extents are
   disjoint). Lucindri: **yes** (gap 0). Confirm and quantify.
4. **Field-restricted proximity:** does `#1(a b)` evaluated within a field only match inside a single
   extent in Indri, and does Lucindri have any equivalent restriction? (Lucindri's `term.field` is
   leaf-only; there is no `#combine[field](...)` extent operator — see TASK-0016 / support map.)

## Plan

1. **Neutralize the tokenizer, keep fields.** Reuse the integer-collection idea (TASK-0011): emit
   trectext documents whose tokens are integers, but now inside **field tags** — e.g.
   `<DOC><DOCNO>d1</DOCNO><TITLE>10 20</TITLE><BODY>30 10 40</BODY></DOC>`, and documents with
   **repeated** tags: `<H1>10 20</H1> … <H1>30 40</H1>`. Integers guarantee identical tokenization in
   both engines, so any score/match difference is field-handling logic.
2. **Index in both engines with matching field config.**
   - Indri: `IndriBuildIndex` with `<field><name>title</name></field>` (etc.) in the param.
   - Lucindri: `fieldNames=title,body,h1`, `contentTags=…`, `indexFullText=true`. **Note a setup
     hazard:** the current `TrecTextDocumentParser.gatherContent` *concatenates* all content tags into
     `fulltext`; verify how named fields (and repeated tags) actually map to `ParsedDocumentField`s, and
     that repeated tags produce multiple fields (not a pre-concatenated one). Fix the harness/parser
     path if needed so the Lucindri index truly has per-field / multi-valued fields.
3. **Phase 0 — align field stats.** Confirm both engines agree on per-field `|C_field|`, `cf_field`,
   `df_field`, and per-doc field length `|d_field|` (via `dumpindex` and a Lucindri stats probe) before
   comparing scores. Disagreement here localizes the issue to indexing, not scoring.
4. **Score/behavior diffs.** For controlled docs, compare Indri vs Lucindri on:
   - field-restricted single terms (`10.title`) — score + the implied `|d|`/`p(w|C)`;
   - repeated-field term counting (a term appearing in two `h1` values) — `tf` and score;
   - **cross-value proximity** (`#1(20 30)` where `20` ends `h1`#1 and `30` starts `h1`#2) — match set on
     both engines (the headline test);
   - field-restricted proximity within a single extent.
   Reuse the `score_probe`/`diff_fuzz` harness pattern; compare scores to ~1e-6 and match sets exactly.
5. **Catalog + decide fixes.** For each confirmed divergence: fix or catalog with rationale. Likely
   candidates: set a **position-increment gap** between repeated field values (so proximity respects
   value boundaries), and align field-restricted `|d|`/statistics if they differ. An extent-restricted
   proximity operator (`#combine[field]`) is out of scope unless the data demands it (tie to TASK-0016).

## Acceptance criteria

- Reproducible fielded integer collections + a driver under `scripts/`, indexed in both engines.
- A findings write-up (in `docs/lucindri-vs-indri-scores.md` or a new `docs/field-scoring.md`) stating,
  with numbers: Indri's field-restricted `|d|`/`p(w|C)` semantics, repeated-field `tf`/length behavior,
  and the cross-field proximity result on both engines.
- Each confirmed divergence is fixed (with a regression test) or cataloged with rationale. In
  particular, a **regression test for the cross-field phrase**: a doc with two disjoint `h1` values must
  match (or not) a boundary-spanning `#1`/`#uwN` the same way Indri does.
- Full searcher suite green; TASK-0011 legal fuzz + TASK-0015 invalid fuzz still clean.
