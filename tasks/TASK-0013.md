# TASK-0013 — Confirm Krovetz stemming is identical in Lucindri and C++ Indri

- **Status:** Done (2026-07-05)
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** Lucindri and Indri both apply "Krovetz" (KStem) stemming, but through **two
  independent implementations** — Lucindri uses Lucene's Java `org.apache.lucene.analysis.en.KStemFilter`
  (a re-port whose own javadoc warns it may differ from the original), while Indri uses its own C++
  `indri::parse::KrovetzStemmer`. All prior conformance work (TASK-0010/0011) deliberately routed
  *around* the stemmer (integer collections), so the two stemmers have **never been directly compared**.
  This task builds a differential harness that stems a large real vocabulary with both engines and
  reports every disagreement, weighted by collection frequency, then locks the outcome with a golden
  regression test (and, if disagreements are material, proposes a fix as a follow-up task).

## Why this matters

Stemming is upstream of everything: if `retrieving` stems to `retrieve` in one system and `retriev`
in the other, the two build different vocabularies and every score/ranking for that term family
diverges — a difference the integer-collection tests are blind to by construction. Krovetz is
dictionary-driven, and the two implementations carry **different dictionaries and exception tables**,
so real disagreements are plausible, not hypothetical. We need to quantify them, decide whether they
are negligible, and guard against silent future drift (e.g. a Lucene version bump changing KStem).

## Ground truth (verified in source, 2026-07-05)

- **Lucindri:** `LucindriAnalyzer/.../EnglishAnalyzerConfigurable.java:151` → `new KStemFilter(result)`;
  the stemmer is Lucene `org.apache.lucene.analysis.en.KStemmer` (+ `KStemData1..8` dictionary tables),
  bundled in the analyzer/searcher jars. `KStemFilter` is public; `KStemmer` is package-private (drive
  it through `KStemFilter`, not directly).
- **Indri:** `indri/include/indri/KrovetzStemmer.hpp`, class `indri::parse::KrovetzStemmer`, method
  `char * kstem_stemmer(char *term)` (thread-safe; performs its own case normalization; stem may be
  *longer* than the input). Source at `indri/src/KrovetzStemmer.cpp`. There is **no** standalone Indri
  stemmer CLI — we must write a tiny harness that links this class.
- Both cap word length (KStem has a max-word-len; words over the cap and non-alphabetic tokens pass
  through unchanged). Feed both the **same lowercased, alphabetic** tokens so tokenization differences
  (already handled elsewhere) are not conflated with stemming differences.

## Plan

Artifacts live under `scripts/stemmer-comparison/` (reproducible, committed); large inputs/outputs stay
under an external `$WORK` per repo convention.

1. **Input vocabulary (isolate tokenization).** Re-index LATimes in Indri with **stemming off**
   (`stemmer.name` empty) and dump the raw vocabulary with collection frequency
   (`dumpindex <index> v` — confirm the exact subcommand/columns at run time). This yields the exact
   set of already-tokenized, lowercased tokens Indri would feed to Krovetz, each with a cf for
   impact-weighting. Keep only alphabetic tokens (Krovetz only rewrites alphabetic words). Optionally
   union in a general English word list (e.g. SCOWL / `/usr/share/dict/words`) to probe beyond the
   corpus, but the corpus vocab is what determines retrieval impact.
2. **Indri oracle.** A ~30-line C++ program (`indri_kstem.cpp`) that includes `KrovetzStemmer.hpp`,
   reads one token per line from stdin, and prints `token\tstem`. Build against the installed Indri
   headers/libs (mirror `indri-config --cxxflags --libs`, or compile the single `KrovetzStemmer.cpp`
   directly — it has no other Indri dependencies). Commit the exact build command.
3. **Lucindri harness.** A small Java main/JUnit helper in `LucindriAnalyzer` (test scope) that runs
   each token through `new KStemFilter(new KeywordTokenizer(...))` (one token in → one token out) and
   prints `token\tstem`. Reuse `EnglishAnalyzerConfigurable` with **only** KStem enabled (lowercase
   already applied, stopwords off) so it exercises the exact production code path.
4. **Diff.** Join the two `token\tstem` maps on `token`; emit every disagreement as
   `token | indri_stem | lucindri_stem | cf`. Summaries: total tokens, #agree, #disagree,
   disagreement rate, and the **cf-weighted** disagreement rate (fraction of *token occurrences*, not
   *types*, that would stem differently — the retrieval-relevant number). Bucket disagreements
   (suffix-family differences, dictionary/exception-table differences, length-cap edge cases,
   one-side-no-op).
5. **Decide & record.** Write findings to `docs/lucindri-vs-indri-scores.md` (or a new
   `docs/krovetz-comparison.md`): the numbers, categorized examples, and a verdict:
   - **Negligible** (very low cf-weighted rate) → document it and lock a **golden regression test**.
   - **Material** → keep the same test as a characterization test, and file a follow-up task to fix
     (options: replace Lucene KStem with a faithful Krovetz port; port Indri's dictionary/exceptions
     into a custom `TokenFilter`; or align on the original Krovetz reference).
6. **Regression test (required, per repo policy).** Add a JUnit test in `LucindriAnalyzer`
   (e.g. `KStemVsIndriGoldenTest`) asserting `KStemFilter`'s output for a curated golden list —
   including a representative sample of agreements **and** every disagreement class discovered — matches
   a checked-in expected-stems file. This pins current KStem behavior and fails loudly if a dependency
   bump changes it. (If we later fix to match Indri, this test's golden file updates to the Indri stems
   and becomes the parity guard.)

## Acceptance criteria

- A committed, re-runnable harness (`scripts/stemmer-comparison/`) that produces the disagreement
  report from scratch (dump vocab → stem both ways → diff).
- A findings write-up with total/disagree counts and the **cf-weighted** disagreement rate, plus
  categorized examples.
- A recorded decision (negligible vs material) and, if material, a follow-up fix task.
- A green JUnit golden test in `LucindriAnalyzer` locking KStem output (fail-before/pass-after
  semantics for any future drift).

## Notes / gotchas

- `kstem_stemmer` case-normalizes internally and may return a stem **longer** than the input — compare
  raw returned strings, do not assume length ≤ input.
- Krovetz is a no-op on non-alphabetic / over-length tokens in both systems; exclude them so the report
  isn't padded with trivial agreements (but confirm the length caps actually match — a differing cap is
  itself a real disagreement worth noting).
- This is a stemmer-only study; tokenization parity is out of scope (already isolated via the integer
  collections in TASK-0010/0011).

  the Lucene harness (`LucindriKStem.java`, `KeywordTokenizer→LowerCaseFilter→KStemFilter`); driver
  `run_comparison.sh`; diff `diff_stems.py`. Compared on the **248,944 alphabetic types** of an
  unstemmed LATimes index (66.2M tokens). **Result: 99.97% of types / 99.9500% of occurrences agree.**
  Only 78 types diverge (31,897 occ), and `later` (Indri `later` vs Lucene `late`) alone is 76% of the
  divergent mass. All divergences are Indri dictionary/exception-table entries Lucene's port lacks
  (head-word no-op list, proper-noun list, irregular-plural conflations like `thieves→thief`,
  `wolves→wolf`, `crises→crisis`) plus a lower Indri word-length cap (25 vs ~50). Some Indri entries are
  worse than Lucene (`hal→hum`, `sal→sum`). **Decision: accept the quantified divergence; do NOT port
  Indri's tables** (impact negligible, and Indri's tables include worse entries). Guarded by golden test
  `KrovetzStemmerParityTest` (agreements + the 7 top-cf divergences; fails loudly on a Lucene bump).
  Findings in `docs/krovetz-comparison.md`. If exact parity is ever required, it's a separate task:
  add a `TokenFilter` carrying Indri's tables and flip the golden expectations to the Indri column.
