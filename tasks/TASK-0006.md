## 1. The two divergences (measured, μ=2000, 6-doc corpus)

Corpus: `1:apple cat dog  2:banana cat  3:apple sun moon  4:cat dog moon  5:dog apple  6:tree bird fish`.

### Bug 1 (Critical) — `#scoreif` / `#scoreifnot` do not filter
- `#scoreif(dog #combine(cat))` should require `dog`, rank by `cat` → Indri `#filreq` = **{1,4}**.
  Lucindri returns **{1,2,4,5}** (ignores the requirement).
- `#scoreifnot(dog #combine(cat))` should reject `dog` → Indri `#filrej` = **{2}**. Lucindri returns
  **{1,2,4,5}** (ignores the rejection).
- Root cause (from earlier code study): the parser sets the first clause's `Occur` to
  `MUST` / `MUST_NOT` (`IndriQueryParser`: `scoreif`→MUST, `scoreifnot`→MUST_NOT), but the operator
  is built as `IndriAndQuery`, whose weight/scorer **ignores clause `Occur`** — so no filtering
  happens. A zero-document filter term returns the full result set (verified).

### Bug 2 (Major) — `#not` adds negated-term docs to the candidate set
- `#combine(cat #not(dog))` → Indri **{1,2,4}** (only `cat` docs). Lucindri **{1,2,4,5}** — doc 5
  (`dog apple`) has `dog` and **no** `cat`, yet is retrieved. Lucindri's `#not(dog)`
  (`IndriNotQuery`/`IndriNotScorer`) contributes `dog`-docs to the `#combine` candidate iterator, so
  a document matching only the negated term is retrieved. In Indri, `#not` is a pure belief modifier
  on the existing candidate set; it does not generate candidates.

## 2. Fixes

### Fix 1 — implement `#scoreif` / `#scoreifnot` filtering
Make the required/prohibited clause actually gate the result set, matching Indri `#filreq`/`#filrej`
(keep only / drop docs matching the filter, score by the remaining subquery). Options to evaluate:
- Have the parser build a proper filtered query (e.g. a Lucene `BooleanQuery` with `FILTER` /
  `MUST_NOT` around the Indri belief subquery), instead of folding the filter clause into an
  `IndriAndQuery` that ignores `Occur`; or
- Teach `IndriAndWeight`/scorer to honor `MUST`/`MUST_NOT` clauses as hard filters.
Pick whichever composes cleanly with Indri scoring (the *scored* subquery must still produce Indri
beliefs over the surviving docs).

### Fix 2 — `#not` must not generate candidates
`#not(t)` should modify belief over docs already in the candidate set, not add `t`'s postings to it.
Ensure `#combine(a #not(b))`'s candidate set is driven by the positive operands only (docs matching
`a`), with `#not(b)` applying its `1 − p(b|·)` contribution to those docs. Confirm the intended
semantics against Indri before changing scoring — e.g. also check `#combine(#not(b))` alone and
`#not` nested in `#band`/`#uw` (where it should be rejected as a non-positional operand, per
TASK-0002 Fix A).

## Architecture (read first) — where the belief machinery lives

The conjunction/candidate-set/scoring classes — `IndriAndQuery`, `IndriAndWeight`, `IndriAndScorer`,
`IndriScorer`, `IndriQuery`, `IndriDisjunctionScorer` (all in package `org.apache.lucene.search`) —
are **part of stock Apache Lucene 8.10.0**, contributed upstream (repo commit `7af881c`: "Updated
Lucene to 8.10 to get latest Indri Code, removed classes that are now in Lucene"; confirmed via the
Lucene 8.10.0 Javadoc). **There is no Lucindri source for them** — do NOT fork/patch Lucene. Read
them for understanding at Apache: `apache/lucene-solr`, tag `releases/lucene-solr/8.10.0`, path
`lucene/core/src/java/org/apache/lucene/search/` (`IndriAndScorer.java`, `IndriAndWeight.java`,
`IndriDisjunctionScorer.java`, `IndriScorer.java`). Both fixes below live entirely in **Lucindri
source we have**: `IndriQueryParser` (parser) and `IndriNot*` (searcher package).

Key facts confirmed by reading the code:
- `IndriAndScorer` (Lucene) builds its candidate set as the **disjunction/union** of its children's
  iterators (via `DisjunctionDISIApproximation`) and, per candidate doc, uses each child's `score()`
  if positioned there else `smoothingScore(doc)`.
- `IndriNotScorer.iterator()` (ours) returns `subScorer.iterator()` — the **negated term's
  postings** — so `#not(b)` injects `b`'s docs into the parent union. That is the Bug-2 leak. Its
  `score()`/`smoothingScore()` (=`log(1 − exp(subscore))`) are correct.
- `#scoreif`/`#scoreifnot`: `IndriQueryParser` sets the first argument's clause `Occur` to
  `MUST`/`MUST_NOT`, but `createBooleanClause` builds `IndriAndQuery`, which **ignores clause
  occur** → no filtering (Bug 1).

### Bug 2 — `#not` (smaller; do first)
3. **Fail-first test:** `#combine(a #not(b))` returns only docs matching `a` (not docs matching only
   `b`) — currently returns the extra doc.
4. **Fix `IndriNotScorer.iterator()`** to a non-generating iterator (`DocIdSetIterator.empty()`) so
   `#not` no longer contributes candidates; it then participates only via `score()`/`smoothingScore()`
   on docs that are candidates through positive siblings. (Leave `score()`/`smoothingScore()` as-is.)
5. **Check the scoring tradeoff against Indri.** With an empty iterator, `IndriAndScorer` calls
   `#not`'s `smoothingScore(doc)` for every candidate — so a candidate that *does* contain `b` gets
   `1 − p_smoothed(b)` instead of `1 − p(b|doc)` with actual tf. Compare `#combine(a #not(b))`
   rankings/scores to Indri (`diff_indri.sh`). If materially divergent, design a deeper fix (make
   the And-scorer position `#not`'s underlying term iterator for *scoring* while excluding it from
   *candidate generation*) — likely needs the Lucene side, so log it as a separate follow-up rather
   than forking Lucene here.
6. **Regression:** `#combine(a b)`, `#or`, `#max`, `#wand`, `#wsum` doc sets unchanged.

### Bug 1 — `#scoreif` / `#scoreifnot` (larger)
7. **Fail-first tests:** `#scoreif(b sub)` returns exactly docs matching `b`, scored by `sub`
   (zero-doc filter term → empty result); `#scoreifnot(b sub)` excludes docs matching `b`. Both
   currently return everything.
8. **Choose the fix** (build in the parser / new searcher classes — never in Lucene):
   - **Approach A (fast first cut — fixes the retrieved SET):** in `IndriQueryParser`, emit for
     `#scoreif(A B…)` a stock Lucene `BooleanQuery`: scored subquery (`#combine(B…)`) as `SHOULD`,
     filter `A` as `FILTER`; for `#scoreifnot`, `A` as `MUST_NOT`. `FILTER`/`MUST_NOT` are honored by
     Lucene. **Caveat:** a filter-doc outside the scored subquery's candidate set won't receive the
     subquery's Indri smoothing score (a `SHOULD` clause contributes 0 when unmatched), so scores may
     differ from Indri. Use A to flip the set and confirm via `diff_indri.sh`.
   - **Approach B (recommended — matches Indri scoring):** add `IndriFilterRequireQuery` /
     `IndriFilterRejectQuery` (+ weights/scorers) in the searcher package that (a) restrict the
     candidate set to the filter query's matches (require) or exclude them (reject), and (b) score by
     the scored subquery's Indri belief **including smoothing** over the surviving docs — modeled on
     Apache `IndriAndScorer` and C++ Indri `FilterRequire/RejectNode`. Parser emits these for
     `scoreif`/`scoreifnot` (first operand = filter; rest = scored subquery).
   Recommend: land A, verify the set, then implement B for score parity.
9. **Parser wiring:** `#scoreif(A B…)` → filter=`A`, scored=`#combine(B…)` (single-B: just `B`).
   Consider accepting `#filreq`/`#filrej` as aliases (Indri's names) too.
10. **Verify against Indri:** `diff_indri.sh` shows `scoreif`, `scoreifnot`, and `not` rows flip to
    **SET-MATCH**; compare scores to `IndriRunQuery`.
