# TASK-0006 — Fix belief-operator retrieval-set divergences vs Indri (#scoreif/#scoreifnot filter; #not)

- **Summary:** The Lucindri-vs-Indri conformance comparison ([[TASK-0005]],

  negated term `t` but **none** of the positive query terms. Make both match Indri's semantics.

## 1. The two divergences (measured, μ=2000, 6-doc corpus)

Corpus: `1:apple cat dog  2:banana cat  3:apple sun moon  4:cat dog moon  5:dog apple  6:tree bird fish`.

### Bug 1 (Critical) — `#scoreif` / `#scoreifnot` do not filter
- `#scoreif(dog #combine(cat))` should require `dog`, rank by `cat` → Indri `#filreq` = **{1,4}**.
  Lucindri returns **{1,2,4,5}** (ignores the requirement).
- `#scoreifnot(dog #combine(cat))` should reject `dog` → Indri `#filrej` = **{2}**. Lucindri returns
  **{1,2,4,5}** (ignores the rejection).
- Root cause (from earlier code study): the parser sets the first clause's `Occur` to
  `MUST` / `MUST_NOT` (`IndriQueryParser`: `scoreif`→MUST, `scoreifnot`→MUST_NOT), but the operator
  is built as `IndriAndQuery`, whose weight/scorer **ignores clause `Occur`** — so no filtering
  happens. A zero-document filter term returns the full result set (verified).

### Bug 2 (Major) — `#not` adds negated-term docs to the candidate set
- `#combine(cat #not(dog))` → Indri **{1,2,4}** (only `cat` docs). Lucindri **{1,2,4,5}** — doc 5
  (`dog apple`) has `dog` and **no** `cat`, yet is retrieved. Lucindri's `#not(dog)`
  (`IndriNotQuery`/`IndriNotScorer`) contributes `dog`-docs to the `#combine` candidate iterator, so
  a document matching only the negated term is retrieved. In Indri, `#not` is a pure belief modifier
  on the existing candidate set; it does not generate candidates.

## 2. Fixes

### Fix 1 — implement `#scoreif` / `#scoreifnot` filtering
Make the required/prohibited clause actually gate the result set, matching Indri `#filreq`/`#filrej`
(keep only / drop docs matching the filter, score by the remaining subquery). Options to evaluate:
- Have the parser build a proper filtered query (e.g. a Lucene `BooleanQuery` with `FILTER` /
  `MUST_NOT` around the Indri belief subquery), instead of folding the filter clause into an
  `IndriAndQuery` that ignores `Occur`; or
- Teach `IndriAndWeight`/scorer to honor `MUST`/`MUST_NOT` clauses as hard filters.
Pick whichever composes cleanly with Indri scoring (the *scored* subquery must still produce Indri
beliefs over the surviving docs).

### Fix 2 — `#not` must not generate candidates
`#not(t)` should modify belief over docs already in the candidate set, not add `t`'s postings to it.
Ensure `#combine(a #not(b))`'s candidate set is driven by the positive operands only (docs matching
`a`), with `#not(b)` applying its `1 − p(b|·)` contribution to those docs. Confirm the intended
semantics against Indri before changing scoring — e.g. also check `#combine(#not(b))` alone and
`#not` nested in `#band`/`#uw` (where it should be rejected as a non-positional operand, per
TASK-0002 Fix A).

## 3. Tests (per the testing policy)

Use `searcher/testutil.TestIndex` (assert on doc **sets**, not scores):
1. `#scoreif(b sub)` returns exactly the docs matching `b`, scored by `sub`; a zero-doc filter term
   → empty result (currently returns everything — fail-first).
2. `#scoreifnot(b sub)` excludes docs matching `b`.
3. `#combine(a #not(b))` returns only docs matching `a` (not docs matching solely `b`).
4. Regression: `#combine`, `#or`, `#max`, `#wand`, `#wsum` unchanged.
5. End-to-end: `scripts/diff_indri.sh` shows `scoreif`, `scoreifnot`, and `not` rows flip to
   **SET-MATCH** vs Indri.

## 4. Acceptance criteria

1. `#scoreif`/`#scoreifnot` retrieve the same doc sets as Indri `#filreq`/`#filrej`; `#not` no
   longer retrieves negated-term-only docs.
2. Unit tests (§3) green; `mvn test` green in `LucindriSearcher`.
3. `docs/lucindri-vs-indri.md` regenerated: the three rows become conformant (update the catalog).
4. No regression in the other operators' doc sets (spot-check via the harness).
