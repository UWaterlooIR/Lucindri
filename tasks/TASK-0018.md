# TASK-0018 — Fix `#band` nested inside `#band` (scores diverge from C++ Indri)

- **Status:** Ready
- **Created:** 2026-07-05
- **Owner:** (unassigned)
- **Summary:** A `#band` used as an **operand of another `#band`** scores differently from C++ Indri
  (per-doc Δ up to ~1.1 log, ranking-changing). Flat `#band(a b c)`, a lone `#band`, `#band` inside a
  window (`#uwN`/`#combine`), and a window inside `#band` all **match** Indri — the divergence is narrow
  and specific to a `#band` operand of a `#band`. Found while verifying TASK-0012 (exact document length):
  once norm quantization was removed, this surfaced as a distinct, **length-independent** structural
  residual, separate from the cataloged filter-in-belief renormalization (TASK-0011 §5).

| **`#band( 492 #band( 732 577 ) )`** (band in band) | **DIVERGES** — Indri top docs d27293 (−7.197), d282 (−7.336) are **absent** from Lucindri's ranking; Lucindri tops d99770 (−7.066), whose Indri score is −7.757 |

From the TASK-0011 fuzz re-run (1,500 queries), the only non-filter queries with maxΔ > 0.05 after exact

Their Δ is ~identical with `exactDocumentLength` on or off (0.68 vs 0.68, 1.10 vs 1.08), confirming this is
**not** a document-length effect.

## Suspected cause

`#band` has its own implementation (`IndriBandQuery` / `IndriBandWeight` / `IndriBandScorer`), separate from
the window operators (`IndriTermOpWeight` + `IndriUnorderedWindowWeight` / `IndriOrderedWindowWeight`).
Because `#band`-in-`#uw` (outer = window) works but `#band`-in-`#band` (outer = band) fails, the suspect is
the **outer band's** occurrence-building:
`IndriBandWeight.createInvertedList` (`LucindriSearcher/.../IndriBandWeight.java`, the position sweep at
lines ~56–99). It reads each operand's `nextPosition()/endPosition()` and runs a `termsFound`/
`previousTermPosition` window sweep that assumes term-like single positions. When an operand is itself a
`#band` (an `IndriTermOpEnum` emitting wide `(start,end)` extents, possibly overlapping), that sweep appears
to count/emit extents differently from Indri's `BAndNode` extent-AND. The mismatch only manifests when both
the outer and inner op are `#band` (the inner `#band`'s extents fed into this exact sweep).

Note C++ Indri semantics to check against: `#band` (`indri/BAndNode`) is a pure boolean-AND over the
document — the belief is combined from the operands' beliefs when all are present — it does **not** impose a
proximity window. Confirm what extent list Indri's `BAndNode` exposes to a parent `BAndNode`, and match it.

## Root cause (CONFIRMED 2026-07-05 — supersedes the "Suspected cause" guess above)

Read the C++ Indri source. **`#band` is not an extent-AND and not a belief op — it is literally an unordered
window with unlimited width.** `InferenceNetworkBuilder::after(BAndNode)` builds:

```cpp
unorderedNode = new UnorderedWindowNode( bandNode->nodeName(), nullsRemoved, -1 );  // -1 = unlimited
```

`#uwN` builds the **same** `UnorderedWindowNode` with `windowSize = N`; `#band` is just `windowSize = -1`.
Both also (a) drop stopword operands, and (b) collapse a single surviving operand to the operand itself.
So the correct `#band` semantics = Lucindri's own `#uwN` algorithm with `distance = -1`.

**The bug is a stale copy of the window algorithm.** Lucindri has two near-identical `createInvertedList`
sweeps:
- `IndriWindowWeight.createInvertedList` (used by `#uwN`/`#odN`) — **correct**. On a completed window it does
  `i = current - 1` (skip past the matched window), **only inside the `if (termsFound == n)` block**; when no
  window starts at `i` it falls through to `i++`. (This conditional skip was the TASK-0010 Phase-3 fix.)
- `IndriBandWeight.createInvertedList` (used by `#band`) — **stale**. Its `i = current - 1` runs
  **unconditionally**, outside the match check. When position `i` fails to complete a band, the inner loop
  ran `current` to `size`, so `i = current - 1 = size - 1` and the **outer scan terminates early**, dropping
  every later band occurrence. TASK-0010 fixed this in the window code but never in the band copy.

Why only nested `#band` shows it: with flat single-position term operands, the tested docs happen to complete
a band at the first start position, so the premature termination doesn't change the count. A **nested `#band`
operand** emits multiple **wide, overlapping** `(start,end)` extents interleaved with the other operand's
positions, so some start positions don't complete a band → early termination → missed occurrences → wrong
`tf` → wrong score and ranking (Indri's top docs vanish from Lucindri's list). `#band`-in-`#uw` is unaffected
because the **outer** operator there is the correct `IndriWindowWeight`.

(Confirmed against `src/UnorderedWindowNode.cpp`: Indri emits a window at **every** start `i` — `i++`, no skip
— and dedups overlaps later at the belief node. Lucindri's conditional-skip non-overlapping counting yields
the same occurrence **count** and matches Indri for `#uwN` empirically; the band copy's unconditional skip
does not.)

## Implementation plan (2026-07-05)

**Fix (recommended — Option A): make `#band` reuse the window algorithm with `distance = -1`, and delete the
stale band code.** This is faithful to Indri (`#band` = `UnorderedWindowNode(-1)`), removes the duplicated
sweep that drifted, and reuses the already-correct, already-tested path.

Changes (searcher module):
1. **`IndriWindowWeight.java`** — relax the constructor parameter type from `IndriWindowQuery` to
   `IndriProximityQuery` (its only use is `super(query, …)`, and `IndriTermOpWeight` already takes
   `IndriProximityQuery`; both `IndriWindowQuery` and `IndriBandQuery` extend it). No behavioral change for
   `#uwN`/`#odN`. The existing distance check `if (((end - start) + 1) > distance && (distance >= 0)) break;`
   already treats any negative `distance` as unlimited, so `-1` needs no new code.
2. **`IndriBandQuery.java`** — change `createWeight` to return
   `new IndriWindowWeight(this, searcher, getField(), /*distance=*/-1, boost)` instead of `IndriBandWeight`.
3. **Delete the now-dead classes** (verified unreferenced anywhere else):
   `IndriBandWeight.java`, `IndriBandScorer.java`, and `IndriConjunctionScorer.java` (only `IndriBandScorer`
   used it; grep confirms no other references). Keep `IndriBandQuery` (still the parsed node for `#band`).

Fallback (Option B, if we want the smallest diff): keep `IndriBandWeight` but make its `i = current - 1`
**conditional** on `termsFound == iterators.size()` (mirroring `IndriWindowWeight`). One-line-ish, but it
keeps two copies of the algorithm — i.e. re-creates the drift that caused this bug. Prefer A.

**Edge cases to preserve (already handled by the shared `IndriTermOpWeight` path, verify with tests):**
- **OOV operand → empty.** `IndriWindowWeight` inherits `oovOperandForcesEmpty() == true` (a `#band`/window
  with an absent term never matches → floored cf=0 background). Same as today's `IndriBandWeight`. ✔
- **Collection-wide cf/df** for the derived band "term" (TASK-0011 fix in `IndriTermOpWeight`) is inherited
  unchanged. ✔
- **Single-operand `#band(x) ≡ x`** and **stopword operands** — Indri collapses these in the builder. Verify
  Lucindri's parser already collapses a single-operand proximity op (TASK-0014 did this for `#1(x) ≡ x`);
  confirm `#band(x)` behaves the same. If the parser doesn't special-case it, `IndriWindowWeight` with one
  operand still returns that operand's postings (buildLeafEnum → single iterator), which is acceptable.

### Tests (searcher module, JUnit + `TestIndex`)

New `BandOperatorTest` (or extend `ProximityOperatorTest`/`ProximityCountTest`):
- **Regression (the bug):** a small controlled index where `#band( a #band( b c ) )` returns the occurrences
  Indri returns. Assert against a **hand-computed** unlimited-unordered-window count (and Dirichlet score),
  and cross-check with the engine-internal invariant `#band(a #band(b c))` == `#uwK(a #uwK(b c))` for `K ≥`
  doc length (band = unlimited window). This test **fails before** (early-termination drops occurrences) and
  **passes after**.
- **Guards (no regression):** `#band(a b c)` still equals the flat-band expectation and equals `#uwK(a b c)`
  for large `K`; `#band` inside `#uw` and `#uw` inside `#band` still match; `#band` with an OOV operand still
  yields the floored background (not a spurious match).
- Use lowercase, non-stopword, stem-invariant tokens (`cat dog sun moon tree`) with known positions so counts
  are exact.
